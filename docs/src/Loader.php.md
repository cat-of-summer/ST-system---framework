# Loader.php
<!-- DOCGEN:END -->

`ST_system\Loader` — тонкая обёртка над нативными языковыми конструкциями PHP (`require`, `require_once`, `include`, `include_once`), объединённая с glob-резолвом путей через `Storage\File::find()`, плюс два хелпера для автозагрузки классов: `registerClass()` (точечная карта "один класс — один файл") и `registerDir()` (директорийный, PSR-4-подобный автозагрузчик с опциональным namespace-префиксом).

Класс `final`, поэтому не предназначен для наследования — только для прямого использования через статические вызовы (`Loader::require(...)`) либо через инстанс, привязанный к конкретному пути (`Loader::create($path)`).

## Зависимости

- `ST_system\Storage\File` — резолв путей и glob-паттернов в реальные файлы (`File::find()`, `$file->find()`), а также единая точка построения путей (`File::make()`).
- `ST_system\Debug` — `Debug::linter($realpath)` выполняет синтакс-проверку файла (`php -l`) перед его подключением через `include`/`include_once`.
- `ST_system\Rule` — валидация конфигурации, передаваемой в `registerDir()`.

## Главное: разница между require/include в этом классе

В обычном PHP `require` и `include` отличаются только тем, что происходит при **отсутствующем** файле (fatal error vs warning) — во всём остальном они ведут себя одинаково, включая обработку синтаксических ошибок и исключений внутри подключаемого файла.

`Loader` **намеренно инвертирует это ожидание** и делает `include`/`include_once` куда более "мягкими", чем `require`/`require_once`:

- **`require` / `require_once`** — ведут себя **точно как нативный PHP**: файл подключается безусловно, никакой предварительной проверки нет. Любая синтаксическая ошибка или исключение внутри файла приводят к обычному fatal error, как и должно быть.
- **`include` / `include_once`** — работают в режиме **best-effort / fault-tolerant**:
  1. Перед подключением файл прогоняется через `Debug::linter($realpath)`, который фактически вызывает `php -l` на этом файле.
  2. Если линтер вернул `code > 0` (найдена синтаксическая ошибка) — метод молча делает `return`, **полностью пропуская подключение файла**, без каких-либо предупреждений или исключений наружу.
  3. Если синтаксис в порядке, реальный `include`/`include_once` выполняется внутри `try { ... } catch (\Throwable $th) {}` — то есть **любое исключение или ошибка**, возникшие при подключении файла (в том числе `Error`/`TypeError` и подобные), **молча проглатываются**.

Таким образом, в `Loader`:
- `require()` / `requireOnce()` — используются, когда файл **обязателен** и его отсутствие/поломка должны прерывать выполнение (обычное, строгое поведение).
- `include()` / `includeOnce()` — используются, когда файл **опционален** и его поломка (синтаксическая ошибка или исключение при выполнении) не должна ронять всё приложение — например, при массовой автозагрузке "плагинов" или конфигурационных файлов из директории, где часть файлов может быть невалидна.

Любое значение `$action`, отличное от этих четырёх, приводит к `throw new \Exception("Method {$action} not found")`.

## Два уровня автозагрузки

`Loader` предоставляет два независимых, но взаимодополняющих механизма ленивой автозагрузки классов, оба построены поверх `spl_autoload_register()`:

### 1. `registerClass()` — точечная карта "класс → файл"

Регистрирует **один конкретный класс** в общей (статической, разделяемой между всеми инстансами `Loader` в рамках процесса) карте `$classes_map`. Сам автозагрузчик (`spl_autoload_register`) регистрируется в SPL-очереди **ровно один раз** за весь процесс — это гарантируется статическим флагом `$autoload_registered`. Каждый последующий вызов `registerClass()` лишь добавляет запись в общую карту, не регистрируя новый обработчик повторно.

Когда PHP впервые реально обращается к классу (создание объекта, статический вызов и т.д.) и не находит его — срабатывает зарегистрированный автозагрузчик, который проверяет карту `$classes_map` по полному имени класса и, если запись найдена, подключает соответствующий файл через `require`.

Особенности:
- Полное имя класса, на которое реагирует автозагрузчик, формируется как `"{$prefix}\\{$class_name}"`, если передан непустой `$prefix` (обрезанный от ведущих/замыкающих `\`), иначе — просто `$class_name`.
- Путь к файлу резолвится **относительно директории/файла, с которым был сконструирован `Loader`** (`$this->file->make($file_path . '.php')`).
- Если резолвленный файл **не существует** на момент вызова `registerClass()`, метод **молча ничего не делает** — это не ошибка. Класс просто не попадёт в карту и не будет автозагружаемым; при реальном обращении к нему PHP выдаст обычную ошибку "class not found", а не что-то более информативное из `Loader`.
- Загрузка **ленивая**: файл подключается не в момент регистрации, а только при первом реальном обращении к классу.

### 2. `registerDir()` — директорийный, PSR-4-подобный автозагрузчик

Регистрирует автозагрузчик, покрывающий **целую директорию** (namespace) сразу, без необходимости перечислять классы поштучно. Тоже построен на `spl_autoload_register()`, но резолвит путь к файлу динамически, исходя из имени запрашиваемого класса — по аналогии с PSR-4.

Особенности:
- Конфигурация (`$config`) валидируется через `Rule::object([...])->throwable()` со схемой:
  - `throw` (bool, по умолчанию `true`) — передаётся напрямую как соответствующий аргумент `spl_autoload_register()` (должен ли сбой автозагрузчика бросать исключение).
  - `prepend` (bool) — тоже передаётся напрямую в `spl_autoload_register()` (регистрировать ли обработчик в начале очереди SPL, а не в конце).
  - `prefix` (string, обрезается от `\`) — если задан, автозагрузчик реагирует **только** на классы, чьё полное имя начинается с `"{$prefix}\\"`; этот префикс отрезается перед дальнейшим резолвом пути. Если `prefix` не задан — автозагрузчик реагирует на **любое** имя класса.
- Идемпотентность по паре (директория, prefix): регистрация отслеживается в статической карте `$directories_map` по ключу `"{pathname}:{prefix}"`. Повторный вызов `registerDir()` с той же директорией и тем же `prefix` — молчаливый no-op: повторной регистрации обработчика в SPL не происходит. Это позволяет безопасно вызывать один и тот же bootstrap-код (например, из нескольких точек входа/инклюдов) сколько угодно раз без дублирования автозагрузчиков.
- Резолв пути: (возможно, обрезанное от префикса) имя класса преобразуется заменой `\` на `/`, к результату добавляется `.php`, и путь резолвится относительно директории, с которой был сконструирован `Loader` (`$directory->make(...)`).
- Подключение файла — простой **`require_once`**, без предварительного линтинга через `Debug::linter()` (в отличие от `Loader::include()`/`Loader::require()`). Если резолвленный файл не существует — ничего не происходит.

### Сравнение двух механизмов

| | `registerClass()` | `registerDir()` |
|---|---|---|
| Область действия | Один класс | Целая директория / namespace |
| Что нужно указать | Имя класса, путь к файлу | Директория, опционально `prefix` |
| Резолв пути | Фиксированный, задан явно при регистрации | Динамический, вычисляется из имени класса |
| Идемпотентность | По полному имени класса (карта общая для всех инстансов) | По паре (директория, prefix) |
| Отсутствующий файл | Молчаливый no-op при регистрации | Молчаливый no-op при фактическом резолве (в момент автозагрузки) |
| Подключение | `require` | `require_once` |

Оба механизма ленивые — фактическое подключение файла происходит только при первом реальном обращении к классу, а не в момент вызова `registerClass()`/`registerDir()`.

## Публичный API

### `Loader::create(...$args): static`
Фабричный метод, эквивалент `new static(...$args)`.

### `__construct(string $path)`
В отличие от большинства других классов фреймворка, конструктор `Loader` публичный — инстансы можно создавать напрямую. Внутри привязывает `$this->file = File::make($path)`.

Бросает `\Exception` с сообщением `"Path must be a local file, URI given: {$path}"`, если переданный путь оказался URI. `Loader` работает **только с локальной файловой системой** — в отличие от `File`, который умеет представлять как локальные пути, так и удалённые URI.

### Статические вызовы: `Loader::require()`, `Loader::include()`, `Loader::require_once()`, `Loader::include_once()`, `Loader::registerDir()`, `Loader::registerClass()`

Реализованы через `__callStatic()`:

- **`require` / `include` / `require_once` / `include_once`** — первый аргумент (`$input`) — путь, glob-паттерн или массив путей/паттернов. Резолвится через `File::find($input, [...доп. конфиг из второго аргумента, 'extension' => 'php'])` — расширение `php` форсируется всегда, независимо от того, что передано в конфиге вызывающим кодом. Для каждого найденного файла (в порядке, в котором их вернул `File::find()`) вызывается `Loader::connect($file->getPathname(), $name)`.
- **`registerDir`** — синтаксический сахар для `Loader::create($dir)->registerDir($config)`, где `$dir` — первый аргумент, `$config` — второй (по умолчанию `[]`).
- **`registerClass`** — синтаксический сахар для `Loader::create($path)->registerClass(...остальные аргументы)`, где первый аргумент становится базовой директорией для `File`.
- Любое другое имя метода — `throw new \Exception("Method {$name} not found")`.

### Вызовы на инстансе: `$loader->require()`, `$loader->include()`, `$loader->require_once()`, `$loader->include_once()`, `$loader->registerDir()`, `$loader->registerClass()`

Реализованы через `__call()`, привязаны к директории/файлу, с которым был сконструирован данный `Loader`:

- **`registerDir`/`registerClass`** — форвардятся напрямую в одноимённые приватные методы.
- **`require` / `include` / `require_once` / `include_once`** — если `$input` пуст **и** `$this->file->isFile()` (т.е. `Loader` был сконструирован с путём к конкретному файлу, а не директории), `$input` по умолчанию становится именем этого самого файла (`$this->file->getFilename()`). Это позволяет писать `Loader::create('path/to/file.php')->require()` без аргументов — и получать подключение именно этого файла. Иначе `$input` резолвится через `$this->file->find($input, [...])` — glob-поиск, выполняемый относительно директории, к которой привязан инстанс.

## Примеры использования

```php
// Строгое подключение обязательного файла — fatal error при отсутствии/ошибке
Loader::require(__DIR__ . '/bootstrap.php');

// Мягкое, отказоустойчивое подключение всех файлов из директории:
// синтаксически неверные или падающие с исключением файлы будут молча пропущены
Loader::include_once(__DIR__ . '/plugins/*.php');

// Директорийная автозагрузка с namespace-префиксом (PSR-4-подобно)
Loader::registerDir(__DIR__ . '/src', [
    'prefix'  => 'My\\Namespace',
    'prepend' => false,
    'throw'   => true,
]);

// Точечная регистрация одного класса
Loader::registerClass('MyClass', 'classes/MyClass', 'My\\Namespace');

// Инстанс, привязанный к конкретному файлу — require без аргументов
// подключает именно этот файл
Loader::create(__DIR__ . '/config.php')->require();
```

## Итог

`Loader` — намеренно узкоспециализированная, локально-файловая утилита (никогда не работает с URI, в отличие от `File`), которая:
1. Даёт единый интерфейс для `require`/`include` с glob-резолвом путей через `File::find()`.
2. Делает `include`/`include_once` безопасными для "опциональных" файлов за счёт предварительного линтинга и проглатывания исключений — намеренно инвертируя привычную для PHP семантику require-vs-include.
3. Предоставляет два взаимодополняющих, идемпотентных механизма ленивой автозагрузки классов: точечный (`registerClass`) и директорийный, PSR-4-подобный (`registerDir`).
