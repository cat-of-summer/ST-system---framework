# HasConfig.php
<!-- DOCGEN:END -->

## Назначение

`HasConfig` даёт любому классу, который его подмешивает, собственную ленивую, иммутабельную (в рамках запроса) конфигурацию, а также единый механизм валидации/нормализации входных `array $config` через `Rule`. Трейт используется практически всеми классами фреймворка (`Access`, `Debug`, `CacheDriver` и все его драйверы, `Storage\Resource`/`File`, все `API\Drivers\*`, `Schemas\*` и т.д.) — это один из двух "универсальных" примесей ST_system наряду с `HasEvents`.

Трейт полагается на класс `ST_system\Config` (глобальное хранилище конфигов, ключом выступает `static::class`) и на `ST_system\Rule` (движок валидации).

## Что добавляет подмешивающему классу

### `static::setConfig(array $config = []): void`

Записывает переданные пары `ключ (dot-notation) => значение` в неизменяемое хранилище `Config` для `static::class` через `Config::setImmutableConfig()`. Обычно используется для программной настройки класса до первого обращения к `config()` (см. `Debug::addDumpMethod()`, который дописывает `dump_methods.{$name}`).

### `static::config(string $key = '')`

Главный геттер конфигурации класса.

- При первом вызове для данного `static::class` (отслеживается статической `$initialized` внутри метода) выполняется однократная "ленивая" инициализация:
  1. если во внешнем конфиге (`Config::config(static::class)` — то, что задано пользователем/окружением через обычный, "мутабельный" слой `Config`) что-то есть, оно копируется в неизменяемый слой (`Config::fillImmutableConfig`), но **только те ключи, которых там ещё нет** (`fillImmutableConfig` не перезаписывает существующее);
  2. поверх докладываются значения `static::getDefaultConfig()` — опять же только туда, где ключа ещё не было.
- В итоге приоритет: то, что уже лежало в immutable-слое (например, записанное вручную через `setConfig()`) > внешний конфиг пользователя > `getDefaultConfig()` подмешивающего класса.
- Дальнейшие вызовы `config()` просто читают `Config::getImmutableConfig(static::class, $key)` — дорогая инициализация не повторяется.
- `$key` — путь в dot-notation (`'firewall.limits'`, `'format.timestamp.output'`); при `$key = ''` возвращается весь массив конфигурации класса.

Подмешивающий класс обычно переопределяет `protected static function getDefaultConfig(): array`, возвращая свои дефолты, например (`Access.php`):

```php
protected static function getDefaultConfig(): array {
    return [
        'credentials' => ['name' => 'pass', 'value' => date('dm')],
        'accessMethod' => 'GET',
        'firewall' => [
            'driver' => 'filesystem',
            'limits' => [[5, 1, 10], [10, 2, 30]],
            'ttl'    => CacheManager::config('default.ttl'),
        ],
    ];
}
```
После этого `Access::config('firewall.ttl')` вернёт значение по умолчанию (или переопределённое пользователем/через `setConfig()`).

Если базовый класс уже переопределял `getDefaultConfig()` (например абстрактный `File extends Resource`), принято мёржить через `array_merge(parent::getDefaultConfig(), [...])`, чтобы не терять дефолты родителя.

### `static::applyConfig(array &$config, array $schema = []): void`

Прогоняет переданный извне массив `$config` (обычно — аргумент публичного метода вида `array $config = []`) через валидатор `Rule`, **по ссылке**, дополняя/перезаписывая его нормализованными и провалидированными значениями. Бросает исключение (`Rule::object(...)->throwable()`), если данные не прошли схему.

Есть два режима:

1. **Без схемы** (`$schema` пуст) — применяется только правило `defaultConfig` целиком к массиву: для каждого ключа собственного конфига класса (`static::config()`), которого нет в `$config` либо значение `null`/`''`, подставляется значение из `static::config($k)`. То есть `$config` "дозаполняется" дефолтами класса без какой-либо иной валидации.

2. **Со схемой** (`$schema` — ассоциативный массив `поле => правило(-а)` в синтаксисе `Rule`) — вызывается `Rule::object(...)->throwable()->apply($config)`. Перед этим схема прогоняется через `resolveAtSpec()`, которая рекурсивно заменяет в строковых спецификациях `@` на `defaultConfig:` (например `'string|@credentials.name'` превращается в `'string|defaultConfig:credentials.name'`).

#### Механизм `@key`

`@key` внутри строки-правила — это сокращение для встроенного алиас-правила `defaultConfig`, зарегистрированного самим `applyConfig()` при первом использовании (через `Rule::create(...)->alias('\\defaultConfig')`, глобально, один раз на процесс). Семантика правила `defaultConfig`:

- Оно чувствительно к "сентинелу" (`seesSentinel()`) — то есть срабатывает даже если поле в `$config` вообще отсутствовало (не было передано вызывающим), а не только когда оно `null`/`''`.
- Параметр (после `:`) — путь конфигурации внутри **текущего класса** (`static::class`), например `credentials.name`. Если параметр не указан, а вызов идёт внутри `Rule::scope(static::class, ...)` (это гарантирует сам `applyConfig`, оборачивающий и режим со схемой, и режим без схемы в `Rule::scope`), то в качестве пути берётся текущий "префикс" поля схемы (`Rule::currentPrefix()`) — это реализовано в `before()`-хуке правила, который подставляет префикс первым аргументом, если параметр явно не задан.
- Если значение отсутствует, `null` или `''` — правило подставляет `$class::config($path)`, где `$class` — это `static::class`, захваченный через `Rule::scope`.
- Особый случай: если само значение поля — массив и параметр не указан (`!isset($p[1])`), правило проходит по **всем** ключам `$class::config()` и дозаполняет в массиве отсутствующие/пустые ключи значениями по умолчанию — это ветка, работающая при вызове без схемы.

Практически это означает: в схеме `applyConfig()` можно написать

```php
static::applyConfig($config, [
    'name'  => 'string|html_decode|@credentials.name',
    'value' => 'string|html_decode|@credentials.value',
    'accessMethod' => 'nullable|string|@accessMethod',
]);
```

и если вызывающий не передал `name`/`value`/`accessMethod` (или передал `null`/`''`), они будут автоматически подставлены из `static::config('credentials.name')`, `static::config('credentials.value')`, `static::config('accessMethod')` — то есть из дефолтного (или пользовательского) конфига именно этого класса. Это ключевой паттерн, который встречается в десятках методов фреймворка (`Access::requestAccess()`, `Access::httpAccess()`, `Access::call()`, `Debug::toFile()`, `Debug::handleError()` и т.д.).

### `protected static function getDefaultConfig(): array`

Точка расширения — по умолчанию возвращает `[]`. Подмешивающий класс переопределяет её, возвращая собственные дефолтные настройки (см. пример выше).

## Нюансы

- Инициализация `config()` выполняется один раз на класс за время жизни процесса (статическая переменная внутри метода, индексированная по `static::class` — поэтому у наследников свой независимый кэш инициализации).
- `applyConfig()` всегда оборачивает применение правил в `Rule::scope(static::class, ...)` — это то, что позволяет правилу `defaultConfig` (и подстановке текущего префикса) понимать, из конфигурации какого именно класса брать значения, даже если вызов идёт из унаследованного метода.
- Метод `resolveAtSpec()` рекурсивно проходит и строки, и массивы правил (в т.ч. вложенные `Rule::forEach([...])`), поэтому `@`-синтаксис работает и внутри составных спецификаций.
