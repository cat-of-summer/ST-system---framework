# Assets.php
<!-- DOCGEN:END -->

`final class Assets` — менеджер CSS/JS/шрифтов/SVG-иконок и веб-манифеста на странице: регистрация ресурсов, буферизация вывода по именованным "буферам" (например `head`/`footer`), объединение (**combine**) и минификация (**minify**) файлов одного типа, а также кеш-совместимый механизм "запись/воспроизведение", который позволяет ассетам корректно всплывать в разметке даже тогда, когда их фактическая регистрация была пропущена из-за попадания в кеш `View` (см. [View.php](View.php.md)). Дополнительно класс умеет генерировать полный набор favicon/PWA-манифеста по одному исходному изображению.

Зависимости: `Storage\File` (резолв путей, `combine()`/`minify()`/`convert()`/`extract()`/`bySprite()`, MIME-детект), `Rule` (валидация и дефолты полей `site.webmanifest`), `Main` (детерминированный хеш, склейка путей), `Storage\Mimes\ImageMime` (проверка допустимости расширения `.ico`), `Cache\CacheManager` (кеширование генерации манифеста).

## Конфигурация по умолчанию

```php
'default_buffer' => 'head',
'bufferization'  => true,
'css'   => ['combine' => true, 'minify' => true],
'js'    => ['combine' => true, 'minify' => true],
'fonts' => ['combine' => true, 'minify' => true],
```

`default_buffer` — имя буфера, используемое, когда вызывающий код не указал буфер явно. `bufferization` — глобальный переключатель между двумя принципиально разными режимами вывода (см. ниже). `css`/`js`/`fonts.combine` — объединять ли файлы одного типа с одинаковыми HTML-атрибутами в один; `.minify` — минифицировать ли получившийся (или единичный) файл.

## Два режима вывода: `bufferization` и оркестрация из `View`

Именно этот переключатель — самое важное для практического использования класса.

- **Immediate-режим** (`bufferization=false`, либо принудительно форсирован оркестрацией `View` — см. ниже): `mount($name)` просто выводит (`echo`) плейсхолдер-якорь инлайново, без output buffering. Соответствие "открыл — закрыл" не отслеживается через `ob_*`; фактическая подстановка контента и HTML ассетов на место якоря происходит позже, единоразово, явным вызовом `finalize($html)` над уже полностью собранной HTML-строкой.
- **Deferred/OB-режим** (`bufferization=true` и нет активной оркестрации): `mount($name)` толкает имя буфера в стек, открывает `ob_start()` и выводит плейсхолдер уже ВНУТРИ этого буфера. Раскрутка происходит через `register_shutdown_function` (регистрируется глобально один раз на процесс): в конце запроса функция один за другим (в порядке LIFO) закрывает все ещё открытые буферы стека, для каждого меняя плейсхолдер на накопленный контент + собранный HTML ассетов прямо внутри уже захваченного вывода. `render($buffer)` умеет закрыть буфер раньше конца запроса — с проверкой LIFO-порядка (если закрывается не самый верхний буфер стека — `RuntimeException`).

Это удобно для standalone-использования `Assets` вне `View`: можно открыть буфер в одном месте шаблона (например, в `<head>`), а добавлять в него ресурсы сколь угодно позже по ходу вывода страницы, и всё будет корректно собрано к моменту завершения скрипта — без необходимости вручную закрывать буфер, если это не нужно.

### Почему `View` форсирует immediate-режим

Пока `View` рендерит страницу, OB/shutdown-модель `bufferization=true` физически не работает: открытый `ob_start()` рвёт границы `View::build()` (там тоже используется output buffering для перехвата вывода шаблона), не переживает возврат из cache-hit (когда `build()` вообще не вызывается) и в любом случае раскручивается только после того, как `View` уже вернула готовую HTML-строку вызывающему коду — то есть менять в ней плейсхолдеры на этом этапе уже поздно.

Поэтому на время своего рендера `View::output()` принудительно переводит `Assets` в immediate-режим через `Assets::beginOrchestration()`/`endOrchestration()` (счётчик вложенности `$orchestration`, снимается в `finally`). Резолюция режима — `immediate(): bool = $orchestration > 0 || !config('bufferization')`. Благодаря счётчику (а не булеву флагу) вложенные `View`-рендеры (дети, оркестрируемые дырами кеша — см. [View.php](View.php.md)) корректно сохраняют immediate-режим на всё время, пока живо хотя бы одно активное дерево рендера. При этом глобальная настройка `bufferization=true` остаётся полностью честной для кода, который использует `Assets` напрямую, без `View` (например, простого legacy-скрипта).

Итог: внутри `View` конечный HTML всегда получает свои ассеты через один финальный вызов `Assets::finalize($html)` на верхнем уровне рендера (см. `View::output()`).

## Плейсхолдер-якорь

`placeholder(string $name)` (приватный) строит детерминированный HTML-комментарий вида `<!-- __ASSETS__:{hash}:{name}__ -->`, где `{hash}` — `Main::hash([__CLASS__, $name])`, мемоизированный в статическом массиве. Соль **обязана** быть детерминирована между процессами: она "запекается" в кешированный скелет `View` (как часть HTML-строки), и на cache-hit в совершенно другом процессе `finalize()`/`render()` должны искать и находить тот же самый якорь. Одновременно хеш от имени буфера делает якорь непохожим на случайный пользовательский контент страницы, что снижает риск случайного (или намеренного) совпадения с реальной разметкой.

## Запись/воспроизведение (`record`/`stopRecording`/`replay`)

Механизм существует **исключительно** ради интеграции с кешем `View`. Кеш `View` "запекает" вывод шаблона в HTML-скелет; на cache-hit шаблон не выполняется — `View::build()` не вызывается вовсе, а значит не выполняется и код, который регистрирует ассеты (`addCss`, `addJs`, `addResource`, `addString` и т. п., обычно вызываемый прямо из тела шаблона). Без дополнительного механизма закешированный узел на повторном рендере просто не принёс бы своих CSS/JS в буфер.

Решение — `View` оборачивает вызов `build()` при заполнении node-level кеша в пару `Assets::record()`/`Assets::stopRecording()`:

- `record()` — открывает новый фрейм записи (стек фреймов; поддерживаются вложенные записи, т. к. `View` может кешировать вложенные узлы).
- Пока фрейм записи активен, каждый низкоуровневый вызов регистрации ассета логирует **сериализуемый примитив** операции — путь, атрибуты, тип, буфер или сырую HTML-строку — но никогда не `File`-объект целиком (объекты `File` не гарантированно сериализуемы и не должны переживать процесс).
- `stopRecording()` — закрывает фрейм и возвращает накопленный список операций (или `[]`, если записи не было).
- `replay(array $ops)` — на cache-hit `View` передаёт сюда список операций, сохранённый вместе со скелетом. `content`-операции восстанавливают сырой HTML-контент (с дедупликацией по хешу контента — тот же контент дважды не добавится). `asset`-операции пере-резолвят путь через `File::make()` (не `File::find()`!) и регистрируют файл в целевой буфер с дедупликацией по хешу `[pathname, attrs]`. Принципиально важно, что `replay()` **не** выполняет повторный поиск на диске/glob — путь на момент записи уже был полностью конкретным файлом, а не паттерном, поэтому повторный `find()` был бы избыточен и мог бы дать другой результат при изменившемся состоянии файловой системы.

Публично из `View` этот механизм используется в `renderCachedSkeleton()`: при промахе кеша (miss) операции записываются вместе со скелетом; при попадании (hit) — воспроизводятся через `replay()` до того, как `Assets::finalize()` соберёт финальный HTML. Подробнее — в [View.php](View.php.md).

## Публичный API и диспатч

Все публичные операции идут через `__callStatic` (статический вызов `Assets::addCss(...)`) или через `__call` на инстансе, полученном через `Assets::create($path, $buffer)` (эквивалент `new Assets($path, $buffer)`). Реализация всех операций — приватные static-методы; `__callStatic`/`__call` — единственная публичная точка входа, что позволяет держать общий список поддерживаемых имён (`mount, render, finalize, addCss, addJs, addFont, addResource, addString, setManifest, svg, sprite`) в одном месте и бросать `BadMethodCallException` на всё остальное.

Инстанс-вызовы — не альтернативная реализация, а синтаксический сахар: они привязывают операцию к конкретному базовому `File` (переданному в конструктор/`create()`) и добавляют относительный резолв путей. Например, `Assets::create(__DIR__.'/assets')->addCss('foo.css')` резолвит `foo.css` относительно директории привязанного файла — вместо того чтобы вызывающему коду приходилось каждый раз склеивать абсолютный путь вручную. Точно так же `svg()`/`sprite()` в инстанс-форме резолвят путь через `$this->file->make(...)`, а `setManifest()` — резолвит `favicon` относительно того же базового пути.

### Методы

- **`static create(...$args): self`** / **`__construct(string $path, string $buffer = '')`** — привязывает объект к базовому `File::make($path)` и целевому буферу (по умолчанию — `config('default_buffer')`).
- **`static beginOrchestration()` / `endOrchestration()`** — публичный счётчик вложенности, которым `View` форсирует immediate-режим на время своего рендера (см. выше).
- **`static record()` / `stopRecording(): array`** — открывает/закрывает фрейм записи операций для последующего `replay()` (см. выше).
- **`static replay(array $ops): void`** — повторно применяет ранее записанный список операций.
- **`mount(string $name)`** (диспетчеризуется как `Assets::mount($name)`) — открывает буфер `$name`. В immediate-режиме — просто эхо плейсхолдера. В deferred-режиме — толкает имя в стек, `ob_start()`, эхо плейсхолдера внутри буфера. Бросает `RuntimeException`, если буфер с таким именем уже открыт и не был закрыт `render()`.
- **`render(string $buffer)`** — закрывает буфер. В immediate-режиме — просто сбрасывает флаг "открыт". В deferred-режиме — если буфер не в стеке активных (то есть управлялся не через OB), сразу выводит накопленный контент + HTML ассетов; если буфер на вершине стека — закрывает свой `ob_start()` и подменяет плейсхолдер в захваченном выводе; если буфер есть в стеке, но не на вершине — `RuntimeException` (нарушение LIFO-порядка: буферы нужно закрывать в обратном порядке открытия, как скобки).
- **`finalize(string $html): string`** — единственная операция, актуальная для интеграции с `View`: заменяет плейсхолдеры **всех** известных буферов в переданной строке на накопленный контент + собранный HTML ассетов, после чего сбрасывает состояние каждого буфера. Именно её `View::output()` вызывает один раз на верхнем уровне рендера, обернув в `Assets::finalize($html)` итоговую HTML-страницу.
- **`addCss($href, array $attrs = [], string $buffer = '')`**, **`addJs($src, ...)`**, **`addFont($src, ...)`** — тонкие обёртки над общим приватным `push($type, $src, $attrs, $buffer)`. `$src` резолвится через `File::find($src, ['fallback' => 'make'])` — поддерживаются glob-паттерны, массивы путей и одиночные пути. Дедупликация — по хешу `[pathname, attrs]` (тот же файл с теми же атрибутами не добавится дважды, а с другими атрибутами — станет отдельной записью). Каждая успешная регистрация одновременно пишет операцию `record()`-фрейма (для последующего `replay()`).
- **`addResource($path, array $attrs = [], string $buffer = '')`** — универсальный регистратор "определи тип по MIME файла": `font/*` → шрифты, `text/css` → css, `application/javascript` → js (идут в тот же комбинируемый пайплайн, что и `addCss`/`addJs`/`addFont`); `image/svg+xml` — инлайнится прямо как HTML-строка (`storeAsset($file->extract($attrs), $buffer)`), минуя механизм combine/minify; любой другой MIME — жёсткая ошибка, `InvalidArgumentException("Unsupported asset mime: ...")`. Строгость намеренная: молчаливый пропуск неподдерживаемого файла с большей вероятностью означал бы забытый тип ресурса, чем осознанное решение.
- **`addString($string, string $buffer = '')`** — добавляет одну или несколько (`array`) сырых HTML-строк в буфер напрямую, минуя `File` и MIME-детект. Дедупликация — по хешу самого контента.
- **`setManifest(array $params = [], string $buffer = '')`** — генерирует и подключает favicon/PWA-манифест. **Вызывается ровно один раз за запрос** — повторный вызов бросает `LogicException` (отслеживается через `static $done`). Достаёт `favicon` как `File` (с авто-`fetch()`, если это удалённый URL), строит `CacheManager` по ключу `[class, 'manifest', $params, путь источника, mtime источника]` с `ttl=-1` (кеш живёт вечно, пока не изменятся входные параметры или сам файл favicon), и через `remember()` либо переиспользует существующий кеш, либо вызывает `generateManifest()`.
- **`generateManifest(File $source, array $params, CacheManager $cache): string`** (приватный) — конвертирует исходное изображение в набор вариантов: `favicon.svg`, `favicon-96x96.png`, `apple-touch-icon.png` (180×180), два `web-app-manifest-*.png` (192×192 и 512×512, используются как `icons` в манифесте), и `favicon.ico` — с фоллбэком на PNG, если `.ico` не входит в `ImageMime::getAllowedExtension()` окружения. Поля JSON-манифеста (`name`, `short_name`, `theme_color`, `background_color`, `display`) валидируются и дефолтятся через `Rule::object(...)->after(...)`: `name` по умолчанию берётся из `$_SERVER['HTTP_HOST']`, `short_name` фоллбэчится на `name`, цвета — на `#fff`, `display` — на `standalone`. Итоговый JSON пишется как `site.webmanifest` через `putContents`, а функция возвращает готовый HTML — набор `<link rel="icon"/"apple-touch-icon"/"shortcut icon"/"manifest">` и `<meta name="theme-color">`.
- **`svg(string $path, array $attrs = [], bool $return_path = false): string`** — автоматически дописывает `.svg`, если у пути нет расширения. Возвращает либо относительный URL файла (`$return_path=true`), либо инлайновую SVG-разметку с инжектированными HTML-атрибутами (через `File::extract($attrs)`) — второй вариант удобен, когда нужно программно менять `fill`/`class`/`width` у иконки без дополнительного HTTP-запроса.
- **`sprite(string $path, string $icon_id, array $attrs = []): string`** — рендерит ссылку `<svg><use></svg>` на элемент SVG-спрайта через `File::bySprite()`. Бросает `InvalidArgumentException` при пустом `$path`.

### Сборка итогового HTML ассетов (`buildAssetsHtml`)

Приватный `buildAssetsHtml(string $buffer): string` вызывается и из `finalize()`, и из `render()`/shutdown-обработчика. Для каждого типа (`css`, `js`, `fonts`) зарегистрированные файлы группируются по хешу их HTML-атрибутов — то есть файлы с идентичным набором атрибутов (например, оба без `defer`) попадают в одну группу и могут быть объединены, а файлы с разными атрибутами (один с `defer`, другой без) остаются в разных группах и всегда выводятся раздельно. Внутри группы объединение (`$files[0]->combine($files, $attrs)`) происходит только если одновременно: файлов больше одного, `config("{type}.combine")` включён, и это не JS-модуль (`attrs['type'] === 'module'` отключает combine — ES-модули нельзя просто конкатенировать друг за другом, это меняет семантику их областей видимости). После объединения — минификация, если включена `config("{type}.minify")`. Если объединение не применяется, каждый файл выводит собственный `<link>`/`<script>` через `toHTML($attrs)`.

## Интеграция с View

`Assets` не полагается на то, что вызывающий код — обязательно `View`, но именно эта интеграция определяет большую часть нетривиального поведения класса:

- `View::output()` оборачивает **весь** рендер верхнего уровня в `Assets::beginOrchestration()`/`endOrchestration()`, форсируя immediate-режим независимо от глобального `bufferization` (см. выше).
- `View::renderCachedSkeleton()` оборачивает вызов `build()` при заполнении кеша в `Assets::record()`/`stopRecording()`, а на cache-hit вызывает `Assets::replay()` — иначе закешированный узел терял бы свои CSS/JS/иконки при каждом повторном использовании кеша.
- Финальная подстановка ассетов в готовую HTML-страницу происходит один раз — вызовом `Assets::finalize($html)` внутри `View::output()`, только на самом верхнем уровне дерева рендера.

Подробности про кеширование `View` (skeleton-кеш, composed-кеш, "дыры", poisoning) — в [View.php](View.php.md).
