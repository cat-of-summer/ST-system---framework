# View.php
<!-- DOCGEN:END -->

`final class View` — движок шаблонов ST_system: резолвит именованные "источники" (`page`, `component`, …) в PHP-файлы шаблонов, рендерит их с передачей props, поддерживает композицию parent/child через слоты ("русская матрёшка" вложенности) и реализует многоуровневое кеширование — per-node skeleton-кеш с placeholder-"дырами" для динамических поддеревьев, а поверх него — опциональный полностью-собранный root-кеш, способный вообще пропустить спуск в детей, если всё поддерево остаётся валидным. Тесно интегрирован с [Assets.php](Assets.php.md): на время рендера принудительно управляет режимом вывода ассетов и использует его механизм record/replay, чтобы кеш `View` не "терял" CSS/JS/иконки на cache-hit.

Зависимости: `Main` (dot-notation `get`/`set`, детерминированный хеш, склейка путей, `merge()`, `uuid()`), `Cache\CacheManager` (хранение skeleton- и composed-кешей).

## Конфигурация по умолчанию

```php
'source'    => [],
'extension' => 'php',
'exclude'   => [],
'cache' => [
    'use'    => false,
    'driver' => 'filesystem',
    'dir'    => …,
],
'cascade' => false,
```

`source` описывает набор именованных источников шаблонов (см. ниже), `extension` — расширение файлов шаблонов по умолчанию, `exclude` — список исключаемых из поиска путей/имён, `cache.use` — включает кеширование (`false` / `true` / `'full'` — режимы описаны ниже), `cascade` — каскадируются ли конфиг-оверрайды текущего узла в его детей по умолчанию.

## Источники шаблонов

`View::page('index', $props)` — синтаксический сахар через `__callStatic`: любое имя статического метода, не входящее в `RESERVED` и присутствующее среди зарегистрированных источников, трактуется как имя источника, а вызов превращается в `View::template('page/index', $props)`.

`sources()` (приватный, мемоизирован в `self::$sources`) строит карту `имя => {dir, ext, config}` одним из двух способов, в зависимости от формы `config('source')`:

- **Строка** — путь к каталогу, чьи непосредственные подпапки автоматически становятся источниками (имя источника = имя подпапки).
- **Ассоциативный массив** — `имя => путь` или `имя => spec`, где `spec` — массив вида `['source' => путь, ...per-source overrides]`. Overrides (например `cache`, `extension`, `cascade`) становятся **персональными дефолтами именно этого источника** и участвуют в цепочке приоритетов конфигурации (см. `recomputeConfig()` ниже).

### Зарезервированные имена

Константа `RESERVED` — `template, get, set, slot, capture, config, setConfig, applyConfig, sources, name, path, cache, cascade`. Источник не может называться ни одним из этих имён (конфликт со служебными методами класса); попытка зарегистрировать такой источник бросает `LogicException` уже на этапе построения карты источников, а не в момент использования — ошибка конфигурации всплывает как можно раньше.

### `template(string $path, ...$args): self`

Базовый резолвер, без алиас-магии: `$path` вида `"source/view"` разбивается на ключ источника и относительный путь вьюхи внутри него. Файл ищется в двух местах — `"{dir}/{view}.{ext}"`, а если не найден — `"{dir}/{view}/index.{ext}"` (fallback на "директория с index-файлом", удобно для группировки связанных партиалов). Если источник не зарегистрирован либо ни один из двух кандидатов не существует (или исключён через `exclude`) — `RuntimeException`.

Гибкие trailing-аргументы: если первый дополнительный аргумент — не массив, он трактуется как `$children` напрямую (без props); иначе это `$props` (пропускается через dot-flatten/rebuild — чтобы точечные и вложенные ключи нормализовались одинаково) с опциональным вторым аргументом `$children`.

Поле `auto_echo` выставляется в `true`, только если `self::$frames` не пуст в момент создания — то есть только когда дочерняя `View` создаётся **изнутри рендера** другой `View`. Это включает поведение автоматического эха при уничтожении объекта (см. `__destruct()` ниже) только там, где оно осмысленно (top-level `View`, на которую никто явно не сослался, обычно и не создаётся без намерения её вывести).

## Приоритет конфигурации (`recomputeConfig`)

Каждый инстанс при создании (и при каждом вызове `->cache()`/`->cascade()`) пересчитывает эффективный конфиг цепочкой приоритетов от слабого к сильному:

1. глобальный дефолт (`static::config()`);
2. персональный дефолт конкретного источника (`spec`-overrides из `sources()[$key]['config']`);
3. каскад от родителя (`$this->parentBundle` — снимок `bundle` родительского фрейма на момент создания);
4. явный override текущего инстанса (`->cache(...)`/`->cascade(...)`).

Благодаря этому у каждого источника (`page`, `component`, …) могут быть собственные дефолты кеширования, но точечный вызов `->cache(false)` на конкретном инстансе всегда их перебивает.

`$this->bundle` — то, что реально каскадируется **детям** этого узла — заполняется только если итоговый `cascade` резолвился в `true`, и собирается как `merge($this->parentBundle, $this->override)`. Важный нюанс: каскадирующийся конфиг **не включает персональные дефолты источника** — только цепочку "предок → предок → явные оверрайды", иначе каскад от компонента случайно тащил бы дефолты компонента в дочерний `page`.

`cache($mode = true): self` и `cascade(bool $on = true): self` — fluent-оверрайды. Оба бросают `LogicException`, если вызваны после того, как вьюха уже была отрендерена (`cache toggles must be called before render`) — конфигурация кеширования одного узла фиксируется до первого рендера и не может "передумать" постфактум.

## Три уровня кеширования

Это ключевая часть модели `View`, определяющая большую часть остального кода класса.

1. **Без кеша** (`cache.use = false`, дефолт) — каждый рендер узла всегда выполняет шаблон заново через `build()`.
2. **Per-node skeleton-кеш** (`cache.use = true`) — узел кеширует свой HTML-"скелет" с placeholder-"дырами" на месте динамических дочерних вьюх (`renderCachedSkeleton()`). На cache-hit шаблон **не выполняется**, но дочерние вьюхи всё равно реконструируются и рендерятся отдельно (дыры заполняются каждый раз).
3. **Composed root-кеш** (`cache.use = 'full'` на корневом узле верхнего уровня рендера) — поверх skeleton-кеша добавляется ещё один уровень: если **всё поддерево** валидно, целиком собранный HTML читается одним чтением, и рендер вообще не спускается в детей (`renderComposedRoot()`).

Режим `'full'`, применённый непосредственно к дочернему узлу (не к корню верхнего уровня), означает нечто иное — см. `__toString()` ниже: для такого узла никогда не создаётся "дыра", он всегда инлайнится напрямую в скелет родителя.

### `output(): string`

Точка входа рендера конкретного инстанса (вызывается из `__toString()`). На верхнем уровне дерева рендера (`empty(self::$frames)`) включает оркестрацию [`Assets`](Assets.php.md#два-режима-вывода-bufferization-и-оркестрация-из-view) через `Assets::beginOrchestration()`/`endOrchestration()` — на всё время рендера `Assets` принудительно работает в immediate-режиме, независимо от глобальной настройки `bufferization`. Дальнейшее поведение форкается по двум осям — "это верхний уровень или нет" и "включено ли кеширование":

- **Верхний уровень + кеш включён** → `renderComposedRoot()`.
- **Верхний уровень + кеш выключён** → обычный `build()`.
- **Не верхний уровень + кеш включён** → `renderCachedSkeleton()`.
- **Не верхний уровень + кеш выключён** → обычный `build()`, но дополнительно "отравляет" активный compose-коллектор (`self::$compose['composable'] = false`), если он сейчас запущен: один по-настоящему динамический узел где угодно в дереве полностью отключает composed-кеш для всего текущего прохода рендера.

Только вывод самого верхнего фрейма проходит через `Assets::finalize($html)` — ассеты собираются и подставляются в разметку ровно один раз за весь рендер страницы.

### `renderComposedRoot(): string` — composed-кеш ("русская матрёшка")

Самый сложный метод класса.

1. **Быстрый путь.** Читает мета-данные `composed`-записи кеша (`cache->getMeta('composed')`). Если там сохранён список зависимостей (`deps`) и текущий `depStamp($deps)` (хеш mtime всех файлов-зависимостей всего поддерева) совпадает с сохранённым `stamp`, а `composedReadsValid($cmeta)` подтверждает, что все ранее собранные `View::get()`-чтения всего поддерева по-прежнему резолвятся в те же значения — читается сам `composed`-HTML. При успехе воспроизводятся записанные операции ассетов (`Assets::replay()`) и переприменяются глобальные `View::set()`-значения оригинального рендера — и кешированный HTML возвращается напрямую. **Дочерние `View` не конструируются вообще**, шаблоны не выполняются.
2. **Путь пересборки.** Если быстрый путь не сработал, открывается фрейм-коллектор `self::$compose` (`composable = true` изначально, с аккумуляторами `descendants`/`deps`/`sets`/`assets`), и вызывается `renderCachedSkeleton()` — она рекурсивно наполняет тот же коллектор через `composeCollect()` для каждого встреченного узла (кешированного или нет), после чего предыдущий коллектор восстанавливается.
3. Новая `composed`-запись пишется в кеш **только если всё поддерево осталось `composable`** (ни один узел не оказался poisoned или явно динамическим) — сохраняются собранный HTML, `stamp` (по объединённым file-зависимостям всего поддерева), сам список `deps`, сводка по узлам (`descendants`), и агрегированные `sets`/`assets`.

### `composedReadsValid(array $cmeta): bool`

Валидирует reads всего поддерева применительно к composed-кешу, в "восстановленном" окружении корня: пушит синтетический фрейм с props корня, накатывает агрегированные `sets` как globals, и для каждого потомка (`descendants`) проверяет, что его сохранённые reads (`readsValid()`) по-прежнему совпадают в этом восстановленном контексте. Любое несовпадение — консервативный откат на полную пересборку (лучше лишний раз пересобрать, чем отдать устаревший HTML).

### `composeCollect(...)`

Приватный статический метод, действующий только когда `self::$compose` не `null` (то есть внутри `renderComposedRoot()`). Записывает reads-ключи и их stamp для очередного узла, объединяет пути файлов-зависимостей, sets и операции ассетов в общие аккумуляторы коллектора.

## `build(): string` — фактическое выполнение шаблона

Жёсткий лимит `MAX_DEPTH = 50` — превышение бросает `RuntimeException` (защита от случайной бесконечной рекурсии вьюх). Если сейчас активна кеш-"граница" (`self::$boundaries` не пуст), файл текущего узла записывается как file-зависимость этой границы (для последующей mtime-инвалидации).

Дальше — стандартная процедура: новый фрейм пушится в `self::$frames`, `ob_start()`, файл шаблона `require`-ится внутри изолированного статического closure — `(static function ($__file, $props) { require $__file; })($this->file, $this->props)`. Ключевой момент: closure статический, поэтому внутри шаблона **не виден `$this`** — единственный способ шаблона взаимодействовать с контекстом рендера — статические методы `View::get()`/`View::set()`/`View::slot()`/`View::name()`/`View::path()`. Это осознанное ограничение API, а не недосмотр. Захват буфера и снятие фрейма происходят в `finally`, что гарантирует целостность стека фреймов, даже если шаблон выбросил исключение по ходу выполнения.

## `renderCachedSkeleton(): string` — per-node skeleton-кеш

1. Читает мета-данные кеша узла. Cache-hit требует **одновременного** выполнения двух условий: `depStamp($deps)` (хеш mtime файлов-зависимостей) совпадает с сохранённым `stamp`, **и** `readsValid($reads_keys, $reads_stamp)` подтверждает, что все внешние `View::get()`-чтения, сделанные при генерации скелета, по-прежнему резолвятся в те же значения. Расхождение хотя бы одного из двух условий — промах кеша.
2. **Hit**: читаются сохранённый HTML-скелет и `payload` — сериализованные и base64-закодированные `{holes, sets, assets}`.
3. **Miss**: открывается новая кеш-"граница" (запись в `self::$boundaries`) с полями `deps`/`holes`/`salt` (случайный `Main::uuid()` для hole-токенов этого узла)/`n` (счётчик дыр)/`mode`/`reads`/`sets`/`base` (глубина стека фреймов на момент открытия границы, нужна для отличения "своих" props от "внешних" reads)/`poisoned`. Вызов `build()` оборачивается в `Assets::record()`/`stopRecording()` — так регистрация ассетов внутри шаблона фиксируется как список операций для будущего `replay()`. Если граница не оказалась poisoned — `{holes, sets, assets}` сериализуются в payload (ошибка сериализации сама по себе тоже считается poisoning — например, если props содержали закрытие или ресурс). Если poisoned — узел **не кешируется вовсе**, дыры заполняются напрямую тут же, а активный compose-коллектор (если есть) помечается некомпозируемым.
4. Payload декодируется (`unserialize` с `allowed_classes => true`); на реальном cache-hit (когда `build()` не выполнялся) дополнительно вызывается `Assets::replay($assets)` — иначе закешированный узел молча терял бы свои CSS/JS/иконки при повторном использовании кеша.
5. Сводка узла (deps/reads/sets/assets) всегда передаётся в активный compose-коллектор — независимо от того, был ли это hit или miss — потому что composed-кеш должен видеть полную картину поддерева в любом случае.
6. Возвращается `fillHoles($skeleton, $holes, $sets)`.

## `fillHoles(string $skeleton, array $holes, array $sets): string`

Сначала применяет собранные `sets` как globals — **до** рекурсии в дыры, чтобы дочерние узлы видели значения, установленные раньше в том же кешированном поддереве (порядок важен: `View::set()`, вызванный родителем перед дырой, должен быть виден внутри дыры). Если дыр нет — скелет возвращается как есть.

Если дыры есть — пушится фрейм (снова с проверкой `MAX_DEPTH`), и для каждого hole-токена реконструируется дочерняя `View` из сохранённых `{name, file, props, children, override}`, вызывается `recomputeConfig()` (конфигурация должна быть пересчитана заново — override сохранён, но объект новый), и получается её `output()` — напрямую, минуя механизм hole-подстановки повторно: вложенные "дыры в дырах" обрабатываются уже за счёт собственной осведомлённости `output()` о текущем стеке границ. В конце все накопленные токены заменяются одним вызовом `strtr($skeleton, $map)` — не по одному `str_replace` на дыру.

## `depStamp()` и `readsValid()` — сердцебиение инвалидации

`depStamp(array $files): string` (приватный static) — вызывает `clearstatcache()`, затем строит карту `file => filemtime(file) или 0`, хеширует через `Main::hash()`. Это **единственный** сигнал инвалидации по файлам — сравнивается mtime, а не хеш содержимого, то есть touch без реального изменения содержимого тоже инвалидирует кеш, а изменение содержимого без изменения mtime (редкий, но возможный случай при ручных манипуляциях с файловой системой) кеш не инвалидирует.

`readsValid(array $keys, string $stamp): bool` (приватный static) — пере-резолвит каждый ранее прочитанный dot-ключ, проходя `self::$frames` изнутри наружу, затем `self::$globals`, используя sentinel-объект, чтобы отличить "значение реально отсутствует" от "значение falsy" (`null`, `0`, `''` и т. п. не должны ошибочно трактоваться как "не найдено"). Сравнивает свежий хеш собранных значений с сохранённым `stamp`.

## `__toString()` — триггер рендера и логика poisoning/дыр

Фактический пуск рендера происходит по `echo $view` (implicit) или явному приведению к строке. Метод идемпотентен — при повторном вызове на уже отрендеренном объекте возвращает `''`.

Поведение резко форкается по наличию активной кеш-границы (`self::$boundaries`):

- **Нет активной границы** → обычный путь через `output()` (описан выше).
- **Есть граница, режим `'full'`** → узел **всегда** инлайнится напрямую через `build()`, дыра для него никогда не создаётся. Если при этом `$this->props` не является serialize()-совместимым — граница (весь скелет-предок) помечается `poisoned`: `'full'` для дочернего узла — это способ явно сказать "этот узел всегда часть моего скелета, не выделяй его отдельным кешируемым фрагментом", ценой того, что несериализуемые props в таком узле портят кешируемость всего родителя.
- **Есть граница, режим не `'full'`, props и children сериализуемы** → узел становится "дырой": в скелет родителя вставляется уникальный токен `<!--VH:{salt}:{n}-->`, а данные, достаточные для полной реконструкции вьюхи (`name`/`file`/`props`/`children`/`override`), откладываются для последующего `fillHoles()`.
- **Props сериализуемы, но children — нет** → узел всё равно безопасно инлайнится через `build()`, но **без** poisoning: несериализуемые children (например, замыкание слота, переданное явно, а не через массив) целиком "потребляются" при `build()` и не должны переживать между процессами — а сериализуемые внуки внутри всё равно корректно станут дырами текущей границы.
- **Ни props, ни children не сериализуемы** → "живой регион": узел инлайнится через `build()`, но граница помечается `poisoned` — её скелет нельзя безопасно переиспользовать между процессами, так как часть вывода зависит от данных, которые невозможно закешировать и валидировать по стампу.

### `__destruct()`

Если `auto_echo` истинен и вьюха ещё не была отрендерена — эхо-ит `$this->__toString()`. Это значит, что дочерняя `View`, присвоенная переменной, но никогда не приведённая к строке явно, всё равно будет отрендерена и выведена при выходе объекта из области видимости. Практический нюанс, о котором стоит помнить: момент вывода в этом случае зависит от таймингов сборщика мусора PHP, а не строго детерминирован порядком кода — если порядок вывода критичен, лучше явно `echo`/приводить к строке, не полагаясь на `__destruct()`.

## Статический API для шаблонов

Шаблоны (файлы, `require`-ящиеся внутри `build()`) не имеют доступа к `$this` и взаимодействуют с контекстом рендера исключительно через следующие статические методы:

- **`get(string $key, $default = null)`** — читает props/globals по dot-ключу. Ищет по фреймам изнутри наружу, затем в globals. Побочный эффект: если сейчас активна кеш-граница и найденное значение резолвилось либо нигде, либо во фрейме **вне** поддерева этой границы (`$foundAt === null || $foundAt < $base`), и этот ключ не был записан через `set()` внутри той же границы — ключ и значение фиксируются в `reads` границы. Это и есть механизм, которым `readsValid()` потом проверяет, не "протухла" ли внешняя зависимость закешированного узла. Reads, резолвленные из props самого узла (внутри поддерева границы), намеренно не трекаются — они уже покрыты обычной инвалидацией по props/файлам.
- **`name(int $i = 0)` / `path(int $i = -1)`** — интроспекция текущего (или произвольного по индексу, включая отрицательные — с конца стека) фрейма: имя вьюхи без префикса источника (`name()`) и путь файла шаблона без расширения (`path()`).
- **`set($key, $value = null)`** — пишет значение в `self::$globals` (переживает текущий узел — видно и последующим siblings, и более поздним `get()`), и, если сейчас активна кеш-граница, дополнительно фиксирует в `sets` этой границы, чтобы на cache-hit тот же side-effect воспроизводился без повторного выполнения шаблона.
- **`slot(...$args)`** — рендерит children/слоты изнутри шаблона. Формы вызова: `slot()` (дефолтный слот), `slot('name', [$defaults])`, `slot([$defaults])`. Дефолты применяются в props текущего фрейма только для тех ключей, что ещё не резолвятся через `get()`. Бросает `LogicException` при вызове вне активного рендера. `children` — либо обычный callable (дефолтный слот), либо именованная карта callables (мульти-слот).
- **`capture(callable $fn): string`** — generic output-buffering хелпер, не связанный с кешированием вьюх: просто `ob_start()`/`ob_get_clean()` вокруг `$fn()`, с гарантированным закрытием буфера в `finally`.

## Интеграция с Assets

`View` — основной практический потребитель [`Assets`](Assets.php.md):

- на верхнем уровне рендера `View::output()` включает оркестрацию `Assets` (`beginOrchestration()`/`endOrchestration()`), форсируя immediate-режим ассетов на всё время рендера страницы, независимо от глобальной настройки `Assets::config('bufferization')`;
- `renderCachedSkeleton()` оборачивает выполнение шаблона (`build()`) в `Assets::record()`/`stopRecording()` при заполнении кеша, а на cache-hit вызывает `Assets::replay()`, чтобы закешированный узел не терял регистрацию своих CSS/JS/иконок;
- итоговая HTML-строка проходит через `Assets::finalize($html)` ровно один раз — на самом верхнем уровне дерева рендера.

Подробное описание режимов вывода `Assets` (immediate vs deferred/OB) и механизма record/replay — в [Assets.php](Assets.php.md).
