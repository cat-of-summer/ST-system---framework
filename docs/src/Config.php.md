# Config.php
<!-- DOCGEN:END -->

## Что это

`ST_system\Config` — финальный класс, реализующий глобальный загрузчик конфигурации фреймворка. У него нет экземпляров: всё состояние хранится в статических свойствах (`$cache`, `$configPath`), а вся публичная поверхность — статические методы. Класс отвечает сразу за три независимых, но связанных источника данных:

1. **Переменные окружения** — чтение `.env`-файла при старте приложения (`init()`/`loadEnv()`) и последующее кешируемое чтение через `env()`.
2. **PHP ini-директивы** — кешируемая обёртка над `ini_get()` (`ini()`).
3. **Файловая конфигурация** — конфиги, лежащие на диске (папка с файлами конфигурации или один файл), доступные через `config()`, плюс параллельное хранилище «immutable»-конфигов в памяти (`getImmutableConfig()`/`setImmutableConfig()`/`fillImmutableConfig()`), которым пользуется `Traits\HasConfig`.

### Как `Config` резолвит конфиг для классов (связка с `Traits\HasConfig`)

Когда какой-либо класс фреймворка использует `Traits\HasConfig` и вызывает `static::config($key)`, происходит следующее (см. `src\Traits\HasConfig.php`):

1. При первом обращении к `static::config()` для конкретного класса трейт проверяет файловый конфиг: `Config::config(static::class)` — то есть под ключом равным **полному имени класса** (например, `ST_system\View`).
2. Если для этого класса в файловой конфигурации что-то нашлось — оно «заливается» (без перезаписи) в отдельное, независимое от файлового кеша хранилище через `Config::fillImmutableConfig(static::class, '', $данные)`.
3. Поверх этого «заливаются» значения по умолчанию из `static::getDefaultConfig()` — тоже без перезаписи уже установленных ключей.
4. Все последующие чтения (`static::config($key)`) идут уже не в файловую систему, а в это in-memory хранилище через `Config::getImmutableConfig(static::class, $key)`.

Таким образом в `Config` фактически два независимых «отсека» кеша:

- **Файловый кеш** (`config()`/`setConfig()`/`fillConfig()`) — данные, прочитанные с диска (из `$configPath`), хранятся под скрытым namespace-ключом (см. ниже про `configKey()`), доступ по пути вида `"<конфиг-неймспейс>.<ключ>"`.
- **Immutable-кеш** (`getImmutableConfig()`/`setImmutableConfig()`/`fillImmutableConfig()`) — данные хранятся прямо под указанным `$key` (без дополнительного namespace), что и используют трейты вроде `HasConfig`, кладя туда объединённый результат «файловый конфиг + defaults» под ключом `static::class`.

### Как физически ищутся файлы конфигурации

`config($key, $default)` смотрит на `static::$configPath` (устанавливается один раз в `init()`):

- Если `$configPath` — **директория**: берётся первый сегмент до точки в `$key` (`explode('.', $key)[0]`) как имя файла, например для `$key = "ST_system\View"` это `"ST_system\View"` целиком (в неймспейсах фреймворка нет точек, есть только `\`). Далее последовательно проверяются файлы `<configPath>/<fileKey>.php`, `<fileKey>.json`, `<fileKey>` (без расширения) — берётся первый существующий, парсится через `parseFile()` и кладётся в кеш целиком под этим `fileKey`. Поскольку `DIRECTORY_SEPARATOR` подставляется перед именем, а имя может содержать `\`, на Windows это фактически создаёт вложенную структуру директорий, зеркалящую пространство имён класса (например, `config/ST_system/View.php`).
- Если `$configPath` — **файл**: он парсится целиком один раз, и все его верхнеуровневые ключи сливаются (`array_merge`) в кеш конфигурации.
- После подгрузки нужного файла путь `$key` резолвится через `Main::dotGet()` по dot-notation внутри уже загрученных данных.

`parseFile()` понимает три формата по расширению:
- `.php` — файл должен вернуть (`return`) массив;
- `.json` — декодируется через `json_decode(..., true)`;
- остальное — построчный `KEY=VALUE` формат (упрощённый, без интерполяции и многострочности — в отличие от `.env`), с обрезкой кавычек и пропуском строк-комментариев, начинающихся с `#`.

### Скрытые namespace-ключи кеша

Методы `envKey()`, `configKey()`, `iniKey()`, `fileKey()` каждый раз возвращают один и тот же (сгенерированный один раз за процесс через `random_bytes(6)` и с префиксом `"\0"`) непредсказуемый ключ. Он используется как «отсек» внутри общего массива `$cache`, чтобы данные env/config/ini/флаг-файла не могли случайно столкнуться по имени с реальными пользовательскими ключами конфигурации (обычный ключ конфигурации никогда не начинается с NUL-байта).

`sentinel()` возвращает единственный на процесс объект `stdClass`, используемый как «метка отсутствия значения» — это позволяет отличить «ключ не найден» от «ключ найден и его значение — `null`» при поиске через `Main::dotGet()`.

## Публичные методы

### `Config::init(array $params = []): void`

Инициализирует конфигурацию приложения. **Может быть вызван только один раз за процесс** — повторный вызов бросает `\LogicException`.

Параметры (`$params`):
- `config_path` (string, опционально) — путь к файлу или директории с конфигурацией. Сохраняется в `Config::$configPath` для последующих вызовов `config()`.
- `dotenv_path` (string, опционально) — явный путь к директории, где лежит `.env`. Если не передан, `.env` ищется автоматически:
  1. рядом с `config_path` (в той же директории, если `config_path` — файл, или в самой директории, если `config_path` — папка);
  2. если там `.env` не найден — в корне проекта (директория, где лежит `composer.json`, определяется через `getDefault()['COMPOSER_ROOT']`);
  3. если и там нет — `.env` не загружается вовсе.

Если `.env`-директория определена, вызывается `loadEnv()`, которая парсит файл и заполняет `$_ENV`/`$_SERVER`/`putenv()` (не перезаписывая уже существующие переменные окружения).

```php
use ST_system\Config;

// Явно указать и конфиг, и .env
Config::init([
    'config_path' => __DIR__ . '/config',
    'dotenv_path' => __DIR__,
]);

// Или положиться на автопоиск .env рядом с config_path / в корне composer-проекта
Config::init(['config_path' => __DIR__ . '/config']);

// Повторный вызов — ошибка
Config::init(); // выбросит \LogicException
```

Формат `.env`, который понимает `loadEnv()` (полноценный, в отличие от простого построчного парсера `parseFile()`):
- пустые строки и строки, начинающиеся с `#` (после `ltrim`), игнорируются;
- опциональный префикс `export ` перед именем переменной;
- имя переменной должно соответствовать `^[A-Za-z_][A-Za-z0-9_.]*$`;
- значение в двойных кавычках: поддерживает экранирование (`\n`, `\r`, `\t`, `\f`, `\v`, `\"`, `\\`, `\$`), интерполяцию `$VAR`/`${VAR}` (из уже распознанных переменных этого файла, затем из `$_ENV`/`$_SERVER`/`getenv()`), а также многострочные значения (кавычка не закрыта в этой строке — парсинг продолжается на следующей);
- значение в одинарных кавычках: берётся буквально, без экранирования и интерполяции, тоже может быть многострочным;
- значение без кавычек: обрезается по первому `#`, которому предшествует пробел/таб или начало строки (инлайн-комментарий), затем `trim()` и интерполяция `$VAR`.

Переменные, уже присутствующие в `$_ENV`, `$_SERVER` или доступные через `getenv()`, из `.env` **не перезаписываются**.

### `Config::reload(): void`

Полностью сбрасывает кеши `env()`, `config()` и `ini()` (обнуляет соответствующие отсеки `$cache`). После вызова следующие обращения к этим трём методам заново прочитают значения (файлы конфигурации будут перепарсены, `env`/`ini` — заново вычислены). **Не** влияет на immutable-хранилище (`getImmutableConfig()`/`setImmutableConfig()`) и не сбрасывает `$configPath`.

```php
Config::reload();
```

---

### `Config::env(string $name, $default = ''): string`

Возвращает значение переменной окружения `$name` в виде строки, с кешированием на процесс.

Порядок разрешения значения (первое непустое совпадение побеждает): `$_ENV[$name]` → `$_SERVER[$name]` → `getenv($name)` → `Config::getDefault()[$name]` (единственное встроенное значение по умолчанию — `COMPOSER_ROOT`) → `$default`.

```php
$debug = Config::env('DEBUG_MODE', 'false');
$root  = Config::env('COMPOSER_ROOT'); // корень composer-проекта, если .env его не переопределяет
```

---

### `Config::ini(string $name, $default = ''): string`

Кешируемая обёртка над `ini_get($name)`: если директива не установлена (`ini_get()` вернул `false`), возвращается `$default`.

```php
$memLimit = Config::ini('memory_limit', '128M');
```

---

### `Config::config(string $key = '', $default = null)`

Читает значение файловой конфигурации по dot-notation пути `$key` (см. раздел «Как физически ищутся файлы конфигурации» выше).

- `$key = ''` — вернуть весь загруженный на данный момент файловый конфиг целиком (массив).
- `$key = 'ST_system\View'` — при необходимости подгрузить и вернуть весь конфиг для этого «файлового ключа».
- `$key = 'ST_system\View.cache.driver'` — то же самое, но вернуть вложенное значение по dot-пути `cache.driver` внутри загруженного файла.

Если `Config::$configPath` не задан (`init()` не вызывался с `config_path`, либо путь пуст) или искомый ключ нигде не найден — возвращается `$default`.

```php
// config/ST_system/View.php должен вернуть, например: return ['cache' => ['driver' => 'filesystem']];
$driver = Config::config('ST_system\View.cache.driver', 'memory');

$wholeFileConfig = Config::config('ST_system\View'); // весь массив из файла
$everything      = Config::config(); // весь загруженный на данный момент файловый конфиг
```

---

### `Config::setConfig(string $key, $value): void`

Программно устанавливает (перезаписывает) значение в файловом кеше конфигурации по dot-пути `$key`, минуя чтение с диска.

```php
Config::setConfig('ST_system\View.cache.driver', 'redis');
```

---

### `Config::getImmutableConfig(string $key, string $subKey = '')`

Читает значение из **immutable**-хранилища (отдельного от файлового кеша, без namespace-префикса) — того самого, которым пользуется `Traits\HasConfig::config()`.

- `$subKey = ''` — вернуть всё, что лежит под `$key` (обычно — имя класса), в виде массива (или `[]`, если ничего нет).
- иначе — вернуть значение по dot-пути `$key.$subKey`, либо `null`, если не найдено.

```php
// Обычно вызывается не напрямую, а через HasConfig::config() у конкретного класса
$value = Config::getImmutableConfig('ST_system\View', 'cache.driver');
```

---

### `Config::setImmutableConfig(string $key, string $subKey, $value): void`

Записывает значение в immutable-хранилище по пути `$key.$subKey` (перезаписывая существующее значение, если оно там было).

```php
Config::setImmutableConfig('ST_system\View', 'cache.driver', 'redis');
```

Именно этот метод дергает `Traits\HasConfig::setConfig()` для программной настройки конфигурации конкретного класса:

```php
// внутри любого класса, использующего HasConfig
View::setConfig(['cache' => ['driver' => 'redis']]);
```

---

### `Config::fillConfig(string $key, $value): void`

Как `setConfig()`, но **не перезаписывает** уже существующие значения — заполняет только отсутствующие ключи (сравнение через сентинел, отличает «нет значения» от «значение `null`»). Если `$value` — массив, он предварительно «расплющивается» через `Main::dotFlatten($value, $key)` и каждый плоский путь заполняется по отдельности (то есть отсутствующие вложенные ключи будут добавлены, даже если по соседству уже есть частично заполненная структура).

```php
Config::fillConfig('ST_system\View', ['cache' => ['driver' => 'filesystem']]);
// Если 'ST_system\View.cache.driver' уже был установлен раньше — значение не изменится.
```

---

### `Config::fillImmutableConfig(string $key, string $subKey, $value): void`

Аналог `fillConfig()`, но для immutable-хранилища и с явным разделением `$key`/`$subKey`. Именно этот метод использует `Traits\HasConfig::config()` для того, чтобы один раз на класс залить в immutable-кеш сначала данные из файлового конфига, а затем (тоже без перезаписи) значения по умолчанию:

```php
if (!empty(Config::config(static::class)))
    Config::fillImmutableConfig(static::class, '', Config::config(static::class));

Config::fillImmutableConfig(static::class, '', static::getDefaultConfig());
```

Если `$value` — массив, он так же расплющивается (`Main::dotFlatten($value, $subKey)`) и заполняется поэлементно, что позволяет комбинировать частично заданный пользователем конфиг с дефолтными значениями на уровне отдельных вложенных ключей, а не только на уровне всего массива целиком.
