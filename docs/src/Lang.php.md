# Lang.php
<!-- DOCGEN:END -->

`final class Lang` — локализация, гибрид подходов Bitrix и Laravel: фразы хранятся не в одном большом файле на локаль, а рассыпаны по дереву каталогов источника (файлы `<локаль>.php`, возвращающие массив, и/или `<локаль>.json`), а ключ обращения "переливается" между путём файловой системы и путём внутри PHP/JSON-массива в любой точке разбиения. Поверх поиска по дереву есть компилируемый кеш (одна плоская карта на пару источник+локаль) с mtime-инвалидацией без пересканирования дерева, runtime-оверрайды в памяти процесса и Laravel-стиль плейсхолдеров/склонений (`choice()`).

Зависимости: `Traits\HasConfig` (конфиг класса), `Cache\CacheManager` (хранение скомпилированной карты), `Main` (dot-notation `dotGet`/`dotSet`/`dotFlatten`, детерминированный хеш `hash()`, `merge()`, `preparePath()`, `glue()`, `basename()`, `pluralIndex()`). Используется [View.php](View.php.md) напрямую: тот открывает рекордер `Lang::record()`/`stopRecording()` вокруг рендера каждой кешируемой границы, чтобы задействованные языковые файлы попали в её зависимости инвалидации — иначе правка перевода не меняла бы закешированный HTML. Механизм тот же, что у `Assets`, контракт — другой (см. «Рекордер зависимостей»).

## Конфигурация по умолчанию

```php
'dir'        => '~',
'source'     => [],
'locale'     => 'en',
'fallback'   => 'en',
'extensions' => ['php', 'json'],
'dir_name'   => '',
'cache' => [
    'use'    => true,
    'driver' => 'filesystem',
    'dir'    => …,
],
'plural' => [],
```

`dir` — общий корень всех источников (см. ниже). `source` — карта именованных источников фраз; пустая по умолчанию, потому что корневой источник `'~'` добавляется автоматически и указывает на `dir`. `locale`/`fallback` могут быть строкой, либо `callable`/`\Closure` — тогда значение резолвится вызовом при каждом обращении, что удобно для локали, зависящей от текущего запроса. `extensions` — расширения файлов фраз в порядке приоритета (при конфликте в одном каталоге побеждает то, что стоит раньше в списке). `dir_name` — имя подкаталога, где лежат языковые файлы (пусто — файлы лежат прямо в дереве источника вперемешку с остальным содержимым, например компонентами). `plural` — карта `локаль => callable(int $n): int` для кастомных правил склонения, переопределяющих встроенные.

Смена языка во время работы — обычный `setConfig(['locale' => ...])`; отдельной runtime-оси нет (см. «Локаль» ниже). `lockLocale()` необратимо запрещает дальнейшую смену — см. «`lockLocale()`» там же.

## Источники (`source`) и общий корень (`dir`)

`dir` задаёт корень, относительно которого резолвятся пути источников, а `config('source')` описывает набор именованных источников поверх него:

```php
Lang::setConfig([
    'dir'    => '~/app',
    'source' => [
        'admin'   => 'admin/lang',                                    // => ~/app/admin/lang
        'dynamic' => ['path' => 'runtime/lang', 'cache' => ['use' => false]],
        'shop'    => ['path' => 'modules/shop/lang', 'extensions' => ['json']],
    ],
]);
```

Следствие жёсткой границы: источник вне `dir` невыразим. Если языковые файлы нужны, например, и из `~/app`, и из пакета в `~/vendor/pkg`, корнем должен быть общий предок (`'dir' => '~'`, источники `'app/admin/lang'` и `'vendor/pkg/lang'`), а не `~/app`.

Правило ровно одно: **`source` — это путь относительно `dir`**. `dir` — жёсткая граница, все языковые файлы лежат под ним, и вывести источник наружу нельзя ни одним из привычных способов:

| спека источника | результат при `'dir' => '~/app'` |
|---|---|
| `'admin/lang'` | `~/app/admin/lang` |
| `'/admin/lang'` | `~/app/admin/lang` — ведущий `/` срезается, это **не** корень ФС |
| `'~/admin/lang'` | `~/app/admin/lang` — ведущий `~` срезается, это **не** `DOCUMENT_ROOT` |
| `'./admin/lang'`, `'admin/lang/'`, `'admin//lang'` | `~/app/admin/lang` — нормализуются |
| `'/'`, `'~'` | `~/app` — сам `dir` |
| `'../lang'` | `InvalidArgumentException` |

Абсолютность выражается **только** через `dir` (`'dir' => '/opt/langs'` — законно): у источников её нет вовсе. `..` не схлопывается внутрь, а бросает `InvalidArgumentException` с указанием имени источника — громкая ошибка честнее тихого побега из корня. Пустой `dir` трактуется как `'~'`.

Технически это `ltrim($path, '/~')` плюс `Main::preparePath($rel, $root, true)` — в отличие от `View`, где источники резолвятся обычным `preparePath()` и ведущий `/` означает абсолютный путь. Расхождение намеренное: у `View` нет общего корня, у `Lang` он есть и является гарантией.

Обращение к неглавному источнику — либо явным префиксом ключа через `:` (`Lang::get('admin:menu.title')`), либо синтаксическим сахаром через `__callStatic` (`Lang::admin('menu.title')` — идентично, все аргументы после ключа перетекают в `get()` как есть). Источник `'~'` добавляется автоматически и указывает на сам `dir`, если явно не переопределён, — поэтому `Lang::get('some.key')` без префикса работает всегда.

### Пер-сорсные оверрайды

Источник в `spec`-форме (`['path' => путь, ...]`) переопределяет персонально для себя `extensions`, `dir_name` и любые ключи `cache.*`; всё, что не задано, наследуется от глобального конфига через `Main::merge` — то есть `['cache' => ['use' => false]]` выключает кеш только этому источнику, сохраняя ему глобальные `driver`/`dir`. Источник, заданный просто строкой-путём, целиком берёт глобальные значения. Merge выполняется один раз при построении карты источников (`sources()`), а не на каждом обращении.

Это же — замена прежнему `cache.exclude`: динамический каталог, который нужно всегда читать живьём, оформляется отдельным источником с `'cache' => ['use' => false]`. Возможности исключить *подкаталог внутри* кешируемого источника (прежние prefix/regex-правила) больше нет — это осознанный размен на явность.

Источник с пустым путём (`''`) молча пропускается (не регистрируется) — способ отключить источник, не удаляя ключ из конфига.

### Зарезервированные имена

Константа `RESERVED` — `get, set, has, choice, config, setConfig, applyConfig, purge, getUsedFiles, sources, record, stopRecording, on, trigger, lockLocale`. Источник с именем из этого списка бросает `LogicException` уже при построении карты источников (`sources()`), не дожидаясь первого обращения к ключу.

Список закрывает один риск: конфликт с **существующим публичным** методом. Для всех перечисленных имён `__callStatic` не сработает вовсе — PHP вызовет настоящий метод, и источник с именем `on` дал бы `ArgumentCountError` где-то в недрах вместо понятной ошибки про источник. `trigger` приходит из `HasEvents` вместе с `on`: алиас `on as private _on` добавляет имя, но не убирает остальные методы трейта. Прежние `setLocale`/`getLocale`/`setFallback`/`getFallback` в списке больше не держатся — этих методов давно нет (см. «Локаль» ниже), а резервировать имя несуществующего метода незачем: источник с таким именем просто некому шадовить.

### `__callStatic(string $name, array $args)`

Любое имя статического метода, не входящее в `RESERVED` и присутствующее среди зарегистрированных источников, трактуется как имя источника: `Lang::admin('menu.title', $params)` превращается в `Lang::get('admin:'.'menu.title', $params)`. Первый переданный аргумент обязателен и приводится к строке как ключ; остальные аргументы прокидываются в `get()` как `$params, $default, $locale`. Неизвестное имя (не источник и не зарезервированное) — `BadMethodCallException`.

## Разрешение ключа: перелив между ФС и массивом

Это центральная идея класса. `Lang::get('catalog.list.product.name')` находит значение во **всех** этих равнозначных раскладках:

- `/catalog/list/product/ru.php => ['name' => ...]` (самое глубокое разбиение — отдельный каталог на каждый сегмент пути);
- `/catalog/list/ru.php => ['product' => ['name' => ...]]` (частичное разбиение — остаток пути живёт внутри массива файла);
- корневой `/ru.json` с полным вложенным путём `['catalog' => ['list' => ['product' => ['name' => ...]]]]` (разбиения нет вовсе).

При конфликте между кандидатами побеждает **более глубокий (специфичный) файл**; в пределах одного каталога — расширение с более высоким приоритетом (порядок в `extensions`).

`find(string $key, string $locale, ?bool &$isScope = null)` — точка входа резолвинга одного ключа:

1. Сперва проверяется runtime-оверрайд (`self::$runtime[$locale][$key]`, точное совпадение полного ключа, включая префикс источника, если он был) — высший приоритет, минует всё остальное.
2. Ключ разбивается `splitKey()` на имя источника (часть до `:`, по умолчанию `'~'`) и путь (часть после); пустой путь — это корневой scope источника, а не `ABSENT`.
3. Источник должен быть зарегистрирован (`sources()`), иначе `RuntimeException`.
4. Ключ мог быть переписан событием `source_call` ещё в `__callStatic()`, если обращение шло через сахар (см. ниже).
5. Берётся карта (`fullMap()`) — **всегда**, независимо от `cache.use`.
6. `resolve()` перебирает кандидатов с наследованием и возвращает первое точное попадание; если его нет — собирает scope-поддерево.

`$isScope` — out-параметр: истинен, только когда значение собрано как поддерево (шаг 6, вторая половина). `get()` по нему решает, мержить ли результат с fallback-локалью; без этого признака фраза-массив (формы множественного числа) склеилась бы с чужой локалью. См. «Чтение scope».

Порядок в шаге 5 важен: `fullMap()` зовётся **до** проверки мемо `$found`, потому что `recordDeps()` внутри обязан отработать и на попадании — иначе кешируемая граница `View` потеряет языковые файлы из зависимостей и не будет инвалидироваться при их правке.

## Наследование фраз

Фраза, объявленная в родительском каталоге, видна в дочерней области под своим коротким именем. `resolve()` строит кандидатов отбрасыванием **левого** сегмента до упора и берёт первое попадание:

```
page:index.copy  ->  index.copy   (нет)  ->  copy   (есть, из pages/ru.php)
```

**Аддитивно**: включается только после промаха, поэтому ключи, которые резолвятся сегодня, резолвятся ровно так же. На каждом кандидате дополнительно проверяется runtime-оверрайд по переписанному ключу (`source:candidate`) — иначе `set('page:index.copy')` не увидел бы вызов `page('copy')`, переписанный обработчиком.

Три осознанных острых угла:

- **Промежуточный кандидат может попасть в чужую область.** `admin.users.title` → `users.title` → `title`; средний кандидат совпадёт с `pages/users/ru.php['title']`, то есть с фразой другой страницы. Это прямое следствие перелива: плоская карта намеренно считает `pages/users/ru.php['title']` и `pages/ru.php['users']['title']` одним ключом `users.title`, поэтому различить их на этапе поиска нечем.
- **Наследование идёт раньше fallback-локали.** `get()` пробует fallback только после полного промаха `find()`, а наследование живёт внутри `find()`. Значит общая фраза в текущей локали побеждает специфичную в fallback — лучше общая строка на нужном языке, чем точная на чужом.
- **Значение `null` останавливает наследование.** `mapLookup()` использует `array_key_exists`, поэтому `null` — это попадание, а не `ABSENT`: он блокирует и наследование, и fallback, и `$default`. Единственный способ явно перекрыть унаследованную фразу «пустотой».

Результат резолва мемоизируется в пределах запроса в `$found` по `(источник, локаль, переписанный путь)` — парой `[значение, $isScope]`, включая `ABSENT`. Карта в пределах запроса неизменна, а без мемо отсутствующий ключ гонял бы полный скан карты по разу на кандидата, да ещё дважды — в локали и в fallback. Для scope-запросов цена промаха ещё выше: там на каждый слой идёт `foreach` по всей карте.

## Чтение scope

Если ни один кандидат не дал точного попадания, `resolve()` не сдаётся, а собирает **scope-поддерево** — все фразы, видимые из запрошенной области, вместе с унаследованными. Отсюда центральный инвариант:

```
get("$src:$scope")[$L] === get("$src:$scope.$L")   для любого $L
```

То есть массивное чтение обязано давать ровно то же, что поштучное. Всё остальное — следствия.

`scopeTree()` накладывает слои от слабого к сильному: корень источника целиком, затем та же цепочка кандидатов, что перебирает `resolve()`, только развёрнутая — от самого общего к самому частному. Для `a.b` это `['', 'b', 'a.b']` (именно `b`, а не `a`: путь укорачивается **слева**, см. острые углы выше). Сверху ложится runtime-оверлей.

С учётом локалей полный порядок приоритета:

```
en:''  <  en:index  <  ru:''  <  ru:index  <  runtime[en]  <  runtime[ru]
```

Локаль — внешний цикл, кандидат — внутренний, поэтому «наследование раньше fallback» сохраняется и здесь: `ru:copy` (общая, текущая локаль) бьёт `en:index.copy` (специфичную, fallback) — ровно как у скалярного чтения.

Три следствия, о которых надо знать:

- **Родительский слой отдаёт всё.** `get('page:index')` вернёт не только общие фразы, но и `task`/`404` подмассивами, и сам `index` внутри себя. Это цена инварианта: `get('page:index.task.back')` резолвится (через укорачивание до `task.back`), значит `get('page:index')['task']['back']` обязан существовать. Плоская карта принципиально не отличает корневой лист от дочерней страницы — это тот же перелив, что делает `pages/index/ru.php` и `pages/ru.php => ['index' => …]` равнозначными.
- **Любой непопавший путь с поддеревом — это scope.** `get('page:index.form')` вернёт корень источника с наложенным `form.*`, а не только `['submit' => …]`. Следствие инварианта, но выглядит неожиданно, если ждать «только своё».
- **Промах остаётся промахом.** Если ни один кандидат не дал непустого поддерева — `ABSENT`, дальше `$default ?? $key`. Без этой проверки `get('page:опечатка')` возвращал бы корень источника целиком вместо самого ключа.

`$params` в scope не подставляются: `strtr()` в `get()` работает только при `is_string($value)`. Плейсхолдеры внутри массива остаются сырыми — подставлять их надо на месте использования, поштучным `get()`.

## События

`Lang::on(string $event, callable $listener)` — статический фасад над `Traits\HasEvents` поверх синглтона (`Traits\HasInstance`), тот же приём, что в `Access`/`Debug`. Обработчики меняют значения **по ссылке**: возвраты `HasEvents::fire()` отбрасывает.

**`source_call(string $source, &$key)`** — срабатывает в `__callStatic()`, то есть **только на сахаре** `Lang::<источник>(...)`, и позволяет переписать ключ до резолва. Основной сценарий — автоматический префикс страницы, чтобы не писать его руками в каждом шаблоне:

```php
Lang::on('source_call', function ($source, &$key) {
    if ($source === 'page' && ($n = View::name()) !== '') $key = $n.'.'.$key;
});
```

Тогда `Lang::page('speech_recognition')` резолвится как `page:index.speech_recognition`, а `Lang::page('copy')` наследуется в `pages/ru.php`. Обработчик обязан гардить `View::name() !== ''` — вне рендера имя пусто. Событие зарезервировано (`getReservedEvents()`), внешний `trigger('source_call', …)` запрещён.

**`Lang::get()` — escape hatch: событие на него не действует.** Это и есть причина, по которой оно живёт в `__callStatic`, а не в `find()`. `find()` видит `Lang::page('title')` и `Lang::get('page:index.title')` одинаково — как (источник `page`, путь …) — и дописал бы префикс обоим. Двойной префикс (`index.index.title`) наследование ещё замаскировало бы, отбросив левый сегмент, но не спасло бы главный случай: если фраза объявлена **и** в корне, **и** в области страницы, то явный `Lang::get('page:collide')` вернул бы страничную фразу вместо запрошенной корневой — молча и неверно. В `__callStatic` сахар отличим от явного вызова, поэтому `get()` всегда резолвит ровно то, что передали.

Срабатывает **один раз на вызов**, а не на попытку резолва: `get()` может звать `find()` дважды (локаль и fallback), но ключ к тому моменту уже переписан, поэтому префикс не удваивается.

Непереведённый ключ возвращается **уже переписанным** (`page:index.copy`, не `page:copy`) — это реальный путь резолва, по нему видно, где фразу искали.

⚠️ Если обработчик опирается на `View::name()`, потребитель **обязан** добавить `View::name()` в `cache.key` кеша `View` — иначе общий кешируемый компонент получит одинаковый ключ на разных страницах и покажет чужую фразу. Подробности и пример — в [View.php](View.php.md).

### `mapLookup(array $map, string $path)` — поиск по плоской компилированной карте

Два сценария, в порядке проверки:

1. **Точный ключ** — `$map[$path]` существует напрямую.
2. **Лист-массив по префиксу** — если точного ключа нет, путь укорачивается посегментно (`errors.0` → пробуем `errors`); как только находится существующий ключ-префикс, а его значение — массив (списки не дот-флаттенятся при сборке карты, см. `Main::dotFlatten`/`arrayIsList`), в него подставляется `Main::dotGet` по оставшемуся хвосту пути (`errors.0` → `$map['errors'][0]`).

Промах — `ABSENT`; сборкой поддерева `mapLookup()` не занимается, это дело `scopeTree()` (см. «Чтение scope»).

### Сборка поддерева

- `fullTree()` — вся карта, развёрнутая `Main::dotSet()` в дерево. Мемоизируется в `$trees` по (источник, локаль): без этого каждый scope-запрос разворачивал бы карту заново.
- `subTree()` — то же, но только по ключам с префиксом `path.`.
- `runtimeTree()` — runtime-оверрайды под scope. Префикс собирается из имени источника и пути; у источника по умолчанию (`'~'`) префикса нет, поэтому ключи с `:` при пустом пути отбрасываются как чужие.
- `mergeTree()` — наложение слоёв. **Не** `Main::merge()`: тот мержит списки поэлементно, из-за чего формы множественного числа из разных локалей склеились бы (`['one','many']` поверх `['一','二','三']` оставило бы хвост чужого языка), и имеет спецветку для замыканий, которая тут не нужна. Правило: оба значения — ассоциативные массивы → рекурсия; иначе (список или скаляр) дочернее заменяет родительское целиком.

## Кеш: компилированная карта

`compiledMap(string $sourceName, string $locale): array` — одна плоская карта (`dot-key => значение`) на пару (источник, локаль), кешируемая через `CacheManager`.

`fullMap()` — вход для резолвинга: при `cache.use = true` отдаёт `compiledMap()`, иначе строит карту через `buildMap()` и держит её только в памяти процесса (`$maps`). **`cache.use` влияет на наличие дисковой записи, но не на семантику**: обе ветки резолвят по одной и той же карте, поэтому результат идентичен побайтово. Цена выключенного кеша — полный обход дерева источника на каждый процесс.

Ключ кеш-записи — **ассоциативный** массив из класса, имени источника, локали и того, что влияет на содержимое карты: `dir`, список `ext` (строкой — порядок расширений это приоритет) и `dir_name`. Ассоциативность здесь обязательна: `Main::hash()` для *списка* сортирует элементы (`sort($parts, SORT_STRING)`), из-за чего списочный ключ теряет чувствительность к порядку — при двух источниках на одно дерево (`['en' => '~/lang', 'ru' => '~/lang']`) пары (источник `en`, локаль `ru`) и (источник `ru`, локаль `en`) схлопнулись бы в одну запись, и ключ вернул бы карту чужого языка. Ассоциативная ветка `Main::hash()` порядок сохраняет.

В ключ намеренно **не** входят `cache.use` (на содержимое не влияет — иначе переключение плодило бы осиротевшие записи) и сырой оверрайд источника целиком: он мог бы принести `\Closure`, а `Main::hash()` хеширует замыкания по `file:startLine:endLine` — путь абсолютный, поэтому при атомарных деплоях через симлинк ключ менялся бы каждый релиз, а замыкание, захватившее ресурс, роняло бы `RuntimeException: Unsupported type in hash` прямо из `Lang::get()`.

Свежесть определяется единственным способом — сравнением стампа: если мета содержит `deps`/`stamp` и `stamp` совпал с пересчитанным `depStamp($meta['deps'])`, карта читается из кеша. При промахе — `buildMap()` полностью обходит дерево источника заново, кеш перезаписывается вместе с метой (`stamp`, `deps`, `files`); при попадании файлы из `meta['files']` всё равно помечаются использованными (`getUsedFiles()` должен отражать реально задействованные файлы независимо от того, был ли это hit кеша или полная пересборка). Внутрипроцессный memo (`self::$maps`) дополнительно исключает повторное обращение к `CacheManager` за тот же (источник, локаль) в пределах одного запроса.

### `buildMap()` — полный обход источника

BFS по дереву каталогов источника. Если `dir_name` пуст, языковые файлы могут находиться в любом каталоге дерева наравне с прочим содержимым (`isLang` истинен с самого начала обхода); если `dir_name` задан (например `'lang'`), обход помечает каталог как "языковой" только при входе в подкаталог с этим именем, и **не спускается глубже** внутрь него — файлы фраз должны лежать непосредственно в `dir_name`, без собственной вложенности.

Найденные кандидаты сортируются перед наложением на карту: по возрастанию глубины, а при равной глубине — по убыванию приоритета расширения. Поскольку каждый следующий элемент **перезаписывает** совпадающие ключи предыдущего, такой порядок гарантирует «глубокий файл побеждает мелкий, а внутри одного каталога — файл с более приоритетным расширением побеждает».

В зависимости (`deps`) собираются **и файлы, и все посещённые каталоги** — новый или удалённый файл/подкаталог меняет mtime родителя, поэтому `depStamp()` ловит структурные изменения дерева без повторного сканирования, сравнивая только mtime уже известного списка путей.

### `depStamp(array $paths): string`

`clearstatcache()`, затем карта `путь => filemtime() или 0, если файла больше нет`, хешируется через `Main::hash()`. Это единственный сигнал инвалидации: `touch` без изменения содержимого тоже инвалидирует кеш, а изменение содержимого с сохранением mtime (редкий случай ручных манипуляций с ФС) — не инвалидирует. Тот же принцип, что и в `View::depStamp()`.

## `isSystemPath()` — почему кеш обходит сам себя

Ненастраиваемый внутренний инвариант, а не пользовательская опция. `buildMap()` пропускает при обходе каталоги кеша: `cache.dir` каждого зарегистрированного источника плюс глобальный `CacheManager::config('default.dir')` (список резолвится один раз и мемоизируется в `self::$systemPaths`).

Без этого класс ломается. Источник по умолчанию — весь document root, а `cache.dir` по умолчанию — `~/cache/Lang`, то есть кеш физически лежит **внутри сканируемого дерева**, а `buildMap()` кладёт в `deps` каждый пройденный каталог. Петля выглядит так: как только карта хоть раз пересобирается при уже существующем каталоге записи, этот каталог попадает в её собственные `deps`; `FileSystemCacheDriver::writeMeta()` пишет мету через временный файл (`.tmp.<random>` + переименование), а создание и удаление записи в каталоге двигает его mtime — значит **каждая** запись кеша сдвигает mtime собственной зависимости. Следующее обращение видит расхождение стампа, пересобирает карту, снова пишет, снова сдвигает — и так на каждом запросе, навсегда, молча и без единой ошибки.

Тонкость, из-за которой это тяжело поймать: `filemtime` имеет разрешение 1 секунда, поэтому в быстрых тестах первый и второй проход попадают в одну секунду и петля не проявляется. Проверять надо минимум тремя проходами, разнесёнными по секундам, с принудительной пересборкой между первым и вторым.

⚠️ `isSystemPath()` резолвит `cache.dir` через `Main::preparePath($dir)` (backtrace-фрейм 0), а `FileSystemCacheDriver` — через `Main::preparePath($config['dir'], 3)` (фрейм 3). Для `~`-якорных и абсолютных путей ветка резолва базу игнорирует и они совпадают — это дефолт и норма. Для **относительного** `cache.dir` пути разойдутся, prune промахнётся мимо настоящего каталога записи и петля вернётся. `cache.dir` должен быть `~`-якорным или абсолютным.

## Runtime-оверрайды (`set()`)

`Lang::set($key, $value = null, ?string $locale = null)` — фразы, заданные в рантайме, живут только в памяти текущего процесса (`self::$runtime`) и имеют наивысший приоритет разрешения (проверяются в `find()` раньше файловой системы и кеша). `$key` может быть массивом — тогда он дот-флаттенится (`Main::dotFlatten`) и каждый получившийся плоский ключ пишется отдельно; либо строкой — тогда это единственная пара ключ/значение. Ключи хранятся ровно в том виде, в котором переданы, включая префикс источника через `:`, если он был указан явно.

## Публичный API чтения

- **`get(string $key, array $params = [], $default = null, ?string $locale = null)`** — резолвит `$key` в текущей (или явно указанной) локали. Отдаёт либо **фразу**, либо **scope целиком** (вложенный массив всех видимых из этой области фраз, вместе с унаследованными) — что именно, зависит от того, попал ли ключ точно; см. «Чтение scope». Fallback-локаль (только если задана и отличается от текущей) для фразы подставляется при `ABSENT`, а для scope **мержится послойно**, так что частично переведённая область добирает недостающие ключи из fallback. При повторном `ABSENT` — `$default ?? $key` (Laravel-стиль: непереведённая строка возвращает сам ключ, а не `null`, что удобно как явный визуальный признак отсутствующего перевода прямо в верстке). Если результат — строка и переданы `$params`, выполняется `strtr()`-подстановка плейсхолдеров; в scope плейсхолдеры не подставляются.

  `get('page:')` (пустой путь) — весь источник целиком: корневые фразы плюс каждая страница отдельным подмассивом.
- **`choice(string $key, $number, array $params = [], $default = null, ?string $locale = null): string`** — форма множественного числа. Важный нюанс: индекс формы выбирается по правилам **той локали, где реально нашлось значение** (`$used`), а не запрошенной — то есть если ключ разрешился через fallback, используются правила плюрализации fallback-локали, а не исходной. Значение-массив трактуется как готовый список форм (`array_values`); значение-строка разбивается по `|` с обрезкой пробелов (Laravel-совместимый формат `"one|few|many"`). `params` дополнительно получает `count => $number` перед подстановкой плейсхолдеров.
- **`has(string $key, ?string $locale = null): bool`** — проверка наличия без выброса — резолвит в текущей локали, затем (если применимо) в fallback.
Отдельного `all()` нет: он отдавал префикс без наследования, то есть `all('page:index')` возвращал ровно те же «только свои» 9 ключей, что и `get('page:index')`, отличаясь лишь померженным fallback. Оба сценария теперь закрывает `get()`, а `'all'` свободно как имя источника.

## `purge()` и `getUsedFiles()`

`purge(): void` — деплой-хук: обходит все зарегистрированные источники и вычищает физические кеш-записи по каждой уникальной паре `(cache.driver, cache.dir)` — то есть пер-сорсные кеш-каталоги тоже, а не только глобальный, — плюс сбрасывает внутрипроцессные memo (`self::$maps`, `self::$fileData`). Не трогает `self::$sources` — это память конфигурации, а не кеша фраз.

⚠️ `purgeBase()` сносит базовый каталог **рекурсивно**. Если источнику задать `cache.dir` без собственного подкаталога (например прямо `~/cache`, значение `CacheManager::config('default.dir')`), `Lang::purge()` снесёт заодно кеши `View`, `Assets` и `Storage\File`. Пер-сорсный `cache.dir` должен указывать на свой подкаталог.

`getUsedFiles(): array` — плоский накопительный список путей всех языковых файлов, реально поучаствовавших в разрешении хоть одного ключа за время жизни процесса (независимо от того, шла карта через полную пересборку или cache-hit).

## Рекордер зависимостей

`record(): void` / `stopRecording(): array` — контракт для внешнего кеша. `View` зовёт эту пару **напрямую**, вокруг `build()` каждой кешируемой границы, чтобы правка перевода инвалидировала отрендеренный HTML — см. [View.php](View.php.md).

**Механизм** — тот же, что у [Assets](Assets.php.md): стек кадров, запись всегда во внутренний кадр (`array_key_last`), `stopRecording()` снимает кадр и отдаёт накопленное. **Контракт — другой, и подменять их друг другом нельзя:** `Assets::stopRecording()` отдаёт *операции* (порядок значим, они уезжают в payload скелета и **replay**-ятся на хите, восстанавливая побочный эффект), а `Lang::stopRecording()` — *пути* (порядок не значим, они уезжают в `deps` и участвуют в подсчёте стампа). Совпадение имён — следствие общего механизма, а не взаимозаменяемости.

`stopRecording()` отдаёт зависимости, задействованные **за время** записи: и файлы, и **каталоги**. Каталоги обязательны — mtime каталога единственный сигнал о *появлении* нового языкового файла, и без них создание `pages/index/ru.php` не инвалидировало бы ничего у потребителя.

Почему именно пара record/stop, а не накопительный `getUsedFiles()` с диффом вокруг рендера: зависимости источника отмечаются **на каждом** обращении к `compiledMap()`/`fullMap()`, включая мемо-хит и хит блоба (для этого рядом с мемо лежит `self::$mapDeps`). Накопительное множество этого не выражает — дифф вокруг второй границы был бы пуст, потому что первая уже «забрала» файлы себе, и вторая осталась бы без зависимостей, то есть с навсегда устаревшим HTML. По той же причине мемо резолва (`self::$found`) стоит **после** загрузки карты, а не до.

Холодный путь (`cache.use = false`) отмечает только файл, реально отдавший значение: файл, который проверили и промахнулись, в зависимости не попадёт, поэтому появление в нём нужного ключа потребителя не инвалидирует. При включённом кеше (по умолчанию) карта отмечает весь список файлов источника разом, и этой дыры нет.

## Плюрализация и плейсхолдеры

`pluralIndex(int $n, string $locale): int` — приватный, только слой пользовательских правил: проверяет `config('plural')[$locale]` (пользовательский `callable`, получает **сырой** `$n`, результат приводится к неотрицательному int), иначе делегирует встроенные схемы в `Main::pluralIndex()` — единственное место, где живёт само правило склонения.

`placeholders(array $params): array` строит карту для `strtr()` в трёх регистровых вариантах на каждый параметр — `:name` (как есть), `:Name` (`mb`-safe первая заглавная), `:NAME` (`mb_strtoupper`) — так авторам файлов фраз не нужно заботиться о регистре плейсхолдера в самой строке перевода, `strtr` подставит нужный вариант в зависимости от того, что реально написано в тексте. Варьируется регистр **в шаблоне**, а не в имени параметра: передав `['name' => 'мир']`, можно писать `:name`, `:Name` или `:NAME` — подставится соответственно `мир`, `Мир`, `МИР`.

## Локаль: `config('locale')` / `config('fallback')`

Отдельной runtime-оси нет: локаль и fallback живут только в конфиге, читаются через `Lang::config('locale')`, пишутся через `Lang::setConfig('locale', 'en')` (или массивом). Прежние `setLocale`/`getLocale`/`setFallback`/`getFallback` удалены — они дублировали конфиг: `setConfig` пишет в статический массив в памяти (`Config::setImmutableConfig` → `Main::dotSet`) и **не переживает процесс**, ровно как и снятые статики, так что «глобальный дефолт против выбора для этого запроса» было различием без разницы.

`config()` переопределён: для ключей `locale` и `fallback` он резолвит `callable`/`\Closure`-значение вызовом (`resolveLocaleValue()`) — **при каждом обращении**, что и делает осмысленной локаль, зависящую от текущего запроса, в том числе в долгоживущих воркерах. Мемоизации нет намеренно; если локаль на запрос фиксирована, дешевле положить в конфиг строку (`setConfig('locale', $x)` в начале запроса) — на строке резолв выходит сразу.

Рекурсия (`'locale' => fn() => Lang::get(...)`, где `get()` сам читает локаль) ловится гардом `self::$resolving` и бросает `LogicException` вместо переполнения стека.

⚠️ Асимметрия: `config('locale')` резолвит замыкание, а `config('')['locale']` — нет, там лежит сырое значение. Так и задумано: `config('')` — это снимок конфига целиком, и резолв на нём выполнял бы пользовательские замыкания при каждом чтении всего конфига, вмораживая в снимок локаль одного момента. Практическое следствие — `applyConfig()` (итерирует `config()` и зовёт `config($k)`) на `Lang` вбил бы резолвнутую строку в чужой массив; сейчас `Lang::applyConfig` не вызывается нигде.

### `lockLocale(?string $locale = null): void` — заморозка локали

Одноразовый переключатель: после вызова `locale` больше нельзя менять через `setConfig()` — попытка бросает `LogicException`. Повторный `lockLocale()` тоже бросает `LogicException` («локаль уже заблокирована») — метод рассчитан ровно на один вызов за время жизни процесса.

```php
Lang::lockLocale();          // фиксирует то, что сейчас разрешается config('locale')
Lang::lockLocale('ru');      // сперва пишет 'ru' в конфиг, затем фиксирует его
Lang::setConfig('locale', 'en'); // LogicException: локаль заблокирована
```

Механика в две строки: если передан `$locale`, он сначала пишется в конфиг через приватный алиас `traitSetConfig` (исходный `HasConfig::setConfig()`, до переопределения в `Lang`) — в обход публичного `Lang::setConfig()`. На момент этой записи `self::$localeLocked` ещё `false`, так что для этого конкретного вызова разницы с публичным `setConfig()` нет; это просто более прямой путь записи, тот же приём, что и у остальных `trait ... as private` алиасов в классе. Затем **вне зависимости** от того, был ли передан аргумент, текущее значение перечитывается через `self::config('locale')` и пишется обратно тем же `traitSetConfig`. Если локаль была задана `callable`/`\Closure` (см. выше), это резолвит её **один раз** и подменяет замыкание на конкретную резолвнутую строку — после `lockLocale()` `config('locale')` больше не вызывает пользовательский колбэк, а отдаёт зафиксированную строку.

`self::$localeLocked` — статическое поле, общее для всего процесса, и сбросить его снаружи нечем: `lockLocale()` рассчитан на однократный вызов **за время жизни процесса**, а не за запрос. В классическом модели «один запрос — один процесс» разницы нет. В долгоживущем воркере (Swoole/RoadRunner-стиль) повторный вызов на втором запросе бросит `LogicException`, поэтому там `lockLocale()` уместен только для сценария «локаль фиксируется один раз при старте воркера и остаётся неизменной для всех запросов, которые он обслужит» — не как per-request гвард. Практический эффект блокировки — гарантия, что ни один модуль или middleware дальше по цепочке не переопределит `locale` на середине обработки, плюс однократный резолв замыкания вместо повторного на каждое обращение к `config('locale')`.
