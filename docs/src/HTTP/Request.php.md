# Request.php
<!-- DOCGEN:END -->

`namespace ST_system\HTTP`

## Назначение

`Request` — обёртка над PHP-суперглобалами (`$_GET`, `$_POST`, `$_SERVER`, `$_COOKIE`, `$_FILES`) с ленивым, мемоизированным разбором атрибутов запроса, опциональным сопоставлением с route-паттерном (`/users/{id}`) и валидацией входных данных через `Rule`. Класс **не `final`** — он специально спроектирован для наследования: `HTTP\Route` рефлексирует тип первого параметра контроллера маршрута, и если это класс-потомок `Request`, инстанцирует именно его (`$request_class::fetch($query_params)`) и передаёт в middleware/контроллер вместо базового `Request`. Подклассы переопределяют `__schema()`, чтобы объявить правила валидации для конкретного маршрута, и/или `__init()` для дополнительной инициализации.

`Access.php` использует статический интерфейс класса (`Request::get()`, `Request::post()`, `Request::headers()`, `Request::cookie()`, `Request::method()`) для извлечения данных при формировании ключей доступа/скоупов.

## Singleton-per-fetch

- `static fetch(array $query_params = []): self` (final) — создаёт **новый** объект и кладёт его в `static::$instance`, полностью заменяя предыдущий инстанс. Это **не идемпотентная** операция: повторный вызов `fetch()` пересоздаёт запрос заново (например, заново читает `php://input`).
  **Важный нюанс наследования** (проверено эмпирически): `$instance` объявлено как `private static` **только один раз** в базовом классе и ни разу не переобъявлено в наследниках. В PHP статическое свойство, не переобъявленное в дочернем классе, — это **одна общая ячейка хранения на всю иерархию классов**, а не отдельная копия на каждый класс-потомок, несмотря на позднее статическое связывание (`static::`) в выражении `static::$instance = ...`. То есть `Base::fetch()` и `Child::fetch()` (где `Child extends Base` и не объявляет своё `$instance`) пишут и читают **одну и ту же** ячейку: вызов `Child::fetch()` перезатирает значение, которое затем увидит и `Base::peek()`/любой другой класс иерархии, если тот не переобъявил `$instance` у себя. Практическое следствие для `Request`: если в одном процессе (например, в одном PHP-воркере, обрабатывающем несколько запросов, либо при последовательном использовании разных подклассов `Request` в рамках одного роутинга) вызывается `fetch()` сначала для одного подкласса `Request`, а затем для другого, второй вызов **перезапишет** общий `static::$instance`, и предыдущий сохранённый где-то инстанс первого подкласса продолжит существовать как объект, но глобальный "текущий" слот `$instance`, на который опирается `__callStatic()`, уже будет указывать на второй. Иными словами, `$instance` — это **не по одному инстансу на класс-наследник**, а фактически один общий слот на весь `Request` и всех его потомков, если ни один из них явно не переобъявляет собственное `private static $instance`.
- `static __callStatic(string $name, array $arguments)` (final) — точка входа для статического стиля вызова (`Request::post('id')`, `Request::method()`). Если инстанса ещё нет — лениво создаёт его через `fetch()`. Если `$name` — реально существующий метод класса, форвардит вызов на инстанс as-is. Иначе трактует `$name` как имя "магического" атрибута (см. `__call()` ниже): резолвит значение и, если передан `$arguments[0]` как ключ, а результат оказался массивом, возвращает `$value[$key] ?? null`, иначе — значение целиком. Именно этот механизм даёт короткую запись `Request::headers('User-Agent')` или `Request::post('name')`.

## Конструктор и хуки для подклассов

`__construct(array $query_params = [])` — приватный (создание объекта возможно только через `fetch()`):
1. Сохраняет переданные `$query_params` в `$this->data['query']` (сырые параметры маршрута, если `pattern()` не используется).
2. Вызывает protected-хук `__schema(): array` (в базовом классе возвращает `[]`, переопределяется в наследниках). Если схема не пуста — прогоняет `validate($schema)`.
3. Вызывает protected-хук `__init(): void` (в базовом классе пустой) — точка расширения для доп. инициализации подкласса, вызывается после того, как валидация (если была) уже применена.

## `pattern()` — извлечение параметров из route-шаблона

`pattern(string $route_template, bool $strict_mode = true): self` (приватный) строит regex для сопоставления URI с шаблоном вида `/users/{id}/posts/{post_id}`:

- Список имён параметров пишется в `$this->data['query_keys']`.
- В `strict_mode = true` (по умолчанию) весь `$route_template` сначала целиком экранируется через `preg_quote()` — то есть любые буквальные `{`/`}` в других частях URL воспринимаются буквально, а не как синтаксис. После этого распознаются **только** последовательности вида `\{name\}` (экранированные `preg_quote`-ом фигурные скобки, оставшиеся от исходных `{name}`), которые и превращаются в именованные группы захвата `(?P<name>[^/]+)`. Это позволяет одновременно безопасно матчить произвольные литеральные URL-символы и распознавать собственный синтаксис плейсхолдеров фреймворка.
- В `strict_mode = false` вместо экранированной последовательности ищутся обычные `{name}`.
- Готовый regex сохраняется в `$this->data['route_template']` в виде `#^...$#` (двойные `/` в шаблоне схлопываются в одинарные).
- Побочный эффект: `unset($this->data['query'])` — сырые `$query_params`, переданные в конструктор, стираются, потому что теперь `query()` должен вычисляться из сопоставления URI с паттерном, а не браться напрямую. Использование `pattern()` и конструкторских `$query_params` взаимоисключающее.

## Валидация: `validate()` и `check()`

Оба метода приватные и берут схему в формате, ожидаемом `Rule::object()`.

- `validate(array $schema): array` — сначала форсирует резолв `$this->data()` (объединённый bag `get+post+query+files`, см. ниже), затем внутри `Rule::scope(static::class, ...)` строит `Rule::object($schema)`, при `$this->throwable === true` включает `->throwable()` (тогда ошибки валидации бросают исключение вместо возврата массива ошибок), и вызывает `->apply($this->data['data'])`. `apply()` — мутирующий вариант: он не только проверяет, но и приводит типы/значения в переданном массиве `data`.
  **Важный побочный эффект**: после применения правил метод проходит по `get`, `post`, `query`, `files` и для каждого ключа, который присутствует в этих массивах, либо переписывает его значением из провалидированного `$this->data['data']` (если ключ там есть), либо **удаляет** ключ из соответствующего массива (если провалидированный `data` его не содержит). То есть `validate()` не просто репортит ошибки — она обрезает и коэрсит содержимое `get()/post()/query()/files()` под результат валидации. Это существенно для авторов подклассов: после вызова `validate()` (например, из `__schema()`) публичные геттеры отдают уже отфильтрованные/типизированные данные, а не сырые суперглобалы.
- `check(array $schema): array` — то же самое, но использует `Rule::object($schema)->check()` вместо `->apply()`: это версия только для проверки, не мутирует исходные данные и не производит побочного обрезания `get/post/query/files`.
- `throwable(bool $bool = true): self` — переключает флаг `$this->throwable`, влияющий на поведение `validate()`/`check()` при ошибках валидации.

## `__call()` — ленивый резолвер атрибутов

`__call(string $name, array $arguments)` (final) обслуживает весь "магический" API инстанса (`$request->post('id')`, `$request->uri()` и т.д.):

1. Если `$name` — существующий метод объекта и не начинается с `_` (внутренние/сырые имена вроде `_post`, `_query` не должны быть перехвачены этой веткой при прямом вызове через `__call`, хотя фактически PHP не станет вызывать `__call` для существующих публичных методов вообще — эта проверка защищает вызовы приватных методов типа `pattern`, `validate`, `check` через `$request->pattern(...)`), вызывает его напрямую.
2. Иначе — если атрибут ещё не вычислен (`!isset($this->data[$name])`), выполняет `switch` по имени и мемоизирует результат в `$this->data[$name]`:
   - **`uri`** — путь из `$_SERVER['REQUEST_URI']` (`parse_url(..., PHP_URL_PATH)`), по умолчанию `/`.
   - **`host`**, **`port`** — берутся напрямую из `$_SERVER['HTTP_HOST']` / `$_SERVER['SERVER_PORT']`.
   - **`scheme`** — `https`, если `$_SERVER['HTTPS'] === 'on'`, иначе `http`.
   - **`origin`** — `"{scheme}://{host}"`.
   - **`url`** — `"{origin}{uri}"`.
   - **`method`** — `$this->post('_method') ?: $_SERVER['REQUEST_METHOD']`. Это классический приём **method override**: HTML-формы умеют отправлять только `GET`/`POST`, но добавив скрытое поле `_method` со значением `PUT`/`PATCH`/`DELETE`, можно заставить `Request::method()` вернуть нужный "виртуальный" HTTP-глагол. Поле читается уже из `post()` (провалидированного/смёрженного с JSON), а не из сырого `$_POST`.
   - **`headers`** — перебирает весь `$_SERVER`, берёт ключи с префиксом `HTTP_`, отбрасывает префикс и превращает `SOME_HEADER_NAME` в `Some-Header-Name` (через `ucwords`/`str_replace`), формируя ассоциативный массив нормализованных имён заголовков.
   - **`_get`** — сырой `$_GET`. **`get`** — результат `_get()` (после валидации — уже отфильтрованный/коэрснутый набор).
   - **`_post`** — `array_merge($_POST, @json_decode(@file_get_contents('php://input'), true) ?? [])`. Тело запроса **автоматически** пытается быть распознано как JSON и слито поверх/рядом с обычными `$_POST` полями — так JSON API-запросы (`Content-Type: application/json`) и обычные form-post запросы обрабатываются единым способом через `post()`. Ошибки чтения потока и декодирования JSON подавляются оператором `@`, при неудаче просто ничего не добавляется (пустой merge). **`post`** — результат `_post()`.
   - **`_cookie`** / **`cookie`** — `$_COOKIE` как есть.
   - **`_query`** — если ранее был вызван `pattern()` (то есть `query_keys` не пуст) и `uri()` совпадает с `route_template`, возвращает только именованные захваченные группы (`array_intersect_key($matches, array_flip($query_keys))`); иначе — пустой массив. **`query`** — результат `_query()`, но эта ветка `switch` фактически исполняется только тогда, когда ключ `'query'` отсутствует в `$this->data` (проверка `!isset(...)` в начале `__call()`). Конструктор всегда сразу заполняет `$this->data['query'] = $query_params` (даже пустым массивом) — а `isset()` для существующего ключа-массива истинен независимо от того, пуст он или нет. Поэтому если `pattern()` **не вызывался**, `query()` всегда возвращает ровно то, что было передано в конструктор, и ветка `_query`/`query` в `switch` не выполняется вовсе. Только `pattern()` явно делает `unset($this->data['query'])`, вновь открывая путь к вычислению `query()` через сопоставление `route_template` с `uri()`.
   - **`_files`** — нормализация "рваной" структуры `$_FILES` (в т.ч. multi-file input вида `name="files[]"`) в плоский список **объектов [[UploadedFile.php]]** на каждое поле формы. Пропускаются только пустые слоты (`error === UPLOAD_ERR_NO_FILE`); все остальные записи — **включая битые** (`UPLOAD_ERR_INI_SIZE`, `UPLOAD_ERR_PARTIAL` и т.п.) — оборачиваются в `UploadedFile`, а отсеиваются уже правилами валидации `file`/`extension`/`mime` (см. [[Rule.php]]). Каждый `UploadedFile` даёт метаданные (`getFilename()`, `getExtension()`, `getMime()`, `getSize()`, `isValid()`) и `save()`, реализует `ArrayAccess` (ключи `name`, `tmp_name`, `type`, `size`, `error`, `extension`) и `toArray()` — поэтому доступ вида `$file['tmp_name']` продолжает работать прямо на объекте. **`files`** — результат `_files()` (`files('audio')` → `UploadedFile[]`). *(Ранее записи были массивами с ключом-опечаткой `extenstion`; теперь это объекты, а ключ называется корректно — `extension`.)*
   - **`_data`** — немедленный (невалидированный) объединённый bag: `array_merge($this->_get(), $this->_post(), $this->_query(), $this->_files())`.
   - **`data`** — форсирует резолв `get()`, `post()`, `query()`, `files()` (то есть их провалидированные версии, если валидация уже прошла, либо просто мемоизированные сырые), затем собирает `$this->data['data']`, проходя по `['get', 'post', 'query', 'files']` и записывая туда значения **по ссылке** (`foreach (... as $k => &$v) $this->data['data'][$k] = &$v;`). Из-за этого до момента вызова `validate()`/`check()` элементы `data['data']` являются живыми алиасами на элементы `get/post/query/files` (мутация одного отражается в другом). Как только `validate()` переприсваивает `$this->data['data'] = $rule->apply(...)` результатом валидатора, эта связь по ссылке рвётся (переприсваивание переменной, а не запись в неё), и уже сам `validate()` вручную прокидывает провалидированные значения обратно в `get/post/query/files` (см. выше).
   - **default** — бросает `\Exception("Method {$name} not found")` с англоязычным текстом (в отличие от общего стиля фреймворка, где сообщения об ошибках обычно на русском — здесь конкретно оставлено как в оригинале, поведение не меняется).
3. После резолва (или если атрибут уже был в кэше `$this->data`) применяется тот же паттерн, что и в `__callStatic`: если передан `$arguments[0]` в качестве ключа и результат — массив, возвращается `$value[$key] ?? null`; иначе — весь массив/значение целиком. Именно это даёт вызовы вида `$request->headers('User-Agent')`, `$request->post('name')`, `$request->files('avatar')`.

## Ключевые "режимы" для внимательного использования

- **Method override**: `method()` учитывает `post('_method')` раньше `$_SERVER['REQUEST_METHOD']` — формы могут подделывать `PUT`/`PATCH`/`DELETE`.
- **Авто-слияние JSON в POST**: тело запроса всегда пытается быть распарсено как JSON и добавлено к `$_POST` внутри `post()`/`_post()`, независимо от `Content-Type`; ошибки молча проглатываются.
- **`pattern()` vs конструкторские `$query_params`**: взаимоисключающие источники `query()` — вызов `pattern()` стирает переданные в конструктор параметры и переключает `query()` на извлечение именованных групп из URI.
- **Валидация как побочный фильтр**: `validate($schema)` (обычно вызывается автоматически из `__schema()` в подклассе) не только проверяет данные, но и **обрезает/коэрсит** содержимое `get()`, `post()`, `query()`, `files()` под провалидированную схему — ключи, отсутствующие в результате валидации, физически удаляются из этих массивов. Авторы подклассов должны иметь это в виду: значения, недопустимые по схеме или не описанные в ней, станут недоступны через обычные геттеры после конструирования запроса.
- **Не `final`, роутинг определяет конкретный подкласс**: `Route.php` через рефлексию типа первого параметра контроллера определяет, какой подкласс `Request` инстанцировать (`is_subclass_of($candidate, Request::class)`), и создаёт его через `$request_class::fetch($query_params)` — так маршруты получают строго типизированные, самовалидирующиеся объекты запроса. Но, как отмечено выше, `static::$instance` — это общая ячейка на всю иерархию `Request`, а не отдельный слот на каждый подкласс.
