# WebClient.php

`final class ST_system\HTTP\WebClient`

Единый HTTP-механизм фреймворка поверх `curl_multi`. Один экземпляр `WebClient` — это один URL: простой (`https://api/status`) или шаблонизированный (`{url}`, `http://t.ru/{p}/get`), способный разворачиваться через `query()` в декартово произведение комбинаций параметров и отправляться батчами.

Это низкоуровневый транспортный слой, на котором построены `IntegrationDriver`, фетч удалённых файлов в `Storage\File`, и вообще любой исходящий HTTP-трафик фреймворка. Класс зависит от:

- `Main` — служебные хелперы (dot-notation доступ к массивам, хэши, склейка путей);
- `Rule` — валидация query-параметров и тела запроса;
- `Cache\CacheManager` — HTTP-кеш с ETag/Last-Modified ревалидацией;
- `Storage\Resource` — ленивое декодирование тела ответа (и кодирование JSON-тела запроса);
- трейты `Traits\HasEvents` и `Traits\HasConfig` — событийная модель и конфигурация экземпляра.

## Быстрый пример

```php
$client = WebClient::create('{url}', ['batch' => 10])
    ->query(['url' => ['a.com', 'b.com']]);

$results = $client->send(); // дождаться всех результатов сразу

// либо потоково, по мере готовности:
while ($r = $client->next()) {
    // $r — одна запись результата
}
```

## Шаблонизация и батчинг

URL, переданный в конструктор (или `create()`), может содержать плейсхолдеры `{name}`. Если плейсхолдеры есть, класс не проверяет и не достраивает схему URL сразу — это откладывается до момента, когда конкретная комбинация параметров подставлена в шаблон (`substitute()` + `applyScheme()`). Если плейсхолдеров нет, схема URL проверяется сразу в конструкторе.

`query(array $params)` принимает параметры шаблона (и, если задана `schema()` с маркерами, параметры тела — см. ниже). Каждое значение приводится к списку (`array_values((array)$v)`), поэтому скаляр становится списком из одного элемента, а массив — списком вариантов. Итоговый набор параметров разворачивается в декартово произведение всех списков (метод `combos()`, реализованный как ленивый «одометр» — комбинации генерируются по одной, без материализации всего множества в памяти).

Полученные комбинации отправляются не все разом, а окнами — по `config('batch')` штук одновременно через `curl_multi`, с паузой `config('delay')` миллисекунд между окнами (пауза не делается перед самым первым окном). Размер окна и задержка настраиваются через конфиг клиента (или группы — см. ниже).

Результаты можно получать двумя способами, которые используют один и тот же внутренний генератор-конвейер (`$this->pipeline`):

- `next(): ?array` — отдаёт один результат за вызов, по мере готовности; возвращает `null`, когда все данные исчерпаны;
- `send(): array` — просто дренирует `next()` в массив до исчерпания.

## Однонаправленная очередь и повторные вызовы

Как только результат конкретной комбинации выдан потребителю, она помечается как `processed` (внутренняя карта `hash(комбинация) => true`) и повторно уже не отправляется. После полного прохода `send()` возвращает `[]`, а `next()` — `null`.

Чтобы полностью перезапустить пачку, нужно вызвать `reset()` — он очищает карту `processed` и состояние конвейера. Повторные вызовы `query()`, `schema()` или `fill()` тоже неявно вызывают `reset()`, так как они меняют состав или содержимое запросов.

Важный нюанс отказоустойчивости: если исключение брошено посреди обработки батча, недоставленные комбинации **не** помечаются как `processed` — при следующем вызове `send()`/`next()` будут доставлены только оставшиеся, необработанные комбинации. Технически это реализовано через `resetPipeline()` (сброс только генератора и флага `drained`, без очистки `processed`), который вызывается в `catch`-блоке `next()` при перехвате исключения из генератора.

## Тело запроса

`schema(array $schema)` задаёт правило валидации тела через `Rule::object($clean)->throwable()`. `fill(array $params)` заполняет само тело — оно одно на весь батч комбинаций (в отличие от query-параметров, которые размножают запросы).

Если среди значений тела встречаются `\CURLFile`-объекты или строки вида `'@/path/to/file'` (проверяется рекурсивно, включая вложенные массивы, метод `detectFiles()`), кодирование запроса автоматически переключается на `multipart/form-data`. Строки `'@/path'` конвертируются в реальные `\CURLFile` по месту. Так как `CURLOPT_POSTFIELDS` в виде массива не поддерживает нативную вложенность, перед отправкой вложенные массивы схлопываются в bracket-нотацию (`key[sub][sub2]`) методом `flattenMultipart()`.

Отправка файлов методом `GET`/`HEAD` невозможна — это проверяется в `validateReady()` и приводит к `LogicException`.

## Параметризация тела через `param()` и декартово произведение

`WebClient::param('name')` возвращает маркер-объект (`stdClass` с полями `__wc = 'param'`, `name`), который распознаётся внутри `schema()`. Маркер можно указать как значение поля схемы напрямую, либо в массиве вместе с правилами-модификаторами:

```php
$client->schema([
    'p2' => [WebClient::param('p2'), 'string|email'],
]);
```

Такое поле становится участником декартова произведения наравне с URL-плейсхолдерами: значения для него передаются через `query(['p2' => [...]])`, валидируются правилами-модификаторами один раз в момент `query()` (а не на каждую комбинацию отдельно), а затем подставляются в тело per-request через `Main::dotSet()` при построении конкретной спецификации запроса (`buildSpec()`).

Важное ограничение: `schema()` **обязана** быть вызвана до `query()`, если используются маркеры параметров — иначе `query()` не будет знать о допустимых именах параметров и бросит `InvalidArgumentException` с перечислением неизвестных имён. Одно и то же имя маркера нельзя привязать более чем к одному полю схемы, и в одной спецификации поля не может быть больше одного маркера — оба случая приводят к `InvalidArgumentException` уже на этапе `schema()`.

## `group()` — общий конвейер нескольких клиентов

```php
WebClient::group(function () {
    WebClient::create('https://a.example/1')->query([...]);
    WebClient::create('https://b.example/2')->query([...]);
}, ['batch' => 20, 'delay' => 50]);
```

`WebClient::group(callable $fn, array $config = [])` собирает все клиенты, созданные внутри переданного замыкания, в общий пайплайн (общий генератор заданий и общий `curl_multi`-цикл). Задания идут в порядке создания клиентов; при этом окна батча могут интерливиться — то есть на границах окон в одном и том же окне могут оказаться запросы разных клиентов-участников.

Вложенные вызовы `group()` схлопываются в одну плоскую группу: если группа уже активна (`self::$groupStack` не пуст), новый вложенный `group()` переиспользует самый внешний групповой инстанс, а не создаёт новый. Возвращается (в том числе из вложенных вызовов) именно этот внешний групповой экземпляр.

Настройки `batch`/`delay` для самого цикла отправки берутся из конфига группы; все прочие per-request настройки (таймауты, заголовки, `verify`, обработчики событий) — от клиента-владельца конкретного задания, а не от группы.

Каскадирование конфига при создании клиента внутри активной группы: явный конфиг, переданный в конструктор клиента, побеждает конфиг самой внутренней группы, тот — конфиг следующей по вложенности группы, и так далее до самой внешней; на пустых местах конфиг достраивается дефолтами через `@`-механизм `applyConfig`.

Группа неизменяема как единый запрос: вызовы `query()`, `schema()`, `fill()` непосредственно на групповом объекте бросают `LogicException` (`guardGroup()`). События регистрируются на клиентах-участниках по отдельности, а не на группе. А вот `next()`, `send()`, `reset()`, вызванные на группе, действуют сразу на все клиенты-участники (перебор `$this->members`).

## Форма результата

Каждый результат — ассоциативный массив со следующими ключами:

```
status, headers, body, data, type, url, effective_url,
errno, error, aborted, cached, config, info
```

- `data` — объект `Storage\Resource`, оборачивающий тело ответа для ленивого декодирования (`->get()`, `->extract($schema)`, `->getDom()`, `->toArray()`); для пустого тела (например, HEAD-запрос или прерванная передача) `data = null`;
- `type` — MIME-тип, резолвнутый для этого ресурса (явный `response_type` из конфига либо распарсенный `Content-Type` ответа);
- `config` — эффективный конфиг клиента-**владельца** этого конкретного задания (полезно, когда несколько клиентов с разными настройками обрабатываются одним общим обработчиком событий группы);
- `headers` — ассоциативный массив, ключи приведены к нижнему регистру;
- `aborted` — признак того, что передача тела была прервана до завершения (например, при 304-ревалидации);
- `cached` — признак того, что результат отдан из HTTP-кеша без реального сетевого запроса (или с обрывом тела при успешной ревалидации).

## Политика `verify` и схема URL

Настройка `verify` (по умолчанию `false`) отвечает сразу за две вещи: включение SSL-верификации в curl и инференс схемы для URL без явного протокола.

- URL без схемы получает `https://` в качестве префикса, если `verify=true`, иначе `http://`;
- явный `http://` в сочетании с `verify=true` бросает `InvalidArgumentException` (`applyScheme()`) — считается несовместимой комбинацией;
- явный `https://` при `verify=false` — валиден, но собственно проверки сертификата (`CURLOPT_SSL_VERIFYPEER`/`CURLOPT_SSL_VERIFYHOST`) при этом просто не включаются.

## Оборачивание тела ответа

Тело ответа не декодируется заранее — оно оборачивается в in-memory `Storage\Resource` (без обращения к диску или сети), основываясь на распознанном MIME-типе (`Content-Type` ответа или явный `response_type` из конфига). Реальное декодирование (JSON в массив, HTML/XML в DOM или текст и т.д.) происходит только по требованию, при обращении к методам `Resource`.

## События

Через трейт `HasEvents` доступны следующие зарезервированные события, все — per-request (то есть привязаны к конкретному запросу, а не к батчу в целом):

- **`prepare(&$spec)`** — срабатывает до вычисления ключа HTTP-кеша и до настройки curl-хендла. Мутации `$spec` внутри обработчика **влияют** на ключ кеша, так как вычисление ключа происходит после этого события.
- **`prepare_response($spec, &$result)`** — срабатывает после получения сырого ответа, но до резолва MIME и декодирования тела.
- **`decode_response($spec, &$result)`** — если ни один слушатель не подписан, либо ни один из них не заполнил `$result['data']`, тело декодируется автоматически (фоллбэк на `decodeBody()`) — `data` становится `Resource` над телом согласно `type`.
- **`error($spec, &$result)`** — срабатывает, если `errno !== 0` или `status >= 400`. Если ни один слушатель не обработал событие (не вернул `false` из `fire`) и `config('exception') === true`, бросается `RuntimeException`. Слушатель может поставить запрос обратно в очередь на повтор, установив `$result['requeue'] = true`; это ограничено настройкой `config('requeue')`: `0` — повторы запрещены, значение `< 0` — без лимита, значение `> 0` — максимум повторов на один запрос. Повторные попытки добавляются в очередь `$retry` и обрабатываются тем же конвейером **после** исчерпания основного генератора заданий — этот механизм одинаково работает и при потреблении через цикл `while ($r = $client->next())`, и через `send()`.
- **`response($spec, &$result)`** — всегда срабатывает последним для успешно завершённого запроса, и это **единственное** событие, которое срабатывает на попадании в кеш (cache-hit) — остальные события (`prepare_response`, `decode_response`, `error`) на кеш-хитах не вызываются.

## HTTP-кеширование с ревалидацией

Кеширование (через `Cache\CacheManager`) применяется только к методам `GET`/`HEAD` и по умолчанию выключено (`config('cache.use') === false`).

Ключ кеша строится из «чистого» URL (без query-строки) плюс всех параметров запроса — как вытащенных из query string, так и из заполненного тела (`fill()`), объединённых в `$spec['params']`.

Поведение при чтении:

- **Свежий кеш** (мета-поле `fresh_until` в будущем либо равно `-1`) отдаётся немедленно, без сетевого запроса — такое задание даже не занимает слот текущего батча;
- **Устаревший, но существующий кеш** ревалидируется: в запрос добавляются условные заголовки `If-None-Match` (по сохранённому ETag) и/или `If-Modified-Since` (по сохранённому Last-Modified) через `applyRevalidation()`. Если сервер отвечает `304 Not Modified`, либо возвращает `200` с теми же значениями валидаторов, передача тела прерывается на уровне header-колбэка curl (`return 0 != $len`, что сигнализирует curl остановить скачивание) — это дешёвый short-circuit, избегающий скачивания ненужного тела. В этом случае потребителю отдаётся сохранённая копия тела, а срок свежести кеша продлевается (`touchCache()`) до `max(config('cache.ttl'), max-age/Expires сервера)` (метод `resolveTtl()`).

Технически кешируется не полный результат, а «облегчённая» копия — без несериализуемых ключей `data` (может быть, например, `DOMDocument`) и `config`. При чтении из кеша тело реконструируется обратно в `data` через `decodeBody()`. Кеш-запись хранится с TTL `-1` (то есть физически не протухает в самом драйвере кеша) — свежесть проверяется исключительно по собственному мета-полю `fresh_until`, что и позволяет читать «протухший» с точки зрения `fresh_until` блоб для ревалидации/обработки 304, ведь нижележащее хранилище всегда его хранит.

Таким образом, есть три принципиально разных способа получить результат без полной загрузки тела с сети: (1) cache-hit по свежему кешу — сеть вообще не используется; (2) 304-ревалидация — короткий сетевой round-trip только за заголовками, тело не скачивается; (3) полный сетевой запрос с последующей записью в кеш.

## Requeue (повторная постановка в очередь)

Механизм повторных попыток полностью управляется слушателями события `error`, которые выставляют `$result['requeue'] = true`. Лимит повторов задаётся настройкой `config('requeue')`:

- `0` — повторы запрещены (значение по умолчанию);
- значение `< 0` — повторы без ограничения по количеству;
- значение `> 0` — не более указанного числа повторов на один запрос.

Драйверы вроде `IntegrationDriver`, которые хотят сами управлять обработкой ошибок, как правило явно отключают `config('exception') = false`, чтобы `WebClient` не бросал исключение самостоятельно, и вместо этого решают через `error`-слушатель, нужно ли переотправить запрос.

## Ключевые методы (сводка)

- `__construct(string $url, array $config = [])` — если сейчас строится группа (внутренний флаг `self::$constructingGroup`), создаёт «голый» групповой инстанс без парсинга URL и сразу возвращается. Иначе каскадирует конфиги активных групп под явный конфиг клиента, сохраняет конфиг, обрезает URL в шаблон, извлекает и дедуплицирует плейсхолдеры `{name}`, либо (если плейсхолдеров нет) сразу проверяет схему URL. Пересобирает правило валидации query. Если создание происходит внутри активной группы — регистрирует себя как участника.
- `static create(string $url, array $config = []): self` — фабрика, тонкая обёртка над конструктором.
- `static group(callable $fn, array $config = []): self` — см. раздел про группы выше.
- `static param(string $name): object` — маркер параметризуемого поля тела для `schema()`.
- `query(array $params): self` — задаёт значения плейсхолдеров/маркерных параметров тела; бросает `LogicException`, если URL не шаблонизирован и нет маркеров тела; бросает `InvalidArgumentException` при неизвестных именах параметров; валидирует через внутреннее правило query, если оно задано; вызывает `reset()`.
- `schema(array $schema): self` — задаёт валидацию тела и извлекает маркеры `param()` из полей; сбрасывает состояние тела/multipart и батча.
- `fill(array $params): self` — заполняет тело запроса (одно на весь батч), валидирует по `bodyRule`, детектит файлы для multipart; сбрасывает батч.
- `next(): ?array` — отдаёт один результат общего генератора; при перехвате исключения из генератора делает частичный `resetPipeline()` (без очистки `processed`).
- `send(): array` — дренирует `next()` в массив.
- `reset(): self` — полный сброс `processed` и состояния конвейера; для группы каскадируется на всех участников.
- `run()` / `validateReady()` (приватные) — подготовка и запуск конвейера; проверяют, что все обязательные параметры заполнены и что multipart-тело не отправляется методом `GET`/`HEAD`.
- `applyScheme()`, `splitUrl()`, `substitute()`, `combos()`, `jobs()`, `groupJobs()` (приватные) — URL-слой и генерация заданий: определение схемы, отделение query-строки, подстановка плейсхолдеров, ленивый одометр декартова произведения, потоки заданий для одиночного клиента и для группы.
- `buildSpec(array $combo)` (приватный) — строит конкретную спецификацию запроса для одной комбинации параметров: подставляет плейсхолдеры, мёржит body-параметры, пересчитывает multipart, формирует итоговые `get`/`body`/`params` (последнее — материал для ключа кеша).
- `dispatch(\Generator $jobs): \Generator` (приватный) — сердце конвейера: разбивает задания на окна по `batch`, выдерживает `delay` между окнами, обслуживает cache-hits без занятия слота батча, для остальных открывает curl-хендлы через `configureHandle()`, дренирует их через `curl_multi_exec`/`curl_multi_info_read`, передаёт результаты через `finalize()` (который решает — выдать результат или отправить на повтор), в `finally` гарантированно освобождает все оставшиеся хендлы.
- `configureHandle($ch, $spec, $state)` (приватный) — настраивает опции curl (таймауты, SSL, редиректы, метод-специфичные опции, заголовки), а также колбэк `CURLOPT_HEADERFUNCTION`, который копит заголовки в общий по ссылке объект `$state` (сознательно `stdClass`, а не массив — чтобы мутации переживали асинхронный жизненный цикл `curl_multi`) и умеет обрывать передачу тела при пустой строке заголовков, если задан обработчик `on_headers` (устанавливается `applyRevalidation()`).
- `buildResult()` (приватный статический) — собирает сырой результат curl-запроса; на прерванных передачах `errno` форсируется в `0` (обрыв тела не считается настоящей ошибкой), `body` — `null` для прерванных передач и для `HEAD`.
- `encodeBody(array $spec)` (приватный статический) — кодирует тело запроса: multipart-массив (сам curl строит тело), либо, если задан заголовок `Content-Type: application/json`, оборачивает тело в `Resource` для JSON-кодирования, иначе — `http_build_query()`.
- `detectFiles(array &$params): bool` (приватный статический, рекурсивный) — конвертирует строки `'@/path'` в `\CURLFile` по месту, детектит уже готовые `\CURLFile` (в том числе во вложенных массивах).
- `flattenMultipart()` (приватный статический) — схлопывает вложенные массивы в bracket-нотацию для multipart-отправки.
- `finalize(array $spec, array &$result, ?CacheManager $cache, int $attempts = 0): bool` (приватный) — постобработка после реального сетевого round-trip (или прерванной ревалидации): при прерванной передаче или 304 с валидным кешем отдаёт сохранённый hit, продлевая свежесть; иначе резолвит MIME, декодирует тело, обрабатывает ошибки (событие `error`, исключение при `config('exception')`, requeue-логика), всегда в конце вызывает `response`; для успешных некешированных ранее 2xx-ответов с активным кеш-хендлом персистит «облегчённую» копию с TTL `-1` и метаданными `etag`/`last-modified`/`fresh_until`.
- `resolveMime()`, `decodeBody()` (приватные) — резолв MIME (приоритет у явного `response_type`) и оборачивание непустого тела в `Storage\Resource`.
- Кеш-хелперы `cacheFor()`, `isFresh()`, `cacheRead()`, `applyRevalidation()`, `touchCache()`, `resolveTtl()` (приватные) — построение кеш-хендла на GET/HEAD, проверка свежести по собственному полю `fresh_until`, чтение и декодирование закешированного результата, установка условных заголовков и колбэка обрыва передачи, продление свежести при подтверждённом кеше, вычисление эффективного TTL с учётом серверных заголовков.

## Основные "режимы" использования

- **Стриминг vs bulk**: `next()` и `send()` потребляют один и тот же внутренний генератор-конвейер — можно выбрать построчную обработку по мере готовности либо получить все результаты сразу.
- **Одиночный клиент vs группа**: группа неизменяема как единица запроса (`query()`/`schema()`/`fill()` на ней запрещены), но объединяет несколько клиентов-участников в общий curl_multi-цикл с общими `batch`/`delay`; каждый участник сохраняет собственный конфиг и собственные обработчики событий.
- **Декартово произведение параметров**: URL-плейсхолдеры и параметризованные (`param()`) поля тела вместе формируют одно пространство комбинаций, разворачиваемое через `query()`.
- **Автоопределение multipart**: происходит автоматически при обнаружении `\CURLFile` или строк `'@/path'` среди значений тела, как при заполнении через `fill()`, так и при подстановке параметризованных полей на уровне отдельной комбинации.
- **Три способа получить результат без полной загрузки с сети**: cache-hit по свежему кешу, 304-ревалидация с обрывом тела, обычный полный сетевой запрос.
- **`config('exception')`**: `true` (по умолчанию) — бросать исключение на необработанные HTTP/curl-ошибки; интеграционные драйверы (например, `IntegrationDriver`) как правило явно ставят `false`, чтобы обрабатывать ошибки через собственный `error`-слушатель, в том числе используя механизм `requeue`.
