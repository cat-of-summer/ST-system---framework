# Access.php
<!-- DOCGEN:END -->

`final class Access` — модуль контроля доступа фреймворка ST_system. Это "входная дверь" перед бизнес-логикой приложения: он умеет как простую credential-проверку (пароль/токен в GET/POST/заголовке/куке/сессии), так и полноценный стейтфул-фаервол со скользящим окном рейт-лимита, баном, верификацией ботов через reverse-DNS, IP allow/deny-правилами, гео-IP фильтрацией и CORS.

Класс использует три служебных трейта:

- `HasInstance` — синглтон (`getInstance()`), нужен потому что у фаервола есть состояние на процесс (кеш-клиент, память между вызовами `handleIp()`/`banIp()`/`unbanIp()`).
- `HasConfig` — конфигурация через `static::config($key)` и валидация входных параметров конкретного вызова через `static::applyConfig($config, $schema)` (под капотом — `Rule`, значения вида `'@credentials.name'` означают "если поле не передано явно, взять дефолт из конфигурации `credentials.name`").
- `HasStaticEvents` — статические слушатели на класс: `Access::on()`/`fire()` идут прямо через трейт, без обёртки над экземпляром синглтона. Подписка на события (`throw`, `screen`, `deny`, `ban`, …) — через понятный статический API.

Внешние зависимости: `HTTP\Request` (чтение credential'ов, заголовков, IP), `HTTP\Response` (отправка HTTP-статусов и заголовков), `Cache\CacheManager` (хранение состояния фаервола и кеша верификации ботов), `Rule` (валидация конфигурации вызовов), `API\Drivers\Geo\GeoDriver` и его наследники `IpInfo`/`SxGeo`/`GeoIP2` (гео-IP драйверы, резолвятся по короткому имени из реестра `handleGeo.drivers.available`).

## Конфигурация по умолчанию

- `credentials.name = 'pass'`, `credentials.value = date('dm')` — имя параметра доступа и дефолтный "пароль". По умолчанию пароль — это **сегодняшние день и месяц** (например, `1407` для 14 июля). Это очень слабый секрет, рассчитанный только на "временную заглушку от чужих глаз", а не на защиту от целенаправленной атаки — в проде его обязательно нужно переопределять.
- `accessMethod = 'GET'` — откуда по умолчанию читается credential.
- `CORS.methods` / `CORS.headers` — списки по умолчанию для `handleCORS()`.
- `salt = ''` — соль для HMAC+XOR "обфускации" состояния фаервола и кеша ботов. Пустая строка (дефолт) означает, что шифрование **полностью отключено** и состояние хранится как обычный plain JSON.
- `firewall.driver = 'filesystem'`, `firewall.limits = [[5,1,10],[10,2,30],[60,60],[600,3600]]` — тройки `[максимум запросов, окно в секундах, бан в секундах (опционально)]`, описывающие несколько одновременно действующих скользящих окон рейт-лимита.
- `firewall.exclude = []` — allowlist IP/CIDR/префиксов, полностью обходящих фаервол.
- `firewall.proxies = []` — карта `CIDR/IP доверенного прокси => имя серверного заголовка`, из которого можно доверенно брать реальный IP клиента.
- `firewall.verifyBots = true` — включает reverse-DNS верификацию поисковых ботов перед применением рейт-лимита.
- `firewall.screen = []` — набор простых эвристик отсева "подозрительных" запросов.
- `firewall.rules = []` — явные allow/deny правила по IP, стране, языку, User-Agent, referer, PTR.
- `bots` — карта "подстрока User-Agent" => "допустимые суффиксы PTR-записи" для Googlebot, Яндекса, Mail.RU и Bing.
- `handleGeo.drivers.default = 'ipinfo'`, `handleGeo.drivers.available` — реестр коротких имён гео-драйверов на полные имена классов.

## Гейты доступа: четыре способа проверить один и тот же credential

Все нижеперечисленные методы читают значение через приватный `extractCredential(string $name, string $method)`, который диспетчеризует по `accessMethod`: `GET` → `Request::get()`, `POST/PUT/DELETE/PATCH` → `Request::post()`, `HEADERS` → `Request::headers()`, `COOKIE` → `Request::cookie()`, `SESSION` → `$_SESSION`. Любой другой метод — исключение `\InvalidArgumentException`. Общая логика сравнения — **нестрогая** (`!=` / `==`), а не `!==`/`===`. Это осознанная деталь поведения: например, credential `"123"` и число `123` будут считаться равными, а пустая строка и `null` в PHP тоже дают нестрогое равенство с некоторыми "пустыми" значениями — при интеграции стоит иметь это в виду.

Несмотря на общий фундамент, у каждого метода — своя семантика успеха/неудачи:

- **`requestAccess(array $config = [])`** — самый настраиваемый гейт. Валидирует `name`/`value`/`accessMethod` (дефолты из `credentials.*`/`accessMethod`), `onFail` (по умолчанию — `throw(401)`) и опциональный `onSuccess`. Если извлечённое значение пусто ИЛИ не совпадает нестрого с ожидаемым — вызывается `onFail`, иначе `onSuccess` (если задан). Типичное применение — "закрыть" целый скрипт/эндпоинт паролем в query-параметре.
- **`call(callable $f, array $config = [])`** — "выполни callback, если пароль верный, иначе тихо ничего не делай". В отличие от `requestAccess`, здесь нет `onFail`-хука вообще — при неудаче метод просто не делает ничего и ничего не возвращает.
- **`startBlock(array $config = []) / endBlock()`** — гейт для куска HTML/вывода внутри шаблона: `startBlock()` проверяет, что `name`/`value`/`accessMethod` заданы, запоминает их в статическое поле `$block` и запускает `ob_start()`. `endBlock()` забирает буфер (`ob_get_clean()`) и **выводит его только если проверка credential прошла** — иначе весь накопленный вывод молча отбрасывается. Удобно оборачивать закрытые куски страницы (`Access::startBlock(); echo ...; Access::endBlock();`) без явного if/else в шаблоне.
- **`httpAccess(array $config = [])`** — классический HTTP Basic Auth поверх `PHP_AUTH_USER`/`PHP_AUTH_PW` (сравнение здесь уже строгое, `!==`). При несовпадении логина или пароля (или отсутствии этих серверных переменных) отправляется `401` с заголовком `WWW-Authenticate: Basic realm="Restricted Area"` — это "чистый" гейт: явной success-ветки нет, просто выполнение продолжается дальше, если проверка прошла.

Общий примитив отказа для всех гейтов и фаервола — **`throw(int $code): void`**: сначала пытается сообщить о событии `throw` через `fire()`; если ни один слушатель не обработал его (`HasStaticEvents::fire()` вернул `false` — дефолт "необработано"), отправляет HTTP-статус через `Response` с заголовком `X-Content-Type-Options: nosniff` и завершает вывод. Благодаря этому поведение "деним" можно централизованно перехватить через `Access::on('throw', ...)`, не трогая код каждого гейта.

## Определение IP и Origin клиента

- **`getClientIp(): string`** — мемоизировано на статическую переменную (считается один раз за запрос). Если `firewall.proxies` не пуст, метод ищет совпадение `REMOTE_ADDR` с одним из настроенных CIDR/IP доверенных прокси и **только тогда** доверяет соответствующему серверному заголовку (например, `X-Forwarded-For`), беря первое значение из списка через запятую. Это защита от IP-спуфинга: заголовкам верят только если запрос реально пришёл от известного прокси. Если `firewall.proxies` пуст (дефолт) — поведение менее строгое: метод наивно доверяет `X-Forwarded-For`, а если его нет — `Client-Ip`, и только в последнюю очередь падает на `REMOTE_ADDR`. Результат нормализуется через `normalizeIp()`, который для валидных IPv6-адресов приводит их к каноничному hex-виду (через `inet_pton`+`bin2hex`), чтобы разные текстовые записи одного и того же адреса не считались разными IP.
- **`getRequestOrigin(): string`** — тоже мемоизировано, берёт заголовок `Origin`, а если его нет — извлекает хост из `Referer`.

Приватные хелперы `ipInCidr()`/`ipMatches()` реализуют матчинг IP по CIDR (побитовое сравнение через `inet_pton`, поддержка и IPv4, и IPv6) и более широкий матчинг `ipMatches()`, который вдобавок понимает префиксную запись с точкой на конце (например, `192.168.` — совпадение по префиксу строки) и точное совпадение нормализованных IP.

## Шифрование состояния (`salt`, `seal`/`unseal`)

Приватные методы `salt()`, `xorStream()`, `seal()`, `unseal()` реализуют самодельную симметричную "обфускацию" JSON-блобов, которые хранятся в кеше (состояние фаервола по IP и кеш верификации ботов):

- `xorStream()` — XOR данных 32-байтными чанками против потока `sha256(salt . pack('N', номер_чанка))` — то есть побайтовый keystream в духе CTR-режима, но без индустриального шифра.
- `seal()` сериализует состояние в JSON и, если `salt` не пуст, шифрует его через `xorStream()` и добавляет спереди 8-байтный HMAC-SHA256 tag (защита от подмены содержимого кеша). Если `salt` пуст — **шифрование полностью пропускается**, и в кеш пишется голый JSON.
- `unseal()` — обратная операция; при пустом `salt` просто парсит JSON. При заданном `salt` сверяет HMAC-tag через `hash_equals()` (защита от timing-атак) и возвращает `[]`, если tag не совпал или блоб короче 8 байт.

Важный практический нюанс: **пока `firewall.salt` не задан явно, состояние бана по IP и результаты верификации ботов хранятся в кеше в открытом виде**. Это не критично для конфиденциальности (данные не секретные — счётчики запросов и флаг "бот да/нет"), но защита от подмены (HMAC) и обфускация появляются только при ненулевой соли.

## Фаервол: `handleIp()`

`handleIp(): void` — главная точка входа фаервола, вызывается рано в жизненном цикле запроса (до основной маршрутизации). Вызов **мемоизирован на процесс** через `static $handled`: сработает только один раз за время жизни процесса PHP, даже если гипотетически IP клиента "поменялся" — повторные вызовы в рамках того же запроса/процесса становятся no-op.

Порядок проверок (каждая следующая проверяется только если предыдущие не сработали):

1. **Валидация IP.** Если `getClientIp()` вернул пустую строку или `filter_var(..., FILTER_VALIDATE_IP)` не проходит — фаервол молча ничего не делает (нет смысла применять лимиты к нераспознанному IP).
2. **`firewall.exclude` — allowlist.** Если IP совпадает (через `ipMatches()`) с любой записью списка исключений — полный обход всего дальнейшего фаервола.
3. **Верификация ботов.** Если `firewall.verifyBots` включён (по умолчанию — да) и `verifyBot()` подтвердил, что это легитимный поисковый бот — тоже полный обход. Верифицированные краулеры никогда не попадают под рейт-лимит.
4. **Эвристики `screen`.** Проверяются в фиксированном приоритетном порядке: `noUserAgent` (пустой `User-Agent`) → `noReferer` (пустой Origin/Referer) → `noLang` (пустой или `*` `Accept-Language`) → `http2Only` (протокол сервера не `HTTP/2.0`). Первое сработавшее условие фиксирует причину, стреляет событие `screen` с причиной и IP, и вызывает `throw(403)`.
5. **`matchRules()` по `firewall.rules`.** Явный вердикт `allow` — полный обход дальнейших проверок (включая рейт-лимит). Явный вердикт `deny` — событие `deny` и `throw(403)`. Если правил нет или ни одно не совпало — идём дальше, к рейт-лимиту.
6. **Скользящее окно рейт-лимита + бан.** Состояние по IP читается из кеша по ключу `hash('sha256', salt . '|' . ip)` (первые 32 hex-символа). Если IP уже забанен (`state['ban'] > now`) — сразу `throw(429)`. Иначе для каждой тройки из `firewall.limits` вычисляется "тайм-слот" (`intdiv(now, window)`); при смене слота счётчик обнуляется, иначе инкрементируется. Если счётчик превысил `max` для хотя бы одного лимита — фиксируется нарушение, а срок бана берётся как **максимальный** запрошенный (`limit[2]`, если задан, иначе `firewall.ttl`) среди всех нарушенных одновременно лимитов. TTL самой кеш-записи — максимум всех `window` (чтобы запись пережила свои окна наблюдения) и дополнительно расширяется до срока бана, если он больше. При нарушении сохраняется `state['ban']`, стреляет событие `ban`, и вызывается `throw(429)`.

Такая многослойная модель (`exclude → bot-bypass → screening → allow/deny rules → rate limit+ban`) позволяет гибко комбинировать "белые списки", защиту от простых ботов-скребков и настоящий адаптивный рейт-лимит с несколькими окнами разной длины одновременно (например, короткая "вспышка" за 1 секунду и долгий лимит за час).

## Администрирование бана

- **`banIp(string $ip, ?int $ttl = null): void`** — ручной бан IP вне обычного потока рейт-лимита. Бросает `\InvalidArgumentException`, если IP не проходит `filter_var`. TTL по умолчанию берётся из `firewall.ttl`, но срок жизни самой кеш-записи всегда расширяется хотя бы до максимального `window` среди настроенных лимитов (чтобы запись не протухла раньше времени бана).
- **`unbanIp(string $ip): void`** — снимает только флаг `ban`, сохраняя накопленные счётчики окон (`state['w']`), если они не пусты. Если после снятия бана в состоянии не осталось ничего значимого (`state['w']` пуст) — запись в кеше полностью удаляется (`purge()`), иначе она пересохраняется с тем же TTL-подходом, что и в `banIp()`.
- **`unbanAll(): void`** — снимает разом все баны и счётчики всех IP, полностью очищая весь namespace фаервол-кеша через `CacheManager::purgeBase()`. Это "рубильник", а не точечная операция.

Все три метода стреляют соответствующие события (`ban`, `unban`, `unbanAll`), что удобно для логирования/алертинга администраторам.

## Верификация ботов: `verifyBot()`

`verifyBot(?array $signatures = null): bool` реализует forward-confirmed reverse DNS проверку — стандартный способ отличить настоящего поискового бота от подделки User-Agent:

1. Если User-Agent пуст или клиентский IP невалиден — сразу `false`.
2. User-Agent матчится по подстроке (`stripos`) против карты сигнатур (переданной явно или из конфигурации `bots`). Если совпадений нет — `false` (это "не бот из нашего списка", а не "плохой бот" — фаервол просто продолжит обычную проверку рейт-лимита).
3. Результат кешируется на **3600 секунд** по ключу, зависящему от соли, IP и User-Agent — то есть кеш не отличает "проверили и не бот" от "проверили и бот", кешируется сам факт проверки, что экономит DNS-запросы при повторных обращениях с того же IP+UA.
4. **Важный security-нюанс:** если список допустимых PTR-суффиксов для сработавшей сигнатуры содержит буквальную точку `"."` как один из элементов (wildcard), верификация **сразу считается пройденной без единого DNS-запроса**. Это осознанная "лазейка" в конфигурации — доверять всему, что просто содержит нужную подстроку в User-Agent, без PTR-подтверждения. Если такая запись случайно попадёт в конфиг `bots`, это фактически отключает защиту от подделки UA для этой сигнатуры.
5. Иначе выполняется полноценная forward-confirmed reverse DNS проверка: `gethostbyaddr($ip)` получает PTR-имя, оно проверяется на совпадение с одним из разрешённых суффиксов (`stripos`), и затем `dns_get_record()` делает **обратный** (forward) запрос по этому имени, чтобы убедиться, что PTR-имя действительно резолвится обратно в тот же IP (A/AAAA-запись совпадает с нормализованным IP клиента). Только при выполнении обоих условий — `verified = true`.
6. При успехе стреляет событие `verifiedBot($ip, $ua)`.

## Правила allow/deny: `matchRules()`

Приватный `matchRules(string $ip): ?string` обходит `firewall.rules` **дважды**: сначала все правила с действием `allow`, затем все с `deny`. Это значит, что **allow всегда побеждает deny независимо от порядка их описания в конфигурации** — если один и тот же IP одновременно подпадает под allow- и deny-правило, победит allow.

Каждое правило — тройка `[action, type, pattern]`, где `type` может быть:

- `ip` / `cidr` — матчинг через `ipMatches()` (точный IP, CIDR или префикс);
- `country` — сравнение с заголовком `HTTP_CF_IPCOUNTRY` (это специфично для Cloudflare — то есть приложение рассчитано на работу за Cloudflare-проксированием, а не на собственное геоопределение по IP);
- `lang` — первые две буквы заголовка `Accept-Language`;
- `ua` / `referer` / `ptr` — регистронезависимая проверка подстроки.

PTR резолвится лениво: `gethostbyaddr()` вызывается только если в правилах реально встретился тип `ptr` — это экономит DNS-запросы, когда PTR-правила не используются вовсе.

## Гео-IP: `handleGeo()`

`handleGeo(array $config = [])` резолвит гео-драйвер по короткому имени из реестра `handleGeo.drivers.available` (полное имя класса тоже допускается напрямую) и проверяет через `is_subclass_of()`, что он реализует `API\Drivers\Geo\GeoDriver` — иначе бросает `\InvalidArgumentException` с понятным сообщением на русском.

Далее вызывается `$driver::create($token)->getDetails($ip)`. Если драйвер бросил исключение (например, сетевая ошибка обращения к внешнему гео-API) — сработает `onError`, если он задан, иначе `onPassed($details, $th)` (с пустым `$details` и объектом исключения вторым аргументом), а если и его нет — просто возвращается пустой `$details`. То есть сбой гео-сервиса по умолчанию **не блокирует** запрос — деградация происходит "мягко".

При успешном получении данных:

1. Проверяется `black_list` — первое совпадающее поле (значение или один из списка допустимых значений) немедленно вызывает `onBlackList($details)` и возвращает результат — остальные проверки (white_list, onPassed) не выполняются.
2. Если чёрный список не сработал и задан непустой `white_list` — проверяется, что **все** поля из списка совпали (а не хотя бы одно); при полном совпадении — `onWhiteList($details)` (или сам `$details`, если хук не задан).
3. Если ни одно из вышеперечисленного не сработало — вызывается `onPassed($details)` (или возвращается `$details`).

Это гибкий примитив, который можно использовать и как строгий гейт (throw при попадании в чёрный список конкретной страны), и просто как обогащение запроса гео-данными без блокировки.

## CORS: `handleCORS()`

`handleCORS(array $config = [])` валидирует `allowed_origins` (по умолчанию `['*']`), `forbidden_origins`, `methods` (каждый — обязательно из списка `CORS.methods`) и `headers`.

Логика:

1. Если запрос имеет `Origin` и он есть в `forbidden_origins` — сразу `throw(403)`.
2. Вычисляется заголовок ответа `Access-Control-Allow-Origin`: `'*'`, если в `allowed_origins` есть wildcard; иначе — конкретный origin запроса, если он есть в списке разрешённых; иначе `null`.
3. Если `origin_header === null` (origin не разрешён) — `throw(403)`.
4. Иначе отправляются `Access-Control-Allow-Credentials: true`, `Access-Control-Allow-Origin` и `Access-Control-Allow-Headers`.
5. Для `OPTIONS`-preflight запроса дополнительно отправляется `Access-Control-Allow-Methods` и сразу же — ответ `200`.

**Нюанс, который стоит знать при эксплуатации:** заголовок `Access-Control-Allow-Credentials: true` отправляется **всегда**, в том числе вместе с `Access-Control-Allow-Origin: *`. По спецификации CORS браузеры реально отклоняют такую комбинацию (wildcard origin несовместим с credentials) — то есть при дефолтном `allowed_origins = ['*']` фактический эффект `Allow-Credentials` в браузере будет проигнорирован. Это существующее поведение кода, а не описание того, как "должно быть" — при необходимости настоящей поддержки credentialed-запросов нужно явно ограничивать `allowed_origins` конкретными доменами, а не полагаться на wildcard.

## Итоговая модель работы

`Access` сознательно разделяет две относительно независимые задачи:

- **Гейты доступа по паролю/токену** (`requestAccess`, `call`, `startBlock`/`endBlock`, `httpAccess`) — простой, "плоский" механизм для защиты отдельных действий или кусков вывода одним общим секретом, с разными вариантами сигнализации успеха/неудачи под разные сценарии использования (явные callback'и, тихий no-op, вывод/подавление буфера, HTTP Basic).
- **Фаервол уровня IP** (`handleIp`, `banIp`/`unbanIp`/`unbanAll`, `verifyBot`, `matchRules`, `handleGeo`, `handleCORS`) — многослойная, полностью конфигурируемая система защиты от злоупотреблений на уровне сети: allowlist, доверие ботам, эвристический отсев, явные правила и, наконец, адаптивный рейт-лимит с несколькими одновременными окнами и баном. Всё это состояние живёт в `CacheManager` и по желанию (через `salt`) может быть защищено от чтения/подмены посторонними, имеющими доступ к файлам кеша.

Оба слоя используют единую систему событий (`Access::on(...)`) для расширения поведения без модификации самого класса — `throw`, `screen`, `deny`, `ban`, `unban`, `unbanAll`, `verifiedBot`.
