# Rule.php
<!-- DOCGEN:END -->

`ST_system\Rule` — движок валидации, коэрсии (приведения типов) и трансформации данных. Это один из самых фундаментальных классов фреймворка: на нём построены валидация конфигов (`Traits\HasConfig::applyConfig`), параметры и тела запросов интеграций (`API\IntegrationDriver`), HTTP-запросы (`HTTP\Request::validate/check`), исходящие HTTP-клиенты (`HTTP\WebClient::schema()/query()`), проверка доступа (`Access`), манифесты ассетов (`Assets`) и десятки схем внешних API/CRM/ботов в `src\API\Drivers\*`.

Класс `final`, инстансы иммутабельны после регистрации как алиас (`->alias()`), но обычные (незарегистрированные) инстансы, которые вы создаёте через `Rule::create()`/`Rule::object()`/`Rule::forEach()` и т.п., свободно донастраиваются fluent-методами до тех пор, пока не заморожены.

## Быстрый старт

Самый частый паттерн во всём фреймворке — «провалидируй или брось исключение»:

```php
use ST_system\Rule;

$data = ['email' => 'user@example.com', 'age' => '17'];

Rule::object([
    'email' => 'required|string|email',
    'age'   => 'required|int|min:18',
])->throwable()->apply($data);
```

Если `age` меньше 18 — будет брошено `\Exception` с текстом ошибок, объединённых через перевод строки. Если правила проходят, `$data` мутируется на месте: `age` станет `(int) 17`, лишние ключи (не описанные в схеме) будут отброшены.

Без исключений, с получением списка ошибок:

```php
$errors = Rule::create('required|string|max:100')->apply($value);
if (!empty($errors)) {
    // $errors — array<string>
}
```

## Точки входа: create / object / forEach

### `Rule::create($spec): Rule`

Универсальный конструктор. Принимает:

- **`\Closure`** — оборачивается напрямую как callback правила (без парсинга):
  ```php
  Rule::create(fn(&$v) => is_string($v) && $v !== '');
  ```
  Замыкание получает значение по ссылке (`&$v`) и второй необязательный параметр — массив параметров DSL (`array $p`). Возврат:
  - `true`/`false` (или `0`) — прошло/не прошло (без явного сообщения об ошибке, если не назначен `handleError`);
  - `array` — список сообщений об ошибках (пустой массив = «прошло»).

- **`string`** — DSL-спека вида `'string|email|max:100'`, парсится через `parseString()` (см. ниже). Если в результате получилось ровно одно суб-правило — оно возвращается напрямую (без обёртки в composite), иначе строится составное правило, которое выполняет каждое суб-правило по порядку (`order`), **останавливаясь досрочно**, если правило провалилось и у него `skip=true`.

- **`array`** — то же самое, но через `compileFieldRules()`: каждый элемент массива сам может быть строкой DSL или инстансом `Rule` (или тем и другим вперемешку), всё разворачивается («флэттенится») в единый плоский список суб-правил и сортируется по `order`:
  ```php
  Rule::create([
      'required',
      'string',
      Rule::create(fn(&$v) => mb_strlen($v) <= 255)->handleError(fn($v) => 'Слишком длинно'),
  ]);
  ```

### `Rule::object(array $schema): Rule`

Строит правило, которое валидирует/коэрсит ассоциативный массив (или объект — автоматически кастуется в массив) по схеме «ключ => спека»:

```php
$schema = [
    'name'  => 'required|string|max:100',
    'email' => 'required|string|email',
    'age'   => ['nullable', 'int', 'min:0'],
    'roles' => Rule::forEach('string|in:admin,editor,viewer'),
];

$errors = Rule::object($schema)->apply($data);
```

Важное поведение: **схема авторитетна** — любые ключи `$data`, отсутствующие в схеме, не переживают `apply()` (просто не попадают в результирующий массив). Также ключ, значение которого в результате обработки осталось «отсутствующим» (например, `sometimes` не сработал и default не подставлен), **опускается** из результата вместо установки `null`.

Поддерживается **dot-notation** — ключи вида `'a.b.c' => spec` группируются в дерево и автоматически компилируются во вложенные `Rule::object()`/`Rule::forEach()` (подробнее в разделе ниже).

### `Rule::forEach($spec): Rule`

Валидирует/коэрсит каждый элемент итерируемого значения (`array` или `\Traversable`):

```php
'tags' => Rule::forEach('required|string|trim'),
```

Принимает `\Closure`, `Rule`, строку DSL или массив спек — как и `create()`. **Важно**: элементы, провалившие валидацию, **автоматически удаляются** из массива (`unset($data[$i])`) — весь массив в целом не проваливается, если явно не сделать `->throwable()` (тогда исключение остановит обработку раньше). Ошибки по проваленным элементам всё равно попадают в возвращаемый список ошибок с префиксом индекса: `"{$i}.{$err}"`.

```php
$data = ['tags' => ['ok', '', 'also-ok']];
$errors = Rule::object(['tags' => Rule::forEach('required|string')])->apply($data);
// $data['tags'] === ['ok', 'also-ok'] — пустая строка отфильтрована
// $errors === ['tags.1.This field is required']
```

## `Rule::filtered()` — правила-фильтры коллекций

```php
public static function filtered(callable $select): Rule
```

Фабрика правил, которые **отбирают** элементы значения и трактуют «ничего не осталось» как
отсутствие поля. Это точка расширения: на ней построены все правила загрузок, но она не знает
ни про загрузки, ни про какой-либо конкретный тип.

`$select($value, array $params)` должен вернуть кортеж `[отобранные[], была ли одиночка на входе]`.
Сентинел приходит в `$select` как `null`. Дальше `filtered()` сам решает:

- отобранные есть → значение заменяется на них (одиночный вход остаётся одиночным);
- отобранных нет → значение сворачивается в **сентинел**, и это **ошибка только в
  required-scope** (см. [filter-режим vs strict-режим](#filter-режим-vs-strict-режим-роль-required));
  иначе — тихий отсев без ошибки.

Возвращённое правило уже помечено `seesSentinel()` и `skip()`; `order()`, `handleError()`
и `alias()` — за вызывающим.

```php
// оставить только положительные числа; пусто → поле исчезает, а с `required` — ошибка
$positive = Rule::filtered(fn($v, array $p) => [array_values(array_filter((array)$v, fn($x) => $x > 0)), false])
    ->order(600)
    ->handleError(fn($v) => 'No positive values left');

Rule::object(['scores' => ['required', $positive]])->apply($data);
```

## apply() vs check()

- **`apply(&$data): array`** — выполняет правило против `$data` **по ссылке**. Коэрсия типов, дефолты, trim и т.п. реально мутируют переданную переменную. Возвращает список ошибок.
- **`check($data): array`** — то же самое, но **без побочных эффектов** («прошло бы или нет»); переменная вызывающего не меняется, т.к. значение передаётся по значению.

```php
$value = '42';
$errors = Rule::create('int')->apply($value); // $value === 42 (int)

$value = '42';
$errors = Rule::create('int')->check($value); // $value всё ещё '42' (string)
```

## DSL-строки правил

Строка вида `'name1:param1,param2|name2|name3:param'` парсится в `parseString()`:

1. Строка разбивается по `|` на сегменты, каждый `trim()`-ится.
2. Каждый сегмент — это `name` или `name:параметры`. Если есть `:`, всё **после первого** двоеточия — строка параметров, которая делится по `,` на `params` (массив строк).
3. `name` резолвится через реестр алиасов (`resolveAlias()` — с учётом текущего scope, см. ниже). Если алиас не найден — `RuntimeException("Unknown rule: '{$name}'")`.
4. Найденный алиас **клонируется** (`->copy()`), и если были параметры — они присваиваются клону (`$clone->params = $params`).
5. Итоговый список суб-правил сортируется по `order`.

Примеры:

```php
'required|string|email'
'nullable|int|min:0|max:120'
'string|trim|max:255'
'array|min:1|foreach:string'
'string|regex:^[A-Z]{2}\d{4}$'
'sometimes|string|in:draft,published,archived'
```

Ограничение DSL: параметры режутся по `,`, а имя/параметры сегмента — по **первому** `:`. Это значит, что вложенное правило с несколькими своими параметрами внутри `foreach:...` работает (например `'foreach:min:1,max:10'` корректно разворачивается в `min:1` и `max:10` для каждого элемента), но конструкции, где сам параметр содержит запятую (например произвольный regex с `,` внутри паттерна, или `foreach:in:1,2,3`), могут распарситься не так, как ожидается. Для таких случаев используйте не строку, а объектную форму: `Rule::regex('...')`, `Rule::in([...])`, `Rule::forEach(Rule::in([...]))`.

## Сентинел «значение отсутствует»

Внутри `Rule` есть единый маркер «поле реально отсутствует» — синглтон `stdClass`, доступный через `Rule::sentinel()` (приватный) и проверяемый через `Rule::isSentinel($value): bool` (публичный, статический). Это центральный механизм, отличающий «поле не передано совсем» от «поле передано как `null`».

По умолчанию правило **не видит** сырой сентинел: если текущее значение — сентинел, а правило не помечено `->seesSentinel()`, `execute()` подменяет его на `null` перед вызовом callback (маскировка), и переносит изменения обратно в `$data` только если правило реально что-то присвоило (не оставило `null`). Правила, которым важно различать «отсутствует» и «явный null» (почти все встроенные: `sometimes`, `required`, `nullable`, `default`, коэрсии типов и т.д.), вызывают `->seesSentinel()` и работают с сентинелом напрямую.

Практическое следствие: пайплайн `'sometimes|string|email'` — если поле отсутствует, `sometimes` «проваливается» (в терминах внутренней механики) но **без сообщения об ошибке** и с `skip=true`, поэтому `string`/`email` для этого поля просто не выполняются, а конечный результат — поле отсутствует в выходных данных, ошибок нет.

## Fluent-модификаторы

Все возвращают `$this` и защищены `guardFrozen()` — попытка вызвать их на замороженном (зарегистрированном через `->alias()`) правиле бросает `RuntimeException('Cannot modify a frozen Rule (aliased rules are immutable)')`.

- **`before(\Closure $fn): self`** — хук перед основным callback, выполняется всегда (`$fn($data, $params)`).
- **`after(\Closure $fn): self`** — хук после основного callback; выполняется **даже если** callback бросил исключение.
- **`handleError(\Closure $fn): self`** — трансформирует/генерирует сообщение(я) об ошибке при провале. `$fn($value, $errors)`; если вернула строку — она **заменяет собой весь массив ошибок** этого шага.
- **`throwable(): self`** — сахар над `handleError`: ставит замыкание, которое бросает `\Exception` с текстом ошибок, объединённых через `PHP_EOL`, вместо их возврата. Самый частый паттерн: `Rule::object($schema)->throwable()->apply($data)`.
- **`order(int $o): self`** — порядок выполнения внутри composite-правила (меньше = раньше). См. таблицу ниже.
- **`skip(bool $s = true): self`** — если это суб-правило провалилось, **остальные правила того же пайплайна пропускаются целиком**. На этом механизме держатся `sometimes`, `nullable`, `required`, `default`, `excludeIf`, `requiredIf`, `prohibitedIf` — короткое замыкание валидации, когда значение легитимно отсутствует/невалидно рано.
- **`seesSentinel(bool $s = true): self`** — правило получает сырой сентинел вместо авто-маскировки в `null` (см. раздел выше).

```php
Rule::create(fn(&$v) => $v > 0)
    ->handleError(fn($v) => "Значение {$v} должно быть положительным")
    ->order(700);
```

## Порядок выполнения (order)

Фиксированные значения `order` у встроенных правил (по возрастанию):

| order | Правила | Смысл |
|---|---|---|
| `-3` | `default`, `strip_tags` | подстановка дефолта / ранняя очистка от тегов — до всего остального |
| `-2` | `trim`, `ltrim`, `rtrim`, `html_encode`, `html_decode`, `uppercase`, `lowercase`, `when()` | трансформации строки перед проверками |
| `0` | `sometimes`, `excludeIf` | самое раннее короткое замыкание по признаку «отсутствует» |
| `100` | `required`/`requiredIf`, `nullable`, `present`, `prohibitedIf` | проверки обязательности |
| `500` | `string`, `int`/`integer`, `float`, `bool`, `callable`, `closure`, `email`, `url`, `array`, `foreach`, `or`/`anyOf`, `date`, `date_format`, `json`, `accepted`, `declined`, `files` | коэрсия типов и базовые форматы |
| `600` | **дефолтный `order` для новых `Rule`** (в т.ч. кастомных правил без явного `->order()`), `mime`/`mimes`, `extension`, `filesize` | |
| `700` | `min`, `max`, `count`, `in`, `notIn`, `regex`, `digits`, `hex_color`, `between`, `starts_with`, `ends_with`, `contains` | ограничения значения — выполняются ПОСЛЕ коэрсии типов |

Практический вывод: пишите `'int|min:0|max:100'`, а не `'min:0|max:100|int'` — хотя порядок в строке DSL не важен для итогового результата (сортировка по `order` происходит всегда), потому что `min`/`max` (700) в любом случае выполнятся после `int` (500).

## Комбинаторы: requiredIf, prohibitedIf, excludeIf, when, anyOf, in, notIn, default, regex

Все — статические фабрики, возвращающие настроенный `Rule`.

- **`Rule::requiredIf($cond): Rule`** — `$cond` — `bool` или `\Closure` (проверяется на каждый вызов). Если условие истинно, поле обязано быть непустым (не сентинел, не `null`, не `''`), иначе ошибка `'This field is required'`. `order(100)`, `skip()`, `seesSentinel()`. Алиас `'required'` — это `Rule::requiredIf(true)`.
- **`Rule::prohibitedIf($cond): Rule`** — обратное: если условие истинно, поле обязано быть пустым/отсутствовать, иначе `'This field is not allowed'`.
- **`Rule::excludeIf($cond): Rule`** — если условие истинно, заменяет значение на сентинел (эффективно удаляет ключ из результата `Rule::object()`), без сообщения об ошибке. `order(0)`, `skip()`.
- **`Rule::when($cond, $spec): Rule`** — условно применяет суб-правило `$spec` (строка DSL или `Rule`), только если `$cond()` истинно. `order(-2)`.
  ```php
  'discount' => Rule::when(fn() => $isVip, 'required|float|min:0'),
  ```
- **`Rule::anyOf(...$specs): Rule`** (алиасы DSL: `or`, `anyOf`, `any_of`) — проходит, если **любая** из спек валидна; сохраняет коэрснутое значение той спеки, на которой сработал успех (пробует по порядку, первый успех побеждает):
  ```php
  Rule::anyOf('int', 'string|in:auto')
  // DSL: 'or:int,string' (без параметров у вложенных правил) или через массив-объект Rule::anyOf(...)
  ```
- **`Rule::in(array $values): Rule`** / **`Rule::notIn(array $values): Rule`** — то же самое, что DSL `in:a,b,c` / `notIn:a,b,c`, но принимает произвольный PHP-массив (без ограничений DSL на запятые внутри значений).
- **`Rule::default($value, bool $valid = false): Rule`** — подставляет `$value`, если текущее значение — сентинел/`null`/`''`. Подстановка **всегда** останавливает остаток пайплайна (`skip()`), сообщение об ошибке не формируется. Флаг `$valid = true` дополнительно делает результат «неуспешным» (в смысле останова пайплайна, не показа ошибки), если итоговое значение loose-равно (`==`) дефолту — даже если оно было передано явно, а не подставлено.
  ```php
  'timeout' => ['nullable|int', Rule::default(30)],
  ```
- **`Rule::regex(string $pattern): Rule`** — объектная форма (в отличие от DSL `regex:...`, не страдает от разбиения по запятым/двоеточиям внутри паттерна): `['string', preg_match($pattern, $v) === 1]`, `order(700)`, ошибка `'Invalid format'`.

## Реестр алиасов, scope() и alias()

Правила регистрируются в глобальном статическом реестре (`self::$registry`) под строковым именем через `->alias($name)`. `alias()` **замораживает** инстанс (`$this->frozen = true`) — дальше он никогда не мутируется; любое дальнейшее использование по имени (через DSL или `Rule::get()`) работает на **клоне** (`->copy()`), а не на оригинале.

```php
public function alias(string $name, $skip = 0): self
```

- Если `$name` не содержит `\`, к нему подставляется текущий scope-префикс (см. `Rule::scope()` ниже): `$name = $prefix . '\\' . $name`.
- Если `$name` содержит буквальный `\` — это «полностью квалифицированное» имя, scope игнорируется (используется, например, для алиасов, которые должны быть видны из любого scope).
- Если имя уже занято **другим** инстансом:
  - `$skip = 0` (по умолчанию) — бросает `RuntimeException("Rule alias '{$name}' already registered")`;
  - `$skip = 1` — молча оставляет существующую регистрацию как есть и возвращает `$this` **незарегистрированным** (не заморожен);
  - любое другое truthy значение — перезаписывает регистрацию новым инстансом.

**`Rule::scope(string $prefix, \Closure $fn)`** — пушит `$prefix` в статический стек на время выполнения `$fn()` и снимает его по завершении (в `finally`, так что стек корректно откатывается даже при исключении). Пока scope активен, `resolveAlias()` (используется и в `parseString()`, и в `Rule::get()`) сначала ищет алиас с текущим префиксом (`"{$prefix}\\{$name}"`), и только если не находит — фоллбэк на глобальное имя.

Это позволяет разным классам регистрировать правила с одинаковыми короткими именами, не конфликтуя друг с другом: например, `HasConfig::applyConfig()` регистрирует `\defaultConfig` (полностью квалифицированное, доступно из любого scope) и использует его через `Rule::scope(static::class, ...)`, так что при вызове `defaultConfig:...` внутри `Rule::currentPrefix()` виден класс-потребитель, и правило может подставлять значения именно из конфига этого класса. Аналогично `IntegrationDriver::__init()` и `HTTP\Request::validate()/check()` изолируют кастомные алиасы правил per-class.

```php
public static function get(string $alias): ?Rule       // резолвит алиас с учётом текущего scope, либо null
public static function currentPrefix(): ?string          // верхний scope-префикс из стека, либо null
```

`Rule::get()` удобен для идемпотентной регистрации («зарегистрировать один раз»):

```php
if (!Rule::get('boolToString')) {
    Rule::create(fn(&$v): bool => (bool)($v = $v ? 'true' : 'false'))
        ->alias('boolToString');
}
```

## Rule::object() и dot-notation схемы

Ключи схемы могут быть заданы через точку, что удобно для глубоко вложенных структур без явного вложения массивов:

```php
Rule::object([
    'user.name'        => 'required|string',
    'user.email'       => 'required|string|email',
    'user.tags.*'      => 'string|trim',
    'meta.created_at'  => 'nullable|date',
]);
```

Внутри `object()` все ключи с `.` собираются в дерево (`insertIntoTree()`), которое затем рекурсивно превращается в обычные вложенные `Rule::object()`/`Rule::forEach()` через `treeNodeToRule()`:

- Сегмент `*` на каком-либо уровне дерева (`'user.tags.*'`) означает, что на этом уровне применяется `Rule::forEach()` — либо простое правило к каждому элементу (если у `*` нет дочерних ключей), либо вложенная объектная схема к каждому элементу массива (если есть, например `'items.*.name'` и `'items.*.price'`).
- **Нельзя** смешивать `*` с именованными siblings на одном уровне дерева — бросает `RuntimeException("cannot mix wildcard '*' with named keys at the same level")`.
- **Нельзя** одновременно определить ключ и как обычный top-level ключ схемы (`'user' => [...]`), и через dot-notation (`'user.name' => ...`) — это неоднозначность, бросает `RuntimeException("key '...' is defined both as a regular key and via dot-notation")`.

Пример с массивом объектов:

```php
Rule::object([
    'items.*.name'  => 'required|string',
    'items.*.price' => 'required|float|min:0',
])->throwable()->apply($order);
```

эквивалентно:

```php
Rule::object([
    'items' => Rule::forEach(Rule::object([
        'name'  => 'required|string',
        'price' => 'required|float|min:0',
    ])),
])->throwable()->apply($order);
```

## Встроенная библиотека правил

Регистрируется один раз в приватном статическом `init()` (защищено от повторного выполнения через `static $done`). Полный список алиасов:

```
sometimes, default, required, nullable, present,
string, int / integer, float, bool, callable, closure,
email, url, array, foreach,
or / anyOf / any_of,
max, min, count, in, notIn / not_in, regex, digits,
hex_color / hexColor, between, date, date_format, json,
starts_with, ends_with, contains,
trim, ltrim, rtrim, html_encode, html_decode, strip_tags,
uppercase, lowercase, accepted, declined,
file, mime / mimes, extension, filesize
```

Краткое описание нетривиальных:

- **`sometimes`** — проходит, если значение присутствует; если сентинел — тихо (без ошибки) останавливает остаток пайплайна (`skip`). Базовый механизм «опционального» поля.
- **`present`** — как `sometimes`, но с сообщением об ошибке `'This field must be present'`, если поле отсутствует.
- **`nullable`** — если значение `null`/`''`/сентинел, тихо останавливает пайплайн (без ошибки) — таким образом последующие `string|email|...` не выполняются для действительно пустых значений.
- **`required`** — `Rule::requiredIf(true)`, ошибка `'This field is required'`.
- **`default:значение[,valid]`** — DSL-форма `Rule::default()`. Например `'default:30'` или `'default:0,true'` (второй параметр `'true'` включает флаг `$valid`).
- **`count`** — DSL `count:N` (или `count:N,M`) реализован как `['array', 'min', 'max']`; **важная тонкость**: composite прокидывает **все** переданные параметры целиком в оба вложенных правила `min` и `max`, а оба они читают только первый параметр (`$p[0]`). Поэтому `'count:2,5'` фактически требует **ровно 2** элемента (и `min`, и `max` получают лимит `2`), а второе число `5` не используется. Для диапазона количества элементов используйте `'array|between:2,5'`, а для точного количества — `'array|count:2'` (одно число).
- **`html_encode` / `html_decode`** — **имена перепутаны местами** относительно интуитивного смысла: `'html_encode'` вызывает `html_entity_decode()` (декодирует `&amp;` → `&`), а `'html_decode'` вызывает `htmlspecialchars()` (кодирует `&` → `&amp;`). Поведение не меняется этим документом — это фиксация существующего контракта; при использовании ориентируйтесь на фактическое действие, а не на название.
- **`foreach:спека`** — DSL-обёртка вокруг `Rule::forEach()`, применяющая переданную спеку к каждому элементу массива. Поддерживает составные спеки через запятую: `'array|foreach:min:1,max:10'` — каждый элемент должен быть длиной/значением от 1 до 10.
- **`accepted` / `declined`** — строгое (`in_array(..., true)`) сравнение с фиксированными наборами `[true, 1, '1', 'yes', 'on', 'true']` / `[false, 0, '0', 'no', 'off', 'false']` — типичное для чекбоксов «согласен с условиями».
- **`files` / `mime`(`mimes`) / `extension` / `filesize`** — валидаторы **загрузок**; работают с объектами [[UploadedFile.php]] (их отдаёт `Request::files()`), а не с сырыми массивами. Имя `files` — во множественном числе, потому что значением поля всегда является **список**. Подробнее — в разделе [«Правила загрузок файлов»](#правила-загрузок-файлов) ниже. Коротко: `files` проверяет подлинность загрузки (`isValid()` → `is_uploaded_file`), `extension` — расширение по клиентскому имени, `mime` — реальный MIME по содержимому (`finfo`, поддерживает wildcard `audio/*`), `filesize:максКБ` — размер одного файла. `files` **автоматически подмешивается** в остальные три.

## Правила загрузок файлов

Правила `files`, `mime` (`mimes`), `extension`, `filesize` валидируют **загрузки** и
оперируют объектами [[UploadedFile.php]] — именно их отдаёт `HTTP\Request::files()` (список
на каждое поле). Поскольку поле файла всегда список, правила принимают как список
`UploadedFile[]`, так и одиночный `UploadedFile` (например, элемент внутри `Rule::forEach`);
одиночный вход остаётся одиночным на выходе, список — списком.

> **Где живёт логика.** Объекты этих правил **создаёт и регистрирует сам `UploadedFile`** —
> в `UploadedFile::registerRules()`, поверх публичной фабрики [`Rule::filtered()`](#rulefiltered--правила-фильтры-коллекций),
> которая даёт им семантику sentinel + required-scope. `Rule::init()` лишь однажды вызывает
> `\ST_system\HTTP\UploadedFile::registerRules()` (полностью квалифицированным вызовом — импорта
> `UploadedFile` в `Rule` нет). Сами проверки (MIME/расширение/размер) написаны прямо в правилах,
> нормализация значения в список — `UploadedFile::filter()`.
> **Новое правило загрузок добавляется в `UploadedFile`, а не в `Rule`.**

- **`files`** — низкоуровневая проверка: оставляет только файлы с `isValid()`
  (`error === UPLOAD_ERR_OK` и `is_uploaded_file`). **Автоматически подмешивается** в
  `extension`/`mime`/`filesize` — их можно писать без явного `files`.
- **`extension:ext1,ext2,...`** — расширение по клиентскому имени (регистронезависимо).
- **`mime:тип,тип/*`** — реальный MIME по содержимому (`finfo`); поддерживает wildcard
  (`image/*`, `audio/*`).
- **`filesize:максКБ`** — размер в килобайтах.

### filter-режим vs strict-режим (роль `required`)

Поведение зависит от наличия алиаса **`required`** в том же пайплайне поля. Правило узнаёт
об этом из внутреннего стека `requiredStack`, который заполняет исполнитель пайплайна,
обнаружив среди правил `required` (проверка — прямо в `Rule::filtered()`).

- **без `required` (filter-режим)** — невалидные/неподходящие файлы **тихо отбрасываются**;
  если валидных не осталось, поле сворачивается в сентинел (ключ пропадает из результата),
  ошибок нет.
- **с `required` (strict-режим)** — если после фильтрации ничего не осталось, правило даёт
  **англоязычную ошибку** своей проверки (`The file failed to upload or is invalid`,
  `The file extension is not allowed`, `The file MIME type is not allowed`,
  `The file exceeds the allowed size`). Под `Request` с `$throwable = true` это исключение → JSON.

`required` (order 100) ловит **полное отсутствие** поля («This field is required»), а
strict-режим файловых правил — случай «файлы прислали, но валидных не осталось». Битые файлы
отбрасываются в обоих режимах; strict лишь делает ошибкой итоговую пустоту. Динамический
`Rule::requiredIf($cond)` (без алиаса `required`) strict-режим **не** включает — детект идёт
только по литеральному алиасу `required`.

```php
// поле-список 'audio'; files подмешивается в extension/mime автоматически
'audio'       => 'required|files|count:1|extension:mp3,wav|mime:audio/*',
// без required — тот же набор проверок, но пустой результат просто исчезнет без ошибки
'attachments' => 'files|max:5|extension:pdf,png',
```

### Сколько файлов принимать

Значение поля — всегда **список**, поэтому количество ограничивается обычными правилами
(`order` 700, то есть уже **после** отсева `files`/`extension`/`mime`):

- `count:1` — ровно один файл (это composite `['array','min','max']` с одним параметром,
  то есть `min:1` + `max:1`);
- `min:1` / `max:5` — нижняя и верхняя границы по отдельности;
- ничего не указано — принимается сколько прислали.

Без явного ограничения `<input name="audio[]">` с несколькими файлами пройдёт валидацию,
а контроллер, берущий `files('audio')[0]`, молча обработает только первый — поэтому для
одиночной загрузки полезно ставить `count:1`.

### Как это устроено внутри

- Общий приватный `pipeline()` прогоняет правила поля, попутно вычисляя `hasRequired`
  (правило с `ruleName === 'required'`, где `ruleName` проставляется в `alias()`) и складывая
  флаг в стек `requiredStack`. Используется во всех композитах (`create(string)`,
  `create(array)`), в `object()` и в `forEach()` — поведение единообразно.
- Файловые правила помечены `->seesSentinel()->skip()`: при strict-ошибке пайплайн
  обрывается (нет дублей от следующих файловых правил), при тихом отсеве значение = сентинел,
  и следующее правило видит уже пустой список.
- `forEach` удаляет элементы, свернувшиеся в сентинел, — так filter-режим работает и
  поэлементно (`Rule::forEach('file')`).
- Регистрация: `Rule::init()` в самом конце вызывает `\ST_system\HTTP\UploadedFile::registerRules()`;
  тот собирает каждое правило через `Rule::filtered()`, доставляет `order`/`handleError` и вешает
  алиасы. Модификаторы применяются **до** первого `alias()`, т.к. `alias()` замораживает инстанс.
- Взаимная загрузка безопасна: `registerRules()` дёргает `Rule::create()`, та зовёт `init()`,
  но `init()` ставит `static $done = true` **до** регистраций, поэтому повторный вход выходит сразу.
- В самом `Rule` от загрузок осталась ровно одна строка — этот FQ-вызов; импорта нет.

## Особенности коэрсии типов

- **`int` / `integer`** — коэрсит числовые строки, `float` (если конечное число), `bool` в `int`; `null`/сентинел коэрсится в `0`. Проваливается только на реально несовместимых значениях (например, произвольная нечисловая строка, массив, объект).
- **`float`** — аналогично, коэрсит `int`/`bool`/числовые строки во `float`; `null`/сентинел → `0.0`.
- **`bool`** — коэрсит ограниченный набор значений: `'0'`, `'1'`, `0`, `1`, `'true'`, `'false'`, `'checked'`, `'on'`, `'off'` (строки — через `filter_var(..., FILTER_VALIDATE_BOOLEAN)`); `null`/сентинел → `false`. Прочие значения проваливаются.
- **`string`** — коэрсит `null`/сентинел в `''` (и проходит); проваливается только на значениях, которые не строка и не null-подобны (массивы, объекты, ресурсы и т.д.).
- **`array`** — коэрсит `null`/сентинел в `[]`; **объект с методом `toArray()`** коэрсится в результат этого метода (`$v = $v->toArray()`). Так `file|array` возвращает плоский массив для одного `UploadedFile`. Для списка объектов (`files('audio')` — это список) `array` оставит список объектов; чтобы получить список плоских массивов, используйте `Rule::forEach('array')`.
- **`trim` / `ltrim` / `rtrim`** — `trim` дополнительно работает и на **массивах строк** (маппит `trim` по элементам через `array_map`) и **удаляет** элементы, ставшие пустой строкой после обрезки (`array_filter`). `ltrim`/`rtrim` работают только со строками (массивы не поддерживаются).
- **`email` / `url`** — composite из `['string', <filter_var FILTER_VALIDATE_EMAIL/URL>]`, т.е. неявно требуют строку до проверки формата.

## Известные тонкости и «подводные камни»

- Замороженные (`->alias()`) правила иммутабельны — попытка вызвать `before/after/handleError/order/skip/seesSentinel` на них бросает исключение. Если нужно кастомизировать поведение стандартного правила — работайте с его клоном (получаемым автоматически при резолве через DSL/`Rule::get()->copy()` — публичного `copy()` нет, поэтому обычно проще обернуть его в новую спеку, а не мутировать напрямую).
- Правило без `->seesSentinel()`, столкнувшись с сентинелом, вызывается с замаскированным `null`; если оно **явно** не присвоило значению что-то отличное от `null`, исходный сентинел в `$data` остаётся нетронутым (значение по-прежнему считается «отсутствующим» для дальнейших шагов пайплайна). Это осознанный механизм, но означает, что такое правило не может «превратить» отсутствие в явный `null` — для этого нужен `->seesSentinel()`.
- DSL режет параметры по `,` на уровне всего сегмента (`name:p1,p2,...`) — паттерны `regex`, наборы `in`/`notIn` и т.п., содержащие запятые/двоеточия в самих значениях, надёжнее задавать через объектные фабрики (`Rule::regex()`, `Rule::in()`), а не через DSL-строку.
- `Rule::object()` **отбрасывает** любые ключи входных данных, не описанные в схеме — это осознанное поведение «схема авторитетна», а не побочный эффект бага.
- `Rule::forEach()` **тихо удаляет** проваленные элементы вместо провала всего поля — если нужна семантика «весь массив невалиден, если хоть один элемент плохой», оборачивайте в `->throwable()`. Дополнительно `forEach` удаляет элементы, которые правило **свернуло в сентинел** (без ошибки) — на этом держится «тихий» отсев в правилах загрузок (`files`/`extension`/`mime` в filter-режиме).

## Практические примеры из фреймворка

Валидация конфига трейтом `HasConfig` (регистрация scoped-алиаса `defaultConfig` один раз, подстановка дефолтов из `static::config()`):

```php
// src/Traits/HasConfig.php
Rule::scope(static::class, function() use (&$config, $schema) {
    Rule::object(array_map(fn($spec) => static::resolveAtSpec($spec), $schema))
        ->throwable()
        ->apply($config);
});
```

Валидация query/тела в `HTTP\WebClient`:

```php
// src/HTTP/WebClient.php
'method' => ['string|uppercase|@method', Rule::in(['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'HEAD', 'OPTIONS'])],
...
$this->bodyRule  = Rule::object($clean)->throwable();
$this->queryRule = $schema ? Rule::object($schema)->throwable() : null;
```

Дефолты и коэрсия в `Access`:

```php
// src/Access.php
'onFail'      => ['callable', Rule::default(fn() => self::throw(401))],
'white_list'  => ['array', Rule::default([])],
'allowed_origins' => ['array', Rule::default(['*'], true), Rule::forEach('url')],
'methods'         => ['array|@CORS.methods', Rule::forEach(['required|string|strtoupper', Rule::in(self::config('CORS.methods'))])],
```

Валидация параметров интеграции в `API\IntegrationDriver` (per-class scope, чтобы разные драйверы не конфликтовали именами правил):

```php
// src/API/IntegrationDriver.php
Rule::scope(static::class, fn() => $this->__init());
...
$config['params'] = Rule::scope(static::class, fn() => Rule::object($config['params']));
```

Валидация HTTP-запроса (`HTTP\Request::validate()`), также в per-class scope:

```php
// src/HTTP/Request.php
Rule::scope(static::class, function() use ($schema, &$errors) {
    $rule = Rule::object($schema);
    if ($this->throwable) $rule->throwable();
    $errors = $rule->apply($this->data['data']);
});
```
