# DefaultSchema.php
<!-- DOCGEN:END -->

`ST_system\Schemas\DefaultSchema` — базовый класс для всех схем структурированных данных в проекте. От него наследуются как типы schema.org (`src/Schemas/SchemaOrg/*`), так и Yandex-фид (`src/Schemas/Yandex/*`). Класс сам по себе не описывает никакой конкретный тип разметки — он даёт подклассам общий движок: объявление полей, их валидацию через `Rule`, связывание вложенных схем (`arrayOf`/`oneOf`/`@ref`), генерацию вывода (`print`/`toArray`) и доступ к данным (`field`/`set`).

## Как объявить новую схему

```php
namespace ST_system\Schemas\SchemaOrg;

use ST_system\Schemas\DefaultSchema;

final class MyType extends DefaultSchema
{
    protected static function getFields(): array
    {
        return [
            'name' => 'required|string',
            'url'  => 'sometimes|url',
        ];
    }

    protected static function getPrint(): \Closure
    {
        return function (DefaultSchema $s): string {
            $data = [
                '@context' => 'https://schema.org',
                '@type'    => 'MyType',
                'name'     => $s->field('name'),
            ];

            return '<script type="application/ld+json">'
                . json_encode($data, JSON_UNESCAPED_UNICODE | JSON_UNESCAPED_SLASHES)
                . '</script>';
        };
    }
}
```

Использование:

```php
$schema = MyType::create()->fill([
    'name' => 'Пример',
    'url'  => 'https://example.com',
]);

echo $schema->print(); // строка с <script type="application/ld+json">...</script>
```

Класс должен жить под пространством имён `ST_system\Schemas` (константа `ROOT_NAMESPACE`), иначе `scope()` бросит `LogicException`.

## Публичные хуки подкласса

Подкласс переопределяет (все — `protected static`, вызываются движком автоматически, напрямую их дергать не нужно):

- **`getFields(): array`** — карта `имя_поля => спецификация`. По умолчанию пустой массив. Форматы спецификации разобраны ниже.
- **`getPrint(): ?\Closure`** — как в примере выше: функция `fn(DefaultSchema $s): string`, которая рендерит финальную строку (обычно `<script type="application/ld+json">...</script>` для schema.org или XML-узел для Yandex-фида). Если не переопределить (вернуть `null`), `print()` сам сериализует `toArray()` в JSON.
- **`getToArray(): ?\Closure`** — `fn(DefaultSchema $s): array`, кастомная сборка массива для `toArray()`/вложенности внутри родительской схемы (обычно используется вложенными под-схемами вроде `Offer`, `Question`, `ListItem`, которые сами не печатаются, а встраиваются в родителя через `$s->field('x')->toArray()`).
- **`_init(): void`** — разовый статический хук инициализации класса (пустой по умолчанию), вызывается один раз на класс лениво при первом обращении (см. «Ленивая инициализация» ниже).

## Инстанцирование и inline-схемы

- **`DefaultSchema::create(...$args)`** — фабрика, эквивалент `new static(...$args)`.
- Обычный подкласс создаётся как `new ChildClass()` (или `ChildClass::create()`).
- Саму `DefaultSchema` также можно использовать напрямую как inline-схему, без отдельного класса:

```php
$inline = new DefaultSchema([
    'fields'  => ['name' => 'required|string'],
    'print'   => fn(DefaultSchema $s) => json_encode($s->toArray()),
    'toArray' => fn(DefaultSchema $s) => ['name' => $s->field('name')],
]);
$inline->fill(['name' => 'X']);
```

## Идентичность схемы: `name()`, `scope()`, `path()`

- **`name(): string`** — kebab-case короткое имя класса (без namespace), например `FaqPage` → `faq-page`. Кэшируется в `$nameCache`.
- **`scope(): string`** — namespace-путь класса относительно `ST_system\Schemas`, переведённый в kebab-case и соединённый точками (например, `ST_system\Schemas\SchemaOrg\Service` → `schema-org`, а `ST_system\Schemas\SchemaOrg\Service\Offer` → `schema-org.service`). Бросает `LogicException`, если класс не находится под `ROOT_NAMESPACE`.
- **`path(): string`** — `scope() . '.' . name()` (или просто `name()`, если `scope()` пуст) — полный «адрес» схемы, используемый как префикс области видимости правил валидации (`Rule::scope`) и как контекст резолюции `@ref`.

## Ленивая инициализация (`ensureInited`)

При первом создании экземпляра класса (в конструкторе) вызывается `ensureInited()`, которая один раз на класс:
1. Рекурсивно поднимается по родительским namespace (в пределах `ST_system\Schemas`) и инициализирует их, если это тоже подклассы `DefaultSchema` (то есть класс-«контейнер», например `Service`, инициализируется раньше своих вложенных `Service\Offer` и т.п.).
2. Вызывает `_init()` конкретного класса внутри `Rule::scope(path(), …)`, чтобы любые правила валидации, регистрируемые в `_init()`, попадали в правильную область видимости.

Повторные вызовы для того же класса — no-op (флаг в `$initialized`).

## Объявление полей (`getFields()`) — форматы спецификации

Ключ массива — имя поля. Значение (`$spec`) может быть:

1. **`\Closure`** — вычисляемое поле. Не валидируется через `Rule`, а вызывается **после** валидации остальных полей: `fn(array $result) => ...`, результат кладётся в `$result[$key]`.
2. **`Rule`** — готовый объект правила, используется как есть.
3. **Строка со строковыми правилами `Rule`**, например `'required|string|url'` — передаётся в `Rule::create()`.
4. **Строка со ссылкой `@ref`**, например `'sometimes|@provider'` или `'@provider'` — ссылка на другую схему по её kebab-имени (`name()`). Ссылка резолвится в FQCN через `resolveRef()` (см. ниже), оборачивается в правило коэрсии (`makeEntityRefRule`) с приоритетом `order(600)` и подмешивается к прочим модификаторам (`required`/`sometimes`/...), которые должны идти в той же строке через `|`.
5. **Строка — полное имя класса** (FQCN подкласса `DefaultSchema`), опознаётся эвристикой `looksLikeClass()` (`strpos($s, '\\') !== false && class_exists($s)`) — трактуется как прямая ссылка на схему, аналогично `@ref`, но без резолюции по имени.
6. **`DefaultSchema::arrayOf($spec)`** — маркер «массив элементов данного типа». `$spec` — либо `@ref`/FQCN подкласса (тогда каждый элемент массива коэрсится в экземпляр этой схемы через `fill()`), либо обычная строка правил `Rule` (тогда к каждому элементу применяется `Rule::forEach($spec)`).
7. **`DefaultSchema::oneOf([$specA, $specB, ...])`** — маркер «одно из». Перебирает варианты по порядку и берёт первый успешно совпавший (ссылка на схему или обычное правило `Rule`); если ни один не подошёл — ошибка со списком вариантов.
8. **Массив** (`is_array($spec)`), например `[self::arrayOf('question'), 'required']` — смесь маркера (`arrayOf`/`oneOf`/`@ref`/FQCN — максимум один) и обычных модификаторов правил (`'required'`, `'sometimes'`, ...). Маркер извлекается и оборачивается в `Rule`, остальные элементы массива добавляются как дополнительные под-правила.

Если тип спецификации не подошёл ни под один случай — `RuntimeException('Schema: unsupported field spec type')`.

### Резолюция `@ref` (`resolveRef`)

`@provider` → берётся `Main::studlyCase('provider')` = `Provider`, и класс ищется по цепочке:
1. `<текущий класс>\Provider` (то есть как вложенный неймспейс текущей схемы — например, `Service\Provider` при вызове из `Service`);
2. далее вверх по родительским неймспейсам вплоть до `ST_system\Schemas`.

Первый существующий класс, который является подклассом `DefaultSchema`, побеждает. Если ничего не найдено — `RuntimeException`.

## `fill(array $data, array $fillParams = []): self`

Основной метод заполнения схемы данными:
1. Сохраняет сырые входные данные в `$rawData` (нужно для `append()`).
2. Прогоняет `beforeHooks` (см. `before()`) — они могут мутировать `$data` по ссылке до валидации.
3. Строит `$ruleSchema` из `getFields()`/`$inlineFields` по правилам, описанным выше, отдельно собирая `$computed` (замыкания).
4. Валидирует данные через `Rule::object($ruleSchema)->throwable()->apply($result)` внутри области видимости `path()` (`withEntityScope`) — при ошибке валидации бросается исключение (`throwable()`).
5. Вычисляет `computed`-поля и кладёт их в результат.
6. Сохраняет результат в `$this->data`, помечает схему как `filled`, сбрасывает кэш печати.
7. Простановка `parent` для всех вложенных `DefaultSchema`-значений (`linkParent`).
8. Прогоняет `afterHooks` (см. `after()`) с `$this`.

Возвращает `$this` — вызовы можно чейнить: `MyType::create()->fill([...])->print()`.

## `append(array $data): self`

Дозаполнение уже заполненной схемы: если схема ещё не заполнена — эквивалентно `fill($data)`; если уже заполнена — вызывает `fill(array_merge($this->rawData, $data), $this->fillParams)`, то есть по сути `fill()` заново со старыми данными, перекрытыми новыми, с полной повторной валидацией.

## `before(callable $fn): self` / `after(callable $fn): self`

Регистрируют хуки, вызываемые внутри `fill()`:
- `before` — до валидации, получает `&$data` по ссылке (можно нормализовать/дополнить сырые данные перед валидацией);
- `after` — после полного заполнения, получает `$this` (заполненный экземпляр).

Оба возвращают `$this`, хуки накапливаются (можно регистрировать несколько).

## `set(string $path, $value): self`

Точечное обновление поля(-ей) уже созданного экземпляра по пути с точечной нотацией (скобки `[`/`]` игнорируются): `set('questions.0.answer', 'Новый ответ')`. Если голова пути — вложенная `DefaultSchema`, обновление делегируется её `set()`; если массив — обновляется элемент по числовому индексу (рекурсивно, если элемент сам `DefaultSchema`). Сбрасывает `printCache` и обновляет `parent` для новых значений. **Не проходит валидацию** через `Rule` — прямая точечная правка данных в обход `getFields()`.

## `print(array $params = []): string`

Возвращает готовую строку разметки:
- Требует, чтобы схема была заполнена (`filled`), иначе `RuntimeException`.
- На верхнем уровне вызова (не вложенном) собирает `$printContext = Main::merge($fillParams, $params)` — то есть параметры, переданные в `fill()`, объединяются с параметрами, переданными непосредственно в `print()`.
- Кэширует результат (`printCache`), но только если вызов верхнеуровневый **и** `printContext` пуст (иначе разные параметры печати давали бы один и тот же кэш — поэтому кэш используется только для «дефолтной» печати без параметров).
- Если подкласс задал `getPrint()`/`inlinePrint` — вызывает его: `$printer($this, $printContext)`. Иначе — генерирует JSON из `toArray($printContext)` (`JSON_UNESCAPED_UNICODE | JSON_UNESCAPED_SLASHES`, без `<script>`-обёртки — обёртку добавляют сами подклассы).
- Счётчик `$printDepth` — статический на весь класс — отслеживает вложенность вызовов `print()` (актуально, если один принтер сам вызывает `print()` дочерней схемы), чтобы верно определять «верхний ли это вызов».

## `field(string $name = '')`

Чтение данных по имени/пути (точечная нотация, скобки `[`/`]` игнорируются). Без аргумента — возвращает весь `$data`. Поддерживает спуск во вложенные `DefaultSchema` (`$value->field($tail)`) и числовые индексы массивов (`item.0.name`). Возвращает `null`, если путь не найден.

## `toArray(array $params = []): array`

Экспорт схемы в обычный PHP-массив (без обёртки печати):
- Если задан `getToArray()`/`inlineToArray` — вызывается он: `$custom($this, $params)`.
- Иначе — обходит `$this->data` и рекурсивно разворачивает вложенные `DefaultSchema` (`deepExport`): вложенные схемы превращаются в свои `toArray()`, массивы обходятся рекурсивно поэлементно, скаляры возвращаются как есть.

Именно `toArray()` (а не `print()`) обычно вызывают вложенные схемы друг у друга, когда встраивают себя в массив родителя (`$s->field('provider')->toArray()`).

## `parent(): ?self`

Возвращает родительскую схему, в которую данный экземпляр был вложен через `fill()`/`set()` (устанавливается `linkParent()`), либо `null` для корневой схемы.

## `arrayOf(string $spec): object` / `oneOf(array $specs): object`

Статические фабрики маркеров для `getFields()` (см. раздел «Объявление полей» выше, пункты 6–7). Возвращают анонимный объект-«тег» (`{__m, spec}` / `{__m, specs}`), который `fill()` распознаёт и превращает в правило валидации/коэрсии.

## Валидация и `Rule`

Вся валидация полей идёт через `ST_system\Rule`: строковые спецификации (`'required|string|url'`) парсятся в цепочки правил, `Rule::object($ruleSchema)` валидирует весь объект целиком, `Rule::forEach()` — применяет правило к каждому элементу массива, `Rule::scope()` — задаёт область видимости для правил, регистрируемых в `_init()` конкретных схем. Ссылочные и `arrayOf`/`oneOf`-правила регистрируются с высоким `order(600)`, чтобы выполняться после большинства обычных правил валидации.

## Область видимости данных (`withEntityScope`)

Для контекста вида `schema-org.service` последовательно открывает вложенные `Rule::scope()` — сначала `schema-org`, затем `schema-org.service` — так что правила, зарегистрированные для родительской схемы (`schema-org`), видны и при валидации дочерней (`schema-org.service`).
